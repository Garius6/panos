# AGENTS.md

## О проекте

Panos — экспериментальный интерпретатор самописного языка программирования,
реализованный на Odin. Язык русскоязычный по ключевым словам и синтаксически
ориентирован на смесь Go и Rust.

Текущий pipeline:

1. `lexer.odin` — токенизация исходного текста.
2. `parser.odin` — построение AST.
3. `resolver.odin` — связывание имен с символами.
4. `type_cheker.odin` — проверка и вывод типов.
5. `compiler.odin` — компиляция AST в байткод.
6. `vm.odin` — выполнение байткода.
7. `main.odin` — запуск файла и диагностический вывод всех стадий.

`test.ps` — основной демонстрационный файл языка. `e2e_test.odin` содержит
сквозные тесты pipeline через `run_code`.

## Команды

Запуск тестового файла:

```sh
odin run . -debug -vet -strict-style -vet-tabs -warnings-as-errors -- test.ps
```

То же через Justfile:

```sh
just debug-file test.ps
```

Автотесты:

```sh
odin test . -debug -vet -strict-style -vet-tabs -warnings-as-errors
```

Тесты могут печатать предупреждения memory tracker о текущих утечках
аллокатора. Если команда завершилась с кодом `0`, тест считается успешным.

## Язык Panos

Ключевые слова и базовый синтаксис:

- `функ имя(арг: Тип) -> Тип ... конец` — функция.
- `функ(арг) ... конец` — лямбда, типы можно вывести из контекста или тела.
- `пер имя = значение` — локальная переменная.
- `[экспорт] конст ИМЯ = значение` (top-level) — именованная константа;
  значение MUST быть Число/Строка/Булево литералом (либо унарный минус
  перед числом) — компилируется подстановкой на месте использования, не
  рантайм-ячейкой (язык сознательно не даёт top-level изменяемого
  состояния). Использование через границу модуля — `модуль.ИМЯ`, как у
  экспортированных функций/типов.
- `возврат значение` — возврат.
- `если условие тогда ... иначе ... конец` — if-выражение.
- `пока условие цикл ... конец` — цикл.
- `продолжить` — перейти к следующей итерации цикла.
- `прервать` — выйти из цикла.
- `тип Имя = структура ... конец` — структура.
- `тип Имя = интерфейс ... конец` — интерфейс.
- `тип Имя = перечисление Вариант; Вариант(Тип, ...) ... конец` —
  алгебраический тип (sum type). Экспортируется вместе со всеми вариантами.
- `выбор subject Шаблон -> тело ... конец` — сопоставление с образцом;
  проверяет исчерпываемость и не даёт `_` не последней веткой. Тело ветки —
  либо одно выражение (`-> выражение`), либо блок из нескольких
  `Шаблон тогда стейтмент1 стейтмент2 ... конец` (значение ветки —
  последнее выражение блока, как в теле функции).
- `реализация Тип ... конец` — методы структуры.
- `реализация Интерфейс для Тип ... конец` — реализация интерфейса.
- `функ имя[T](x: T) -> T`, `тип Имя[T] = структура/перечисление ...` —
  дженерики. Лямбды полиморфны неявно (без `[T]`, из ожидаемого типа).
  `реализация Тип ...` для generic-типа — БЕЗ `[T]` в заголовке (целевой
  тип уже generic); метод может добавить свой `[U]`. Generic-типы не
  могут реализовывать интерфейсы. Bounded generics и explicit
  instantiation не поддержаны. См. `docs/language.md#дженерики`.

Типы:

- `Число`, `Булево`, `Строка`, `Пусто`
- туплы: `(Число, Строка)`
- функции: `функ(Число, Строка) -> Булево`
- массивы: `Массив(Число)`
- соответствия/map: `Соответствие(Строка, Число)`
- опциональные значения: `Опция(Число)`
- обычные ошибки: `Результат(Число, Ошибка)`

`Опция`/`Результат` — не примитивы языка, а generic-перечисления из
стандартной «прелюдии» (core/prelude.odin), неявно доступной каждому
модулю без `импорт`. Их конструкторы — как у любого ADT (Стадия 18):
ТОЛЬКО квалифицированные `Опция.Есть(...)`/`Результат.Успех(...)` и т.п.,
голые `Есть(...)`/`Успех(...)` не резолвятся.

Строки поддерживают escape-последовательности `\n`, `\t`, `\r`, `\"` и `\\`.

Функции, методы интерфейсов и методы структур требуют явных типов параметров и
возвращаемого значения. У лямбд типы параметров и возвращаемого значения можно
опускать, если type checker может вывести их из ожидаемого типа или тела.
Универсального типа-заглушки в языке нет.
Пользовательские структуры и интерфейсы типизируются номинально: структуры
совместимы с интерфейсами только через явную `реализация Интерфейс для Тип`.

Коллекции:

```panos
пер числа: Массив(Число) = массив()
числа.добавить(1)
числа[0] = 10
пер x = числа.получить(0, 0)

пер цены = соответствие(
    "яблоко" = 10,
    "груша" = 20,
)
цены["банан"] = 5
пер есть_банан = цены.есть("банан")
```

Поддержанные методы массивов:

- `.длина()`
- `.добавить(значение)`
- `.получить(индекс, иначе)`
- `.есть(индекс)`
- `.содержит(значение)`

Поддержанные методы соответствий:

- `.длина()`
- `.есть(ключ)`
- `.получить(ключ, иначе)`
- `.удалить(ключ)`

Индексирование `[]` строгое: отсутствующий ключ map или индекс массива вне
границ должен приводить к runtime error. Метод `.получить(..., иначе)` нужен
для безопасного чтения.

Ошибки и отсутствие значения:

```panos
пер о: Опция(Число) = Опция.Есть(41)
пер число = о? + 1

пер р: Результат(Строка, Ошибка) = Результат.Неудача(Ошибка("фс", "нет файла"))
пер сообщение = р.получить_ошибку(Ошибка("нет", "нет ошибки")).сообщение
```

- `Ошибка(код, сообщение)` — стандартное значение ошибки.
- `Опция.Есть(значение)` / `Опция.Нет()` — конструкторы `Опция(T)`.
- `Результат.Успех(значение)` / `Результат.Неудача(ошибка)` — конструкторы `Результат(T, E)`.
- `выражение?` разворачивает `Есть`/`Успех` или досрочно возвращает
  `Опция.Нет()`/`Результат.Неудача(...)` из текущей функции.
- `паника("сообщение")` — критичный сбой; возвращает внутренний тип `Никогда`,
  совместимый с обычными типами в ветках выражений.

Поддержанные методы `Опция(T)`:

- `.есть()`
- `.пусто()`
- `.значение()` — строгое извлечение, runtime error для `Опция.Нет()`.
- `.получить(иначе)` — безопасное значение по умолчанию.
- `.ожидать("сообщение")` — извлечение или runtime panic.
- `.результат_или(ошибка)` — `Опция.Есть(T)` в `Результат.Успех(T)`, `Опция.Нет()` в
  `Результат.Неудача(ошибка)`.
- `.заменить_значение(значение)` — заменить значение только у `Есть`
  (свой type-параметр, может сменить тип).
- `.запас(другая_опция)` — вернуть текущую `Есть`, иначе запасную `Опция`.

Поддержанные методы `Результат(T, E)`:

- `.успех()`
- `.ошибка()`
- `.значение()` — строгое извлечение, runtime error для `Неудача`.
- `.причина()` — строгое извлечение ошибки, runtime error для `Успех`.
- `.получить(иначе)` — значение успеха или значение по умолчанию.
- `.получить_ошибку(иначе)` — причина ошибки или ошибка по умолчанию.
- `.ожидать("сообщение")` — значение успеха или runtime panic.
- `.ожидать_ошибку("сообщение")` — причина ошибки или runtime panic.
- `.опция()` — `Результат.Успех(T)` в `Опция.Есть(T)`, `Результат.Неудача(E)` в `Опция.Нет()`.
- `.ошибка_опция()` — `Результат.Неудача(E)` в `Опция.Есть(E)`, `Результат.Успех(T)` в `Опция.Нет()`.
- `.заменить_значение(значение)` — заменить значение только у `Успех`
  (свой type-параметр, может сменить тип T).
- `.заменить_ошибку(ошибка)` — заменить причину только у `Неудача`
  (свой type-параметр, может сменить тип E).
- `.запас(другой_результат)` — вернуть текущий `Успех`, иначе запасной
  `Результат`.

Не использовать `.или(...)` как имя метода: `или` зарезервировано под будущий
логический оператор/ключевое слово языка.

## Рабочие соглашения

- Сохранять русскоязычные сообщения ошибок и диагностический вывод.
- Предпочитать минимальные вертикальные срезы: lexer → parser → resolver →
  type checker → compiler → VM → e2e-тест.
- При добавлении синтаксиса сначала покрывать его сквозным тестом в
  `e2e_test.odin`, если это практически возможно.
- Не коммитить `panos.dSYM/` и другие локальные артефакты сборки.
- Не менять unrelated untracked файлы без явного запроса.

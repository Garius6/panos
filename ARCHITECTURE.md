# Architecture

Внутренняя архитектура panos — для тех, кто меняет `core/*.odin`, не для
пользователей языка (синтаксис/API — `AGENTS.md`, `docs/`). Документирует
**факты, ограничения и решения**, а не поведение фич — что где лежит и
почему сделано именно так, включая места, где решение НЕ самое очевидное.

Pipeline (`lexer → parser → resolver → type_cheker → compiler → vm`) —
см. `AGENTS.md`, не дублируется здесь.

## Value и память

- `Value :: union { f64, bool, ^Panos_String, ^Compiled_Function,
  ^Aggregate_Value, ^Array_Value, ^Map_Value, ^Error_Value, ^Option_Value,
  ^Result_Value, ^Interface_Value, ^Variant_Value, ^File_Value,
  ^Socket_Value }` (compiler.odin). Всё, кроме `f64`/`bool` — указатель на
  GC-управляемую память.
- **Структуры — reference semantics**, не value semantics. Присваивание
  копирует указатель, не поля: `пер b = a; b.x = 99; a.x` → `99` (живой
  тест, подтверждено). Это осознанная, не случайная модель — обсуждалась
  и подтверждена явно (см. `[[копируемое-и-reference-semantics]]` ниже).
  `Set_Local`/`Set_Property` (vm.odin) просто копируют `Value`-union
  как есть, без диспетчеризации по типу.
- **Aggregate_Value (структуры) не хранит RTTI** — только `header` +
  `elements: [dynamic]Value` (позиционно, без имён полей и БЕЗ имени
  типа). Это значит: рантайм физически не может ответить "что за тип
  у этой структуры" без компилятора со стороны. Прямое следствие —
  `value_to_display_string` (Печатаемое, Стадия 23) даёт ТОЛЬКО
  позиционный дамп (`(3, 4)`) для структур без явной реализации
  `Печатаемое`, никогда не `Точка(x: 3, y: 4)`. `Variant_Value`
  (перечисления) хранит `type_name: string`, но НЕ имя конкретного
  варианта (только `tag_index`) — тоже неполный RTTI.
- GC: `gc_new`/`gc_protect`/`gc_unprotect` (vm.odin/gc.odin) — protect
  вручную перед аллокациями, которые могут триггернуть GC, пока
  промежуточное значение ещё не привязано к корню (стеку/локали).

## Соответствие (Map) — НЕ хеш-таблица

`Map_Value.entries` (compiler.odin) — плоский `[dynamic]Map_Entry_Value`.
`map_find_index` (vm.odin) — линейный перебор с `value_equals` на каждой
записи, O(n) lookup. Никакого хеширования в рантайме нет вообще —
название "Соответствие"/"Map" описывает семантику API, не реализацию.
Ключи ограничены `is_valid_map_key_type` (type_cheker.odin) до
`Число`/`Булево`/`Строка` — НЕ потому что структуры/enum'ы нельзя
сравнивать (`value_equals` уже умеет рекурсивно, с защитой от циклов),
а как отдельное, снимаемое одной строкой ограничение. См.
`[[хешируемое-выброшена]]`.

## Диагностики: accumulate, не panic

Лексер/парсер/резолвер/тайпчекер копят диагностики в собственный
`[]Diagnostic` (не паникуют на первой ошибке) — `report()`/
`report_parse()`/`report_resolve()`. Компилятор запускается ТОЛЬКО после
`typecheck_program` с нулём diagnostics (гейт в `main.odin`/тестовом
harness) — значит `panic`'и внутри `compiler.odin` на "невозможных" по
семантике узлах (`Error_Expr`, отсутствующий `Call_Info` и т.п.)
корректны как *internal* инварианты: пайплайн физически не должен был
дать компилятору увидеть такой узел.

`TY_POISON` — error-sentinel, protivoyaditsya каскадным ошибкам:
контрадикторный тип не блокирует остальной вывод, просто "заражает"
всё, что от него зависит, без повторных diagnostic'ов поверх уже
отчитанной первопричины. Обязателен независимо от eager/constraint-based
устройства решателя (см. ниже) — constraint-based тоже несёт
error-placeholder через solver, просто он живёт там, а не в разрозненных
`report()`.

## Type checker: eager unify + constraint-based batch — сознательный сплит, НЕ незавершённая миграция

**Факт**: `type_cheker.odin` использует ДВЕ разных стратегии разрешения
типов одновременно:

1. **Eager unify** (`unify_types`, вызывается напрямую по ходу обхода
   AST) — подавляющее большинство мест (~38 call site'ов).
2. **Constraint-based batch** (`Constraint`/`emit_constraint`/
   `solve_constraints`, накопление в `ctx.pending_constraints`, решение
   одним проходом) — ТОЛЬКО join-точки: `если`/`иначе`
   (`infer_if_expr`), ветки `выбор` (`infer_match_expr`), элементы
   `Массив`/`Соответствие`-литералов (`infer_array_expr`/
   `infer_map_expr`).

**Это не половина миграции** — полный переход на constraint-based
рассматривался (изначально планировался для Стадии 7, Generics) и был
**сознательно отклонён**: 43 вызова `unify_types`, большинство внутри
eager bottom-up `infer_expr`/`check_expr`, где тайпчекер не просто
сверяет типы, а **делает структурные решения по уже разрешённому типу**
(какой метод резолвится на `.Массив` vs `.Соответствие`, exhaustiveness
`выбор` по `.Enum` и т.д.) — такие места нельзя отложить на batch-solve
без полного redesign на HM-стиль с отложенной диспетчеризацией (недели
работы, высокий риск регрессии по всем тестам, ради нуля текущих
функциональных проблем).

**Аудит (проверен точечно, не все 38 мест — предположение)**: каждое
оставшееся eager-место попадает ровно в одну из трёх категорий, без
исключений:

1. **Одиночная пара "известное vs известное"** (большинство) — обе
   стороны уже полностью выведены ДО вызова `unify_types` (аннотация
   лямбды vs ожидаемый тип, тело функции vs объявленный возврат,
   аргумент варианта vs поле ADT, `x = value` совместимость). Откладывать
   нечего — это не "N вещей сходятся друг с другом", а "это конкретное
   место сверяется с ЭТИМ конкретным ожиданием", с собственным точным
   diagnostic'ом на своём span.
2. **Структурная диспетчеризация** (пример — `.Plus` в
   `infer_binary_expr`, type_cheker.odin ~2832): InferVar-направленный
   unify решает, пойдёт ли `+` по пути Строка+Строка / Число+Число /
   Складываемое-sugar — ТРИ разных последующих кодогена, не "какой у
   этого один тип". Отложить в solver означало бы, что сам solver
   должен уметь выбирать между стратегиями компиляции.
3. **Sequencing-critical binding** (пример — вызов метода структуры/
   enum, type_cheker.odin ~3292/3327): `unify_types(obj_type,
   method_type.params[0])` связывает свежие InferVar'ы из
   `instantiate_scheme` ДО того, как следующая строка проверяет
   аргументы через УЖЕ связанный `method_type`. Порядок здесь не
   архитектурная прихоть — без него аргументы физически нечем
   проверять.

**Решение (действующее)**: сплит остаётся. Полный переход на
constraint-based НЕ делается — обсуждалось явно, отклонено дважды (при
планировании Стадии 7 и повторно при пересмотре архитектуры позже).
Обсуждался средний путь — унифицировать ПОВЕРХНОСТЬ API (всегда
`emit_constraint(...)`, для eager-мест — с немедленным `solve_constraints
(ctx)` следующей строкой, функционально идентично прямому
`unify_types`) — НЕ принят: он бы визуально уравнял sequencing-critical
и join-точечные места, пряча реальную разницу поведения за одинаковым
синтаксисом, что хуже нынешней честной асимметрии (код-форма отражает
факт: увидел `unify_types` напрямую — знаешь, это синхронная одиночная
проверка; увидел `emit_constraint` — знаешь, это часть batch'а,
решаемого позже).

**Как решать новое место типизации**: если это N уже-выведенных типов,
которые должны совпасть друг с другом, и вы хотите видеть ВСЕ
несовпадения за один проход (не только первое) — `emit_constraint`. Если
это одна пара "известное vs известное", или диспетчеризация по уже
разрешённому `.kind`, или scoped-critical порядок — `unify_types`
напрямую. Не изобретать третий механизм.

## Generics (Стадия 7) — сделано на eager unify, без constraint-based

Полиморфизм ранга 1 (лямбды/функции/структуры/enum'ы/методы) реализован
ЦЕЛИКОМ на eager-unify инфраструктуре (см. выше) — constraint-based НЕ
понадобился ни для одной фазы (A-F), несмотря на изначальное
предположение плана, что переход нужен именно здесь под
`generalize`/`instantiate`.

- `Type_Scheme :: struct { forall: [dynamic]int, body: ^Type }` —
  полиморфная схема, `symbol_schemes: map[Symbol_Id]Type_Scheme` на
  `Type_Ctx`.
- `instantiate_scheme(ctx, scheme) -> ^Type` — свежая InferVar на каждый
  `forall`-id при КАЖДОМ использовании (вызов функции, конструктор,
  вызов метода) — иначе T "зацементировался" бы после первого
  использования.
- **`Module_Graph.symbol_schemes`** — граф-уровневая копия схем,
  накапливается в `resolve_and_typecheck_all` (module_loader.odin) после
  КАЖДОГО модуля. Нужна, потому что `graph.symbol_types` — общий
  указатель на map, растущий кумулятивно между модулями сам по себе, а
  `Type_Ctx.symbol_schemes` — НЕТ (каждый `new_type_ctx()` создавал
  пустую карту). Асимметрия была причиной бага: вызов generic-функции
  ЧЕРЕЗ АЛИАС МОДУЛЯ (`либ.функция(...)`) не получал свежую
  инстанциацию T, в отличие от same-file вызова — найден и исправлен
  (см. коммит "Fix generic inference bug").
- `instantiate_type` (type_cheker.odin) — НЕ имеет `case .Interface` в
  своём `#partial switch`: типы этого kind проходят через неё
  НЕТРОНУТЫМИ (тот же указатель на входе и выходе, fallback `return
  pruned`). Побочный эффект, использованный Стадией 28 (generic-
  интерфейсы): Self-идентичность (`params[0] == iface_type`) в
  `interface_method_types_match` автоматически переживает подстановку T,
  никакой отдельной защиты self-ссылки не потребовалось.
- **Ловушка**: `instantiate_type`'s `case .Struct`/`case .Enum` ОБЯЗАНЫ
  копировать `implemented_interfaces` в свежую инстанциацию (`t2.
  implemented_interfaces = pruned.implemented_interfaces`) — иначе КАЖДАЯ
  инстанциация generic-типа молча теряет факт "реализует интерфейс X",
  даже если шаблон честно его реализует. `.Struct` получил этот фикс в
  Стадии 7; `.Enum` — только в Стадии 25 (недостижимо раньше, enum'ы не
  реализовывали интерфейсы вообще) — если появится третий generic-
  container-подобный `Type_Kind` с собственной instantiate_type-веткой,
  проверить этот же копи-пастабельный момент сразу.
- **Ловушка**: sugar-код (Ord/Eq/Арифметика, Стадия 22/23/25),
  сравнивающий "тот же ли тип у обоих операндов", должен использовать
  `unify_types(left_t, right_t)`, НЕ `left_t == right_t`/`!=` (голый
  указатель) — для generic-типа с НУЛЕВЫМ-payload вариантом (`Опция.
  Нет()`) T не выводится из значения, остаётся свободной InferVar;
  `unify_types` корректно связывает её с другим операндом, `==` ложно
  отвергает как "другой тип" (найдено при Стадии 25, тест на
  `Опция.Нет() < Опция.Есть(1)`).

## Interfaces — номинальная типизация, Self через pointer identity

- Структуры И перечисления (Стадия 25) совместимы с интерфейсом ТОЛЬКО
  через явную `реализация X для Y` — никакой структурной совместимости
  (в отличие от TypeScript).
- **Interface_Value.data — `Value`, не голый указатель** (compiler.odin,
  Стадия 25 расширила с `^Aggregate_Value`). Рантайм-приведение к
  интерфейсу (`.Cast_Interface`, vm.odin) принимает Aggregate_Value
  (структура) ИЛИ Variant_Value (перечисление) — vtable-механизм ищет
  методы по имени `"ИмяТипа::метод"` среди скомпилированных функций,
  kind-агностичен по построению. Мутация ПОЛЯ через интерфейсный тип
  (`x.поле = значение`, где `x` статически типизирован интерфейсом)
  остаётся struct-only — у Variant_Value нет settable-полей.
- **Self-тип**: параметр/возврат метода интерфейса, объявленный как тип
  САМОГО интерфейса (`функ клонировать() -> Копируемое`), механически
  отличим — это позиция, чей resolved-тип УКАЗАТЕЛЬНО равен `iface_type`
  (потому что ПРОХОД 1 регистрирует `iface_type` до того, как ПРОХОД 2
  резолвит аннотации методов в тот же указатель). `interface_method_
  types_match` (type_cheker.odin) требует, чтобы такая позиция у impl'а
  была указательно равна `target_type` (конкретный реализующий тип), а
  НЕ структурно равна интерфейсному типу — иначе тело метода не
  получало бы доступа к полям конкретной структуры.
- **Generic-интерфейсы** (Стадия 28) — `Interface_Decl.type_params`
  (parser.odin), T резолвится в `ctx.current_type_params` при резолве
  `interface_methods` (тот же механизм, что поле generic-структуры).
  T выводится ИМПЛИЦИТНО за каждый `реализация`-блок из конкретных типов
  impl'а — explicit-инстанциация синтаксисом (`Интерфейс(Число)`) НЕ
  поддержана нигде (ни в impl-блоках, ни как аннотация типа параметра)
  — не нужна единственному текущему потребителю (Итерируемое). Перед
  каждой проверкой контракта — свежая InferVar на T через
  `instantiate_type` (иначе T первого impl'а зацементировался бы для
  всех последующих через `unify_types`-связывание в `interface_method_
  types_match`).
- `implements_prelude_interface(ctx, t, iface_sym) -> bool` — nominal
  проверка (указательное сравнение с `t.implemented_interfaces`), НЕ
  "есть ли метод с таким именем" — типизация номинальная.

## Call_Info — sugar-механизм "аннотировать узел, не переписывать AST"

Единая точка истины про то, как `Call_Expr`/`Binary_Expr`/etc должен
компилироваться, когда обычная диспетчеризация не подходит:
`Call_Info :: struct { kind: Call_Kind, symbol_ref, text_name, variant }`,
`map[Expr]Call_Info` на `Type_Ctx`. Тайпчекер во время инференса пишет
запись, keyed по САМОМУ AST-узлу (не переписывая его форму);
`compiler.odin` при кодогене того же узла проверяет map и выбирает
альтернативную ветку компиляции.

Используется (минимум 6 раз, все — Стадия 22/23): Сравниваемое/
Равнозначное (`<`/`>`/`==` на структурах → вызов `сравнить`/`равно`),
4× Арифметика (`+`/`-`/`*`/`/` на структурах), Печатаемое (`печать(x)` →
`.вСтроку()` для implementing-структур), Копируемое (оказался БЕСПЛАТНЫМ
— `.клонировать()` обычный прямой вызов метода, существующий generic
interface-dispatch уже справлялся без нового `Call_Kind` вообще).

**Ограничение механизма** (найдено при проектировании Итерируемого):
Call_Info хорошо решает "как скомпилировать ЭТОТ узел иначе", но НЕ
решает случаи, где сама ФОРМА control flow должна отличаться (не другой
опкод на том же месте, а другая структура ветвления/байндинга). См.
`For_In_Stmt` ниже — там понадобился отдельный механизм.

## For-in (Стадия 17→23) — типизированное решение формы, не sugar на существующих узлах

For-in (`для x в ... цикл ... конец`) ДО Стадии 23 десахаривался ЦЕЛИКОМ
на этапе ПАРСИНГА (`parse_for_stmt_into`, parser.odin) в
`Let_Stmt`+`While_Expr`+`Index_Expr`+`Property_Expr`-цепочку, жёстко под
`.длина()`+`[idx]` (парсер не имеет типов). Добавление iterator protocol
(Итерируемое, Стадия 23) потребовало, чтобы for-in умел выбирать МЕЖДУ
ДВУМЯ РАЗНЫМИ формами control flow (length-check-loop vs
Option-match-loop) — это НЕВОЗМОЖНО выразить через Call_Info (см. выше),
потому что формы отличаются не опкодом на одном месте, а количеством/
структурой AST-подобных операций (Index_Expr vs Match_Tag+
Get_Variant_Field).

**Решение**: `For_In_Stmt` — необесахаренный AST-узел (паттерн/
`в`-выражение/тело как есть), резолвится/типизируется обычным образом.
`infer_for_in_stmt` (type_cheker.odin) решает форму по типу
`в`-выражения и пишет решение в `ctx.for_in_infos[stmt]` (тот же принцип
аннотирования, что Call_Info, просто своя карта, потому что `Stmt`, не
`Expr`, и решение сложнее одного kind). `compile_for_in_stmt`
(compiler.odin) эмитит байткод НАПРЯМУЮ по этому решению — либо
fast-path (idx-счётчик + `.Invoke_Collection` "длина" + `.Get_Index` —
тот же байткод, что раньше строило десахаривание), либо
iterator-protocol (повторный вызов `.следующий()` + `.Match_Tag`/
`.Get_Variant_Field` — те же опкоды, что уже использует компиляция
`выбор`, БЕЗ создания синтетического `Match_Expr`-узла). Оба пути через
`Loop_Context` (`ctx.loops`) — `прервать`/`продолжить` работают
идентично `While_Expr`.

**Паттерн для будущих фич**: если новая фича требует РАЗНОЙ ФОРМЫ
control flow по типу (не просто разного опкода в одном месте) — заводить
собственный необесахаренный AST-узел + typecheck-time решение +
compiler-time прямую эмиссию байткода, а не пытаться либо (а) десахарить
на этапе парсинга (тип неизвестен), либо (б) натянуть на Call_Info
(не покрывает разницу форм).

## Отклонённые/выброшенные из плана архитектурные направления

Эти решения ЯВНО рассматривались и отклонены — если кто-то предложит их
снова, сначала перечитать обоснование здесь, не переоткрывать заново:

- **Value semantics для структур вместо reference semantics** — ⌈не
  сделано⌋. Обсуждалось подробно (пользователь поднял вопрос "не
  нравится reference semantic"). Вывод: подходящая референтная группа
  для panos — Python/JS/Ruby (reference-by-default для объектов), не
  C++/Rust (explicit value/reference). Полная смена модели — retrofit
  такого масштаба, что затронула бы buквально всё: runtime
  representation, GC, calling convention, semантику КАЖДОГО присваивания
  и вызова функции, плюс уже сделанное поверх текущей модели (Стадия 24
  actor model's copy-on-send дизайн ЯВНО спроектирован вокруг того, что
  структуры — reference). Альтернатива (обсуждена, тоже не принята):
  структуры immutable по умолчанию + explicit `Ячейка(T)`-обёртка для
  mutable-shared состояния (Rust `Cell`/`RefCell`/OCaml `ref`-style) —
  концептуально элегантнее, но требует переосмысления МУТАЦИИ как
  таковой в языке, не просто присваивания. **Оставлено как есть**.
- **Явные C++-style ссылочные типы (`T` vs `&T`)** — ⌈не сделано⌋, тот
  же контекст, что выше. Панос сейчас не имеет ссылок/указателей как
  типов первого класса вообще.
- **Хешируемое (интерфейс для хеш-кода в Map-ключах)** — ⌈выброшено из
  плана⌋. Соответствие — не хеш-таблица (см. раздел выше), интерфейс
  без реальной подсистемы под ним бессмыслен. Если понадобятся
  struct/enum-ключи — вопрос ОДНОЙ строки (`is_valid_map_key_type`), не
  нового интерфейса. Если понадобится O(1) lookup — отдельный VM-perf
  проект (настоящая хеш-таблица), не языковая фича.
- **По-умолчанию (Default-интерфейс, `тип() -> Тип`)** — ⌈выброшено из
  плана⌋. Единственный названный потребитель (`Массив.заполнить(N)`) не
  существует нигде в stdlib — мотивация была спекулятивной.
- **Полный переход type checker'а на constraint-based** — ⌈отклонено⌋,
  см. раздел выше про eager/constraint-based сплит.

## Известные ограничения (не баги, осознанные trade-off'ы)

- Structural dump (Печатаемое fallback) для структур без явной
  `реализация Печатаемое` — позиционный, без имени типа/полей (нет RTTI,
  см. выше).
- `Копируемое`/`Итерируемое`/`Печатаемое`/`Сравниваемое`/`Равнозначное`/
  Арифметика — НЕТ auto-derive: тело метода пишется руками во всех
  случаях. Для `Копируемое` это значит: "глубокая копия" не проверяется
  языком — если тело забудет рекурсивно клонировать вложенное
  struct-поле, получится тихая поверхностная копия. Задокументировано,
  не предотвращается компилятором.
- Generic-интерфейсы (Стадия 28) не поддерживают explicit type-argument
  синтаксис (`Итератор(Число)`) — T только имплицитный, выводится per-impl.
- Параметры функций/лямбд immutable по умолчанию (Стадия 27,
  Kotlin/Swift-style) — БЕЗ opt-out синтаксиса. Для мутации — копия в
  локальный `пер`.
- `конст`-биндинги (Стадия 27) — binding-immutability, НЕ deep
  immutability: `конст x = Точка(1,2); x.поле = 5` разрешено (структуры
  reference semantics, конст фиксирует только сам биндинг).

panos

Введение

panos — экспериментальный скриптовый язык программирования с русскими ключевыми словами и синтаксисом, близким к Go и Rust. Статическая типизация, структуры и интерфейсы, перечисления (ADT) с pattern matching, дженерики, замыкания, простой mark-and-sweep сборщик мусора.

Каждый пример кода в этой документации — рабочая программа, которую можно отредактировать и запустить прямо на странице (кнопка «▶ Запустить» под блоком кода — тот же WASM-компилятор и VM, что и в отдельной песочнице).

тип Фигура = перечисление
    Точка
    Круг(Число)
    Прямоугольник(Число, Число)
конец

функ площадь(ф: Фигура) -> Число
    возврат выбор ф
        Точка -> 0
        Круг(р) -> р * р * 314 / 100
        Прямоугольник(ш, в) -> ш * в
    конец
конец

функ старт() -> Число
    площадь(Фигура.Прямоугольник(3, 4))
конец

Это не учебный/игрушечный язык в смысле «одна фича ради демонстрации» — компилятор целиком: лексер → парсер → resolver → type checker → компилятор в байткод → VM, плюс LSP-сервер (panos-lsp) с диагностикой, hover, автокомплитом и rename/references на весь граф импортов.

Реализация написана на Odin и активно развивается — документ описывает текущее поведение языка, а не финальный дизайн. Смотрите ROADMAP.md в репозитории за списком нереализованных пока направлений (FFI, дальнейшие DOD-волны и т.д.).

Установка

Готовые бинарники

Каждый релиз собирает два бинарника — panos (CLI-интерпретатор) и panos-lsp (LSP-сервер) — под три платформы:

ПлатформаАрхив/файл
Linux (amd64)panos-linux-amd64, panos-lsp-linux-amd64
macOS (arm64)panos-macos-arm64, panos-lsp-macos-arm64
Windows (amd64)panos-windows-amd64.exe, panos-lsp-windows-amd64.exe

Скачайте нужные файлы со страницы последнего релиза, переименуйте (или оставьте как есть) и положите куда-нибудь в PATH. На macOS/Linux не забудьте выставить бит исполнения:

chmod +x panos-macos-arm64
mv panos-macos-arm64 /usr/local/bin/panos

Готовых сборок под Intel-Mac (macos-13) и arm64-Linux пока нет — только собрать из исходников (ниже).

Сборка из исходников

Нужен установленный Odin — тот же тулчейн, что использует CI (dev-2026-06, см. .github/workflows/*.yml; более новые dev-сборки Odin, скорее всего, тоже подойдут, но не гарантировано — API core: иногда меняется между dev-релизами).

git clone https://github.com/Garius6/panos.git
cd panos

# CLI-интерпретатор
odin build . -out:panos
# или, если установлен just (https://github.com/casey/just):
just build

# LSP-сервер
odin build ./lsp -out:panos-lsp
just build-lsp

# WASM-сборка для браузера (демо в demo/)
odin build wasm -target:js_wasm32 -o:size -out:demo/panos.wasm
just build-wasm

just build-all собирает все три сразу. Полный список команд — в Justfile в корне репозитория.

Hello world в браузере

Ставить ничего не нужно — веб-песочница компилирует и исполняет panos прямо в браузере (тот же компилятор и VM, собранные в WASM). Там же по умолчанию открыт test.ps — файл-пример, который использует практически все конструкции языка сразу: структуры, интерфейсы, дженерики, ADT и выбор (включая литеральные шаблоны), деструктуризация, Опция/Результат, циклы, замыкания. Хороший способ посмотреть, «как это всё выглядит вместе», не читая документацию по главам.

Ограничения браузерной версии (в отличие от нативного CLI):

  • Нет локальных файловых импортов — своего второго .ps-файла в песочницу не положить (один текстовый буфер). Стандартная библиотека (математика, коллекции, строки и т.д.) при этом работает — она вшита прямо в WASM-сборку. Подробности и точный список — в разделе «Модули» → ограничение в браузере.
  • Нет доступа к файловой системе/сети/stdin — фс, ос, ввод_вывод, сеть импортируются без ошибки, но вызов их функций падает в панику во время выполнения.

Использование CLI

Запуск файла

panos файл.ps [аргументы...]

Всё после имени файла — аргументы самой программы, не флаги panos. Прочитать их внутри скрипта:

импорт ос

функ старт() -> Строка
    пер аргументы = ос.аргументы()
    если длина(аргументы) == 0 тогда
        "нет аргументов"
    иначе
        аргументы[0]
    конец
конец

panos файл.ps привет мир
# -> "привет" (ос.аргументы() == ["привет", "мир"])

Без аргумента-файла panos открывает REPL.

-v / --verbose

Флаг обязан стоять до имени файла (после — уже аргумент программы):

panos -v файл.ps

Печатает четыре секции по порядку — AST, результат типизации, ассемблерный дамп скомпилированного байткода, и уже затем обычный вывод исполнения. Полезно, чтобы посмотреть, во что разворачивается конкретная конструкция языка, или разобраться в диагностике компилятора; для обычного запуска не нужен.

Поиск модулей

импорт имя (не путь в кавычках) ищет файл имя.ps по очереди в четырёх местах, использует первое найденное:

  1. рядом с импортирующим файлом (тот же каталог)
  2. в подкаталоге модули/ рядом с импортирующим файлом
  3. в каталоге, на который указывает переменная окружения PANOS_STDLIB
  4. в каталоге std/ — там же, где сам бинарник panos (пакетная стандартная библиотека)

Если имя совпадает со встроенным нативным модулем (ввод_вывод, фс, ос, сеть, строки) и файл нигде не нашёлся — используется он. Практическое следствие: положить файл с таким же именем в свой проект можно, но он молча перекроет нативный модуль, если найдётся раньше него в этом порядке поиска.

Импорт вложенного стандартного модуля (std/сеть/http.ps, std/кодирование/json.ps) требует пути в кавычках, потому что имя с / не проходит как обычный идентификатор:

импорт "сеть/http" как http
импорт "кодирование/json" как json

Базовые типы и операторы

Кодировка и идентификаторы

Исходники — UTF-8. Ключевые слова, стандартные типы и большая часть диагностик компилятора русскоязычные. Идентификаторы (имена переменных, функций, типов) не обязаны быть русскими — можно смешивать латиницу и кириллицу как удобно, компилятору всё равно.

Переменные

функ старт() -> Строка
    пер x = 10
    конст имя: Строка = "panos"
    имя
конец

пер объявляет обычную (переприсваиваемую) переменную, конст — неизменяемый биндинг: имя = "другое" после конст имя = ... даёт ошибку типизации (“попытка переприсвоить константу”). Оба — полноценные объявления переменной, различие только в возможности переприсвоения. конст — binding-immutability, не глубокая неизменяемость: если имя ссылается на структуру, её поля (имя.поле = ...) всё равно можно менять — у panos структуры reference semantics (см. «Структуры и методы»), конст фиксирует только сам биндинг, не то, на что он указывает.

Аннотация типа (: Тип) необязательна для обеих форм, если type checker может вывести тип из выражения инициализации. Переменная не может иметь тип Пусто — это ловится как ошибка на этапе типизации, не в рантайме.

Shadowing (повторное объявление того же имени, пер/конст в любой комбинации) запрещено в ТОМ ЖЕ scope (“Имя x уже объявлено”), но разрешено во ВЛОЖЕННОМ scope — внутренний биндинг временно затеняет внешний, снаружи блока видно исходное значение.

Базовые типы

  • Число — единственный числовой тип (нет отдельных int/float).
  • Булево — истина / ложь.
  • Строка.
  • Пусто — тип отсутствия значения (аналог void); не может быть типом переменной.
  • Никогда — тип выражений, которые никогда не возвращают обычное значение (например, паника(...)); нет синтаксиса для явной аннотации этим типом, он появляется только выведенным.

Арифметика и сравнение

функ старт() -> Строка
    пер a = 10 + 5 * 2      // 20
    пер b = a > 12          // истина
    пер c = "pan" + "os"    // "panos"
    c
конец

+ - * / — только числа, кроме +, который дополнительно работает как конкатенация строк. Остаток (%) и возведение в степень (**/^) как операторы не поддержаны — для степени есть математика.степень в стандартной библиотеке.

Сравнения: < > <= >= == <> (обратите внимание — «не равно» это <>, а не привычный !=). Все шесть числовые: </>/<=/>= не определены для строк, структур, массивов и т.п.

Логические операторы

функ старт() -> Булево
    (истина и ложь) или (не ложь)
конец

и/или — инфиксные ключевые слова (не &&/||), не — префиксный унарный оператор. и связывает крепче или и слабее сравнений: a == b и c == d читается как (a == b) и (c == d), без скобок вокруг сравнений.

Odin-рантайм печатает булевы значения как true/false (например, в выводе примеров на этой странице) — это формат вывода нативной сборки, не часть синтаксиса языка. Внутри самого panos-кода булевы литералы всегда истина/ложь.

Строки

Двойные кавычки, стандартный набор escape-последовательностей:

  • \n — перевод строки
  • \t — табуляция
  • \r — возврат каретки
  • \" — кавычка внутри строки
  • \\ — обратный слэш
функ старт() -> Строка
    "ключ = \"значение\"\n"
конец

Индексирование строки по числу возвращает односимвольную строку:

функ старт() -> Строка
    "panos"[0]
конец

Тупы

функ старт() -> Число
    пер позиция = (10, 25)
    позиция.0 + позиция.1
конец

Тип — (Число, Число). Доступ к полям по позиции через .0, .1 и т.д., без именования полей. Тупл можно и разобрать целиком на отдельные биндинги — пер (a, b) = позиция, см. «Структуры и методы» → деструктуризация.

Остальные составные типы

Встречаются по мере знакомства с языком — здесь только имена для справки:

  • функция: функ(Число, Строка) -> Булево — глава «Функции»
  • массив: Массив(Число) — глава «Массивы и соответствия»
  • соответствие (карта): Соответствие(Строка, Число) — там же
  • опция: Опция(Число) — глава «Опция и Результат»
  • результат: Результат(Число, Ошибка) — там же

Функции

Объявление

функ сложить(a: Число, b: Число) -> Число
    a + b
конец

функ старт() -> Число
    сложить(3, 4)
конец

Типы параметров и возвращаемого значения у именованной функции всегда обязательны — -> Тип нельзя опустить, даже если функция ничего осмысленного не возвращает (тогда пишется -> Пусто). Тело — это последовательность выражений/операторов; значение последнего выражения и есть результат функции.

импорт ввод_вывод

функ записать_лог(сообщение: Строка) -> Пусто
    ввод_вывод.печать(сообщение)
конец

функ старт() -> Пусто
    записать_лог("привет")
конец

Этот пример использует импорт — стандартная библиотека недоступна в браузере (подробнее в разделе про модули), поэтому блок не запускается прямо на странице. Запустите локально: ./panos файл.ps.

Значение с типом, отличным от Пусто, в конце Пусто-функции просто отбрасывается — не обязательно явно приводить последнее выражение к Пусто.

Явный возврат

функ проверить(x: Число) -> Строка
    если x < 0 тогда
        возврат "отрицательное"
    конец
    "неотрицательное"
конец

функ старт() -> Строка
    проверить(-5)
конец

возврат — обычный ранний выход, не обязателен, если функция и так заканчивается нужным значением.

Параметры неизменяемы

Параметры функций и лямбд immutable по умолчанию (как в Kotlin/ Swift) — переприсвоить их внутри тела нельзя:

функ f(x: Число) -> Число
    x = x + 1  // Type Error: попытка переприсвоить константу 'x'
    x
конец

Opt-out’а нет — если параметр нужно менять, копируйте его в локальный пер в начале тела:

функ f(x: Число) -> Число
    пер счётчик = x
    счётчик = счётчик + 1
    счётчик
конец

Это ограничение касается только САМОГО биндинга параметра (нельзя писать x = ...), не того, на что он ссылается — если параметр структурного типа, его поля (x.поле = ...) по-прежнему мутируемы (у panos структур reference semantics, см. «Переменные» и «Структуры и методы»).

Точка входа

Программа начинает выполнение с функции старт. У неё может быть любой подходящий возвращаемый тип — движок печатает результат старт(), если он есть (именно так на этой странице печатается вывод под каждым примером).

Функции как значения и лямбды

Тип функции записывается как функ(Тип1, Тип2, ...) -> ТипРезультата. Функции — значения первого класса: их можно передавать как аргументы, хранить в переменных.

функ применить(ф: функ(Число) -> Число, значение: Число) -> Число
    ф(значение)
конец

функ старт() -> Число
    применить(функ(x)
        x + 1
    конец, 41)
конец

Лямбда — это функ(параметры) тело конец без имени. Типы параметров и результата лямбды можно не указывать явно, если type checker способен вывести их из ожидаемого типа (выше — из типа параметра ф у применить). Если ожидаемый тип функции неизвестен компилятору в месте, где стоит лямбда, вывод типов параметров может не сработать — тогда нужно аннотировать их явно, как у обычной функции (функ(x: Число) ... конец).

Управляющие конструкции

If как выражение

функ старт() -> Строка
    пер x = 5
    если x > 0 тогда
        "плюс"
    иначе
        "ноль или минус"
    конец
конец

если — выражение, не оператор. Если есть иначе, обе ветки должны иметь совместимые типы (веткa с паника(...) имеет тип Никогда и совместима с любой другой веткой). если без иначе имеет тип Пусто.

пока

функ старт() -> Число
    пер сумма = 0
    пер i = 0

    пока i < 10 цикл
        i = i + 1

        если i == 3 тогда
            продолжить
        конец

        если i > 7 тогда
            прервать
        конец

        сумма = сумма + i
    конец

    сумма
конец

пока используется ради побочных эффектов и сам по себе имеет тип Пусто. Внутри — продолжить (перейти к следующей проверке условия) и прервать (выйти из цикла).

для

для — синтаксический сахар над пока (разворачивается на этапе разбора, никакой отдельной сущности в рантайме нет). Три формы.

По элементам итерируемого значения:

функ старт() -> Число
    пер сумма = 0
    для элемент в массив(1, 2, 3) цикл
        сумма = сумма + элемент
    конец
    сумма
конец

По записям карты (карта сама не индексируется — только через .записи()):

функ старт() -> Число
    пер карта: Соответствие(Строка, Число) = соответствие(
        "a" = 1,
        "b" = 2,
    )
    пер сумма = 0
    для (ключ, значение) в карта.записи() цикл
        сумма = сумма + значение
    конец
    сумма
конец

Числовой диапазон, границы включительны с обеих сторон:

    конец

конец

Деструктуризация

пер Тип(a, b, ...) = значение разбирает структуру на отдельные биндинги по полям — в том же порядке, в каком поля объявлены (симметрично конструктору выше):

тип Точка = структура
    x: Число
    y: Число
конец

функ старт() -> Число
    пер точка = Точка(3, 4)
    пер Точка(x, y) = точка
    x + y
конец

Тот же синтаксис без имени типа — пер (a, b, ...) = кортеж — разбирает тупл:

функ старт() -> Число
    пер (a, b, c) = (1, 2, 3)
    a + b + c
конец

Обе формы требуют точного совпадения количества имён с количеством полей/элементов — несовпадение или неверный тип структуры (пер Вектор(x, y) = точка, где точка на самом деле Точка) — ошибка компиляции. конст (a, b) = ... тоже работает — все извлечённые имена становятся неизменяемыми, как обычный конст.

Типизация номинальная, не структурная: тип реализует интерфейс только через явный блок реализация ... для ..., случайное совпадение сигнатур методов не считается.

Интерфейсный тип у переменной/параметра даёт динамическую диспетчеризацию — какой конкретно метод вызовется, решается по фактическому значению в рантайме:

конец

конец

конец

реализация Фигура для Квадрат
    функ площадь(это: Квадрат) -> Число
        это.сторона * это.сторона
    конец
конец

функ общая_площадь(a: Фигура, b: Фигура) -> Число
    a.площадь() + b.площадь()
конец

конец

Generic-интерфейсы

Параметр типа может принадлежать самому интерфейсу, а не только реализующей его структуре/перечислению — T в квадратных скобках после имени интерфейса:

тип Контейнер[T] = интерфейс
    функ получить_значение() -> T
конец

тип Обёртка = структура
    x: Число
конец

реализация Контейнер для Обёртка
    функ получить_значение(это: Обёртка) -> Число
        это.x
    конец
конец

функ старт() -> Число
    пер о = Обёртка(42)
    о.получить_значение()
конец

В реализация Контейнер для Обёртка тип-аргумент писать не нужно — T выводится из конкретной сигнатуры метода в теле реализации (здесь — Число, потому что получить_значение возвращает это.x: Число). Разные реализации одного generic-интерфейса свободно фиксируют разные T.

Главный пример в языке — Итерируемое[T] из прелюдии, который управляет тем, что для x в значение цикл умеет обходить произвольные пользовательские типы, а не только Массив/ Соответствие.

Интерфейсы прелюдии

Шесть интерфейсов из прелюдии — она доступна каждому файлу без импорт, как Опция/Результат (см. «Модули») — дают пользовательским типам операторный синтаксис (<, ==, + и т.д.) и управляют тем, как значение печатается, копируется и обходится в для-in. Реализуются они так же, как любой другой интерфейс — реализация Интерфейс для Тип — просто typechecker и компилятор узнают эти конкретные имена и превращают вызов оператора/встроенной функции в вызов нужного метода.

Сравниваемое

тип Деньги = структура
    копейки: Число
конец

реализация Сравниваемое для Деньги
    функ сравнить(это: Деньги, другое: Деньги) -> Число
        это.копейки - другое.копейки
    конец
конец

функ старт() -> Булево

конец

сравнить возвращает Число по конвенции отрицательное/ноль/ положительное — этого достаточно, чтобы компилятор выразил через него все четыре оператора: <, >, <=, >=. У численных типов </> и так работают из коробки; для остальных (структур, перечислений) реализация Сравниваемое — единственный способ получить эти операторы, дефолта нет.

Четыре независимых интерфейса — Складываемое/Вычитаемое/Умножаемое/ Делимое, методы сложить/вычесть/умножить/разделить — вместо одного, потому что операции независимы: вектор естественно поддерживает сложение и вычитание, но не деление на другой вектор. Реализуйте только те, которые осмысленны для вашего типа.

Структуры в panos — reference-типы: присваивание (пер b = a) копирует указатель, не поля, так что мутация через b была бы видна и через a. .клонировать() — единственный способ получить настоящую независимую копию. Тот же метод используется как copy-on-send override для сообщений между процессами.

Итерируемое[T] — единственный в языке generic-интерфейс (сам параметр T объявлен у интерфейса, не только у реализующего типа — см. «Интерфейсы» → generic-интерфейсы). Реализующий тип сам хранит и мутирует своё состояние между вызовами — это неизменяемый биндинг, но поля через это.поле = ... мутируемы (reference semantics структур). для x в значение цикл ... конец на типе с реализация Итерируемое вызывает .следующий() в цикле автоматически; без реализации для-in работает только на Массив/Соответствие напрямую.

Перечисления (ADT) и выбор

Варианты

Вариант без полей, с одним полем, с несколькими полями — можно смешивать в одном перечислении:

тип Фигура = перечисление
    Точка
    Круг(Число)
    Прямоугольник(Число, Число)
конец

функ старт() -> Число
    пер ф: Фигура = Фигура.Прямоугольник(3, 4)
    выбор ф
        Точка -> 0
        Круг(р) -> р * р * 314 / 100
        Прямоугольник(ш, в) -> ш * в
    конец
конец

выбор обязан покрыть все варианты (или иметь _ — wildcard-ветку). Непокрытый вариант — ошибка компиляции, не рантайма.

тип Фигура = перечисление
    Точка
    Круг(Число)
    Прямоугольник(Число, Число)
конец

функ старт() -> Строка
    пер ф: Фигура = Фигура.Круг(10)
    выбор ф
        Точка -> "точка"
        Круг(_) -> "круг"
        _ -> "другое"
    конец
конец

Конструктор варианта без полей вызывается по имени, без скобок (Точка, не Точка()).

    выбор о
        Опция.Есть(х) -> х
        Нет -> -1
    конец
конец

конец

Литеральные шаблоны

выбор работает не только на перечислениях — на Число/Строка/ Булево тоже, с ветками-литералами:

функ описать(x: Число) -> Строка
    выбор x
        1 -> "один"
        2 -> "два"
        _ -> "другое"
    конец
конец

функ старт() -> Строка
    описать(1) + " " + описать(99)
конец

Для Число/Строка домен значений неперечислим — компилятор не может доказать, что вы покрыли всё, поэтому финальная ветка _ (или биндер, как в примере ниже) обязательна. Булево — исключение: у него ровно два значения, и истина/ложь вместе дают настоящую исчерпывающую проверку без _:

функ старт() -> Строка
    выбор истина
        истина -> "да"
        ложь -> "нет"
    конец
конец

Ветка-биндер (без _, просто имя) ловит “всё остальное” и связывает в него подошедшее значение — как и биндер на перечислении:

функ разобрать(s: Строка) -> Число
    выбор s
        "да" -> 1
        "нет" -> 0
        остальное -> -1
    конец
конец

Литеральные шаблоны работают и внутри конструктора варианта, ограничивая конкретное поле:

тип Событие = перечисление
    Клик(Число, Число)
конец

функ старт() -> Строка
    пер e = Событие.Клик(1, 2)
    выбор e
        Событие.Клик(1, y) -> "клик по x=1"
        Событие.Клик(x, y) -> "обычный клик"
    конец
конец

Отрицательные числовые литералы (-1 -> ...) как шаблон не поддерживаются — грамматика неоднозначна на границе строк с бинарным вычитанием.

Конструктор-шаблоны на структурах

Тип(...) работает не только на вариантах перечисления — на структурах тоже, с теми же под-шаблонами (литерал, _, биндер, вложенный конструктор), разобранными по полям в порядке их объявления:

тип Точка = структура
    x: Число
    y: Число
конец

функ классифицировать(п: Точка) -> Строка
    выбор п
        Точка(0, 0) -> "начало координат"
        Точка(0, y) -> "на оси Y"
        Точка(x, 0) -> "на оси X"
        Точка(_, _) -> "где-то ещё"
    конец
конец

функ старт() -> Строка
    классифицировать(Точка(0, 5))
конец

У структуры, в отличие от перечисления, нет вариантов — ветка Тип(...), где ВСЕ под-шаблоны _/биндер (как Точка(_, _) выше), покрывает структуру целиком и действует как catch-all — обязана быть последней, как и голый _. Ветка с частично сужающими под-шаблонами (Точка(0, 0)) сама по себе ничего не покрывает — без финальной catch-all ветки компилятор требует её явно, той же логикой, что и для Число/Строка выше.

Прелюдия

Опция/Результат — исключение из правила «нужен импорт»: это обычные generic-перечисления, объявленные во встроенной прелюдии, доступной каждому файлу неявно, без единого импорт. Подробнее — в главе «Опция и Результат».

Ограничение в браузере

Весь std/*.ps (математика, коллекции, флаги, тест, кодирование/json, кодирование/toml, сеть/http) вшит прямо в WASM-сборку демо (нет файловой системы, которой можно было бы его отдать), так что импорт этих модулей работает и в интерактивном редакторе. строки — тоже работает (это встроенный модуль языка, не файл). Но у флаги и сеть/http это работает только частично — сами типы/чистые функции доступны, а вызов того, что внутри них трогает ос (.распарсить()) или сеть (http.получить и т.п.), всё равно упадёт в панику ниже — вшитость файла не убирает его внутренние зависимости.

Не работает то, что реально требует файловой системы/сети/стдина: фс, ос, ввод_вывод, сеть — модули импортируются (не ошибка на импорт), но вызов их функций падает в понятную панику “недоступно в браузере” во время выполнения. Не работают и локальные файловые импорты пользовательских модулей (пример с геометрия.ps ниже) — у браузерного редактора всего один текстовый буфер, положить туда второй файл негде.

Примеры с такими зависимостями на этой странице помечены как нерабочие в интерактивном редакторе (нет кнопки «▶ Запустить»). Запустите их локально: ./panos файл.ps.

Процессы

panos поддерживает лёгковесные процессы в духе Erlang/Akka: изолированные единицы исполнения, которые общаются только сообщениями через mailbox — никакой разделяемой памяти между ними. Планировщик однопоточный и кооперативный (без реального параллелизма), но модель изоляции та же, что у настоящих акторов.

Порождение процесса

тип Сообщение = перечисление
    Увеличить
конец

функ счётчик(состояние: Число) -> Пусто
    выбор получить()
        Сообщение.Увеличить -> счётчик(состояние + 1)
    конец
конец

функ старт() -> Строка
    пер proc: Процесс(Сообщение) = запусти счётчик(0)
    отправить(proc, Сообщение.Увеличить)
    отправить(proc, Сообщение.Увеличить)
    "готово"
конец

запусти <вызов> порождает новый процесс, который начинает исполнять тело вызванной функции, и сразу возвращает типизированный хэндл Процесс(T) — T здесь означает «этот процесс принимает сообщения типа T». Сам вызов не выполняется в текущем процессе — только его функция и уже вычисленные аргументы передаются новому.

отправить(процесс, сообщение) кладёт сообщение в FIFO-mailbox адресата — обычная функция, не оператор, ничего не блокирует.

Приём сообщений

получить() — блокирующий (точнее, кооперативно приостанавливающий текущий процесс) builtin: если mailbox пуст, процесс засыпает до следующего сообщения; когда оно приходит, получить() возвращает его, и выполнение продолжается с того же места.

Тип T у получить() не объявляется явно — он выводится из того, как вы разбираете результат в выбор. Ветки обязаны использовать полную квалификацию Тип.Вариант (Сообщение.Увеличить, не голое Увеличить) — только так typechecker узнаёт, какое перечисление ожидать:

тип Команда = перечисление
    Стоп
    Ход(Число)
конец

функ обработчик(пройдено: Число) -> Число
    выбор получить()
        Команда.Стоп -> пройдено
        Команда.Ход(шаг) -> обработчик(пройдено + шаг)
    конец
конец

функ старт() -> Строка
    пер proc: Процесс(Команда) = запусти обработчик(0)
    отправить(proc, Команда.Ход(5))
    отправить(proc, Команда.Стоп)
    "готово"
конец

Тело процесса — обычно рекурсивная функция: каждая ветка выбор либо вызывает себя же (с обновлённым состоянием как явным аргументом, никаких мутируемых полей объекта — состояние процесса это просто параметры функции), либо завершает работу, возвращая значение (тип возврата у обработчик — просто обычный тип функции, запусти его не читает и никак не использует после завершения процесса).

Ответ отправителю: себя()

Чтобы процесс мог ответить, ему нужен адрес отправителя. себя() — тоже builtin, возвращает Процесс(T)-хэндл текущего процесса, который можно передать как часть сообщения:

тип Запрос = перечисление
    Прочитать(Процесс(Число))
конец

функ хранилище() -> Пусто
    выбор получить()
        Запрос.Прочитать(отвечающему) -> отправить(отвечающему, 77)
    конец
конец

функ старт() -> Число
    пер proc: Процесс(Запрос) = запусти хранилище()
    отправить(proc, Запрос.Прочитать(себя()))
    получить()
конец

старт() — обычный процесс (тот самый, с которого начинается программа), и получить() на его собственном mailbox работает так же, как везде: ждёт, пока хранилище() пришлёт ответ, и разворачивает результат.

Copy-on-send

Сообщение, отправленное через отправить, копируется в момент отправки — мутация исходного значения после отправки не видна получателю, и наоборот. Копирование рекурсивное (глубокое) и происходит автоматически; переопределить его для конкретного типа можно через реализация Копируемое (см. «Интерфейсы прелюдии»). Примитивы (Число/Булево) и строки не копируются — они и так неизменяемы, шарить их безопасно.

Жизненный цикл

Процесс завершается, когда его функция возвращает значение (или доходит до конца тела). Программа в целом завершается сразу, как только вернулся первый процесс — тот, что начал работу с старт() — не дожидаясь остальных, даже если они ещё живы или ждут сообщений. Это то же поведение, что у main() в Go: осиротевшие процессы просто перестают существовать вместе со всей программой.

Отправка сообщения уже завершившемуся процессу — тихий no-op, не ошибка и не Результат: синхронная проверка «жив ли процесс» на момент отправки всё равно ничего не гарантирует (он может завершиться микросекундой позже), так что Результат создавал бы ложное чувство надёжности.

Дедлок

Если получить() блокирует процесс, а прислать ему сообщение больше некому (все остальные процессы либо завершились, либо сами ждут), это настоящий, а не приближённый дедлок — планировщик обнаруживает его и завершает программу понятной ошибкой вместо зависания:

тип Сообщение = перечисление
    X
конец

функ старт() -> Число
    выбор получить()
        Сообщение.X -> 0
    конец
конец

Runtime Error: все процессы заблокированы в ожидании сообщений (дедлок)

функ старт() -> Пусто

    ввод_вывод.строка("с переводом строки")
конец

.печать(значение)/.строка(значение) принимают значение любого типа: если он реализует Печатаемое, используется .вСтроку(), иначе — structural-дамп полей.

импорт ввод_вывод

функ старт() -> Строка
    ввод_вывод.прочитать_строку()
конец

ввод_вывод.прочитать_строку() -> Результат(Строка, Ошибка) читает одну строку из stdin. Для нескольких строк подряд эффективнее ввод_вывод.поток() — он даёт объект с тем же интерфейсом, что и файл (см. ниже), так что .прочитать_строку() на нём можно вызывать в цикле, не открывая stdin заново на каждый вызов.

фс: файлы целиком

импорт фс

функ старт() -> Строка
    фс.записать("заметки.txt", "первая строка\n")
    пер результат = фс.прочитать("заметки.txt")
    если результат.успех() тогда
        результат.значение()
    иначе
        "не удалось прочитать: " + результат.причина().сообщение
    конец
конец
  • фс.есть(путь) -> Булево
  • фс.прочитать(путь) -> Результат(Строка, Ошибка) — весь файл целиком
  • фс.записать(путь, содержимое) -> Результат(Число, Ошибка) — перезаписывает файл целиком (создаёт, если его не было), число — количество записанных байт

Файловые дескрипторы: построчная работа

Когда файла целиком мало — например, нужно писать по частям или читать построчно — фс.открыть(путь) даёт дескриптор (создаёт файл, если его не было; открывает на чтение и запись):

импорт фс

функ старт() -> Строка
    пер файл = фс.открыть("лог.txt").ожидать("не удалось открыть лог")
    файл.записать("событие 1\n")
    файл.записать("событие 2\n")
    файл.закрыть()

    пер снова = фс.открыть("лог.txt").ожидать("не удалось открыть лог")
    пер первая_строка = снова.прочитать_строку().получить("(пусто)")
    снова.закрыть()
    первая_строка
конец

Методы дескриптора (тот же набор что у ввод_вывод.поток()):

  • .прочитать() -> Результат(Строка, Ошибка) — всё оставшееся содержимое
  • .прочитать_строку() -> Результат(Строка, Ошибка) — одна строка
  • .записать(текст) -> Результат(Число, Ошибка)
  • .закрыть() -> Пусто

Вызов метода на уже закрытом дескрипторе не паникует — возвращает Результат.Неудача(Ошибка(...)), как любая другая I/O-ошибка.

импорт ос

конец
  • ос.окружение(имя) -> Опция(Строка)
  • ос.установить_окружение(имя, значение) -> Результат(Число, Ошибка)
  • ос.удалить_окружение(имя) -> Булево
  • ос.аргументы() -> Массив(Строка) — аргументы командной строки, переданные скрипту (не считая имени файла и флагов самого panos), см. «Использование CLI».

Сеть и HTTP

Требует импорт сеть и реальную сеть — недоступно в браузерной песочнице (см. «Модули» → ограничение в браузере). Примеры не запускаются на странице, запускайте локально: ./panos файл.ps.

сеть: TCP-клиент

импорт сеть

функ старт() -> Строка
    пер соединение = сеть.подключиться("example.com", 80).ожидать("не удалось подключиться")
    соединение.отправить("GET / HTTP/1.1\r\nHost: example.com\r\nConnection: close\r\n\r\n")
    пер ответ = соединение.получить().получить("(пусто)")
    соединение.закрыть()
    ответ
конец
  • сеть.подключиться(хост, порт) -> Результат(Соединение, Ошибка) — TCP, без TLS (https:// не поддержан, см. ниже)
  • .отправить(текст) -> Результат(Число, Ошибка)
  • .получить() -> Результат(Строка, Ошибка) — читает до закрытия соединения удалённой стороной (до EOF)
  • .получить_строку() -> Результат(Строка, Ошибка) — одна строка
  • .закрыть() -> Пусто
  • сеть.кодировать_url(текст) -> Строка — percent-encoding (сеть.кодировать_url("а б") → "%D0%B0%20%D0%B1")

сеть/http: HTTP/1.1-клиент

Простой клиент поверх сеть, написан на самом panos (std/сеть/http.ps) — импортируется по пути в кавычках, потому что имя содержит /:

импорт "сеть/http" как http
импорт строки

функ старт() -> Строка
    пер ответ = http.получить("http://example.com/").ожидать("запрос не удался")
    если ответ.успех() тогда
        ответ.тело
    иначе
        "код ответа: " + строки.из_числа(ответ.статус)
    конец
конец

Только http:// — TLS не поддержан (core:net Odin не даёт его “из коробки”, отдельная криптобиблиотека вне текущего объёма). Каждый запрос шлётся с Connection: close, тело читается до конца соединения — без разбора Content-Length/chunked encoding на стороне клиента, так что сервер должен закрывать соединение сам после ответа (обычные HTTP/1.1 серверы так и делают).

Функции:

  • http.получить(ссылка) -> Результат(Ответ, Ошибка)
  • http.получить_с_заголовками(ссылка, заголовки) -> Результат(Ответ, Ошибка)
  • http.отправить_пост(ссылка, тело) -> Результат(Ответ, Ошибка)
  • http.отправить_пост_с_заголовками(ссылка, тело, заголовки) -> Результат(Ответ, Ошибка)
  • http.построить_query(параметры) -> Строка — собирает ?ключ=значение&... из Соответствие(Строка, Строка), значения уже percent-encoded

Тип Ответ:

тип Ответ = структура
    статус:    Число
    заголовки: Соответствие(Строка, Строка)
    тело:      Строка
конец

Методы: .успех() (2xx), .редирект() (3xx), .ошибка_клиента() (4xx), .ошибка_сервера() (5xx), .заголовок(имя) -> Опция(Строка).

Стандартная библиотека

Модули из std/ (репозиторий: std/математика.ps, std/коллекции.ps и т.д.) — написаны на самом panos, а не встроены в компилятор. Резолвятся как обычный импорт (см. «Модули») через PANOS_STDLIB/ модули//репозиторный std/. HTTP-клиент (сеть/http) вынесен в «Сеть и HTTP» — он использует сеть, поэтому недоступен в браузере; всё остальное на этой странице — чистые вычисления, работает и в интерактивной песочнице.

функ старт() -> Число

пи(), е(), квадрат(x), абс(x), мин(a, b), макс(a, b), степень(x, n) (целая степень, n может быть отрицательным), квадратный_корень(x) (метод Ньютона, x <= 0 даёт 0 — без Опция/ паники). коллекции

Generic-комбинаторы над Массив(T) — компараторы передаются лямбдой, не встроенным <, поэтому отсортировать работает и на типах без реализация Сравниваемое:

импорт коллекции

функ старт() -> Число
    пер числа = массив(3, 1, 4, 1, 5)
    пер отсортированные = коллекции.отсортировать(числа, функ(a, b) a < b конец)
    пер сумма = коллекции.свернуть(числа, 0, функ(накопитель, x) накопитель + x конец)
    отсортированные[0] + сумма
конец
  • отфильтровать[T](массив, предикат: функ(T) -> Булево) -> Массив(T)
  • отобразить[T, U](массив, преобразование: функ(T) -> U) -> Массив(U)
  • свернуть[T, U](массив, начало: U, шаг: функ(U, T) -> U) -> U
  • отсортировать[T](массив, меньше: функ(T, T) -> Булево) -> Массив(T) — insertion sort, стабильная

флаги: разбор аргументов командной строки

импорт флаги

функ старт() -> Строка
    пер конфиг = флаги.новая_конфигурация()
    конфиг.добавить_флаг(флаги.Флаг("имя", флаги.ТипФлага.тСтрока))
    пер результат = конфиг.распарсить_флаги(массив("-имя=мир"))
    результат.получить("имя", флаги.ЗначениеФлага.фСтрока("")).вСтроку()
конец

Объявите флаги через Флаг(наименование, тип) (ТипФлага.тСтрока / .тЧисло / .тБулево), добавьте их в Конфигурация через .добавить_флаг, затем разберите либо явный массив строк (.распарсить_флаги(аргументы) — как выше, удобно для тестов), либо сразу ос.аргументы() через .распарсить() (требует импорт ос, см. «Использование CLI»). Поддержаны формы -флаг=значение и -флаг значение; для тБулево значение не читается — само присутствие флага значит истина. Необъявленный флаг или пропущенное значение — паника, не Результат (тот же принцип, что у кодирование/json/кодирование/toml ниже: ошибка во входных данных — критичный сбой).

кодирование/json, кодирование/toml

Оба модуля — тонкие парсеры в дерево Значение с одинаковым интерфейсом доступа. Импортируются по пути в кавычках (имя с /):

импорт "кодирование/json" как json

функ старт() -> Строка
    пер результат = json.разобрать("{\"имя\": \"panos\", \"версия\": 1}")
    если результат.ошибка() тогда
        возврат "ошибка парсинга"
    конец
    результат.значение().поле("имя").значение().как_строку().получить("?")
конец

разобрать(текст) -> Результат(Значение, Ошибка). У JSON дополнительно есть сериализовать(значение) -> Строка (обратное преобразование) — у TOML только парсинг, без записи.

Методы Значение — общие для обоих модулей:

  • .как_строку() / .как_число() / .как_булево() -> Опция(...)
  • .поле(ключ) -> Опция(Значение) — для объекта (JSON) / таблицы (TOML)
  • .элемент(индекс) -> Опция(Значение) — для списка

Варианты Значение отражают форму каждого формата: JSON — Ноль/Булево(Булево)/Число(Число)/Строка(Строка)/ Список(Массив(Значение))/Объект(Соответствие(Строка, Значение)); TOML — то же самое, но без Ноль (в TOML нет null) и Объект называется Таблица.

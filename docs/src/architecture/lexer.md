# Лексер

## Что

Точка входа — `tokenize :: proc(input: string, file_id: u16 = 0) -> ([dynamic]Token, [dynamic]Diagnostic)`
(`core/lexer.odin:527`) — создаёт `Lexer` через `new_lexer`, вызывает
`next_token_lex` до `EOF`, собирает токены и diagnostics.

**`Token`** (`core/token.odin:92`) — `data: string` (исходный текст),
`kind: TokenKind` (81 вариант — литералы/операторы/ключевые слова/EOF),
`span: Span`, `nl_before: bool` (была ли перед этим токеном пустая строка —
парсер использует это, чтобы отличить `Массив(Число)` — генерик на той же
строке — от `-> Число\n(выражение)` — тело функции, начинающееся со скобки
на СЛЕДУЮЩЕЙ строке), `doc: string` (текст непосредственно предшествующего
блока `///`-докстрингов, пусто если не было).

**`Span`** (`core/token.odin:86`) — `{file_id: u16, start: u32, end: u32}` —
байтовые смещения, НЕ line/column (конвертация через `span_line_col`,
`token.odin:130`).

**`Lexer`** (`core/lexer.odin:8`) — `input: string`, `pos: int` (текущая
байтовая позиция), `ch: rune` (текущий Unicode-символ), `width: int`
(байтовая ширина `ch`, 1-4 для UTF-8), `file_id: u16`, `diagnostics`,
`interp_paren_depth: [dynamic]int` (стек глубины скобок — отдельный
счётчик на каждый АКТИВНЫЙ уровень строковой интерполяции).

**`lookup_ident`** (`core/lexer.odin:219`) — таблица ключевых слов:
`switch ident { case "пер": return .Let; ... }` — 25+ русских ключевых слов.

## Зачем

Лексер существует отдельно от парсера, потому что решает задачи, которые
парсер не может решить на уровне синтаксиса:

- **Распознавание кириллических идентификаторов**: `unicode.is_alpha`
  корректно работает с кириллицей — парсер получает УЖЕ классифицированные
  токены (ключевое слово vs идентификатор), не сырую последовательность рун.
- **Обработка escape-последовательностей** (`\n`, `\t`, `\r`, `\"`, `\\`) —
  декодируются на этапе лексинга, парсер получает уже готовую строку.
- **Строковая интерполяция** (`"a \(f(x)) b"`) — многотокенный конечный
  автомат: лексер выдаёт `InterpStringStart("a ")`, обычные токены для
  `f(x)`, `InterpStringMid(" b")` — парсер сам НЕ знает про интерполяцию
  вообще (десахаривается парсером в `+`-конкатенацию, см. ниже).
- **Многосимвольные операторы** (`->`, `<=`, `<<` и т.п.) — лексер делает
  lookahead один раз, парсеру не нужно реконструировать их из отдельных
  символов.
- **Докстринги (`///`)** — должны быть накоплены и привязаны к СЛЕДУЮЩЕМУ
  непробельному токену — требует просмотра нескольких строк вперёд, парсер
  не может задним числом привязать документацию к уже классифицированному
  токену.

## Почему так, а не иначе

**`read_number` поглощает `.` ТОЛЬКО если сразу за ним идёт цифра**
(`core/lexer.odin:128-140`, исправлено в коммите `5300091`):

```odin
read_number :: proc(l: ^Lexer) -> string {
	start := l.pos - l.width
	for unicode.is_digit(l.ch) {
		advance(l)
	}
	if l.ch == '.' && unicode.is_digit(peek_char(l)) {
		advance(l)
		for unicode.is_digit(l.ch) {
			advance(l)
		}
	}
	return l.input[start:l.pos - l.width]
}
```

Раньше условие было безусловным (`l.ch == '.'` без проверки следующей
цифры) — `t.1.длина()` (tuple-индекс 1, затем вызов метода) читался бы
как ОДИН числовой токен `"1."`, поглощая `.` перед `длина` — вместо
отдельного `Dot`-токена для парсера. Проверка lookahead предотвращает
ложное распознавание десятичной точки.

**Известное оставленное ограничение** (комментарий к тому же коммиту):
`вложенный.1.0` (tuple-индекс tuple-индекса, где ВТОРОЙ индекс тоже голая
цифра) лексер всё ещё не может разобрать САМ — `"1.0"` байт-в-байт
неотличимо от float-литерала на уровне лексера с однобайтовым lookahead.
Разрешается на уровне ПАРСЕРА (см. [парсер](./parser.md) — обработка
`.Dot` в property-доступе, где уже есть контекст: если числовой токен
после `.` сам содержит `.`, парсер разбивает его на два `Property_Expr`).

**Интерполяция строк парсится через стек `interp_paren_depth`**
(комментарий, `lexer.odin:17-24`): вложенная интерполяция
(`\(f(\(x)))`) толкает свой элемент стека на каждый активный уровень;
когда встречена `)` и вершина стека — 0, это НЕ скобка вложенного
выражения, а закрывающая для самого `\(` — лексер возобновляет
сканирование строкового фрагмента вместо обычного `.RParen`.

**Byte offset, не rune-aware** (`token.odin:127-128`): для кириллицы
колонка (в байтах) не совпадает с визуальной позицией (multi-byte UTF-8),
но НОМЕР СТРОКИ всегда точен — этого достаточно для перехода к строке в
редакторе (LSP), полная rune-точность колонки не нужна.

**Лексер не паникует на неожиданном символе** (`report_lex` + `advance`,
БЕЗ токена для символа) — accumulate-not-panic, тот же паттерн, что в
парсере/резолвере/тайпчекере: ошибочный символ не должен ронять весь
процесс, особенно важно для LSP при live-typing (незавершённый/
синтаксически неверный код при каждом нажатии клавиши).

## Точки входа для типичной правки

| Изменение | Файл/функция |
|---|---|
| Новое ключевое слово | `lookup_ident` (`core/lexer.odin:219`) — добавить `case "слово": return .НовыйKind`; добавить вариант в `TokenKind` (`core/token.odin`) |
| Новый оператор/разделитель | `next_token_lex` (`core/lexer.odin`, свитч по `l.ch`) — добавить `case`, использовать `peek_char` для многосимвольных |
| Новая escape-последовательность в строках | обработка `\`-последовательностей внутри чтения строкового фрагмента (`core/lexer.odin`) — добавить `case` в свитч после `\` |
| Изменить правила чисел (например добавить hex/octal) | `read_number` (`core/lexer.odin:128`) |
| Изменить диагностику лексера | `report_lex` — дедуплицирует по (span, message), не паникует |

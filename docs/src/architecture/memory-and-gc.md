# Память и сборщик мусора

## Что

Mark-and-sweep, non-moving (не двигает объекты в памяти) сборщик мусора.

**`GC_Header`** (`core/gc.odin:46`) — `{marked: bool}`. Встроен ПЕРВЫМ полем
в каждый heap-управляемый вариант `Value` (см.
[компилятор и VM](./compiler-and-vm.md)) — единственный способ, которым
сборщик находит mark-бит для ЛЮБОГО heap-объекта без отдельной таблицы
метаданных.

**`GC_State`** (`core/gc.odin:61`) — `all_objects: [dynamic]Value`
(вообще все живые аллокации), `protect_stack: [dynamic]Value` (временная
защита значений — см. ниже), `bytes_allocated`/`next_threshold` (авто-
триггер), плюс отдельный free-list НА КАЖДЫЙ heap-тип
(`free_aggregates`, `free_arrays`, `free_strings` и т.д. — sweep не
вызывает `free()`, а складывает объект в пул для переиспользования).

**Триггер сборки** — двойной: (1) автоматический — `gc_new` проверяет
`bytes_allocated > next_threshold` (`GC_MIN_THRESHOLD = 1024*1024` байт,
`core/gc.odin:103`); (2) явный — `force_gc(vm)` (публичный хук для
детерминированных тестов "было N объектов, стало M").

**`collect_garbage`** (`core/gc.odin:511`) — `mark_roots(vm)` (пометить
достижимое от корней), `sweep(vm)` (уплотнить `all_objects`, отправить
непомеченное в пул), пересчитать `next_threshold = max(bytes_allocated*2, GC_MIN_THRESHOLD)`.

**Корни сборки** (`mark_roots`, `core/gc.odin:336`): `vm.stack`,
`vm.gc.protect_stack`, константы всех `Compiled_Function` в реестре,
почтовые ящики (`mailbox`) и очереди сигналов ВСЕХ процессов, стеки
процессов, ОТЛИЧНЫХ от текущего (стек текущего процесса — это `vm.stack`,
уже обойдён).

## Зачем

`GC_Header`, встроенный первым полем в каждый heap-тип, даёт сборщику
ЕДИНЫЙ способ найти mark-бит любого объекта простым приведением указателя
— без параллельной таблицы "адрес → метаданные", которую пришлось бы
поддерживать синхронно с любой аллокацией/освобождением.

## Почему так, а не иначе

**Non-moving, а не copying/compacting GC** (`core/gc.odin:42-45`,
комментарий): по всему `vm.odin`/`compiler.odin` разбросаны СЫРЫЕ
указатели на `Value`-объекты — перемещающий сборщик потребовал бы
исправления ВСЕХ этих ссылок при каждой сборке. Признано избыточным для
текущего масштаба проекта.

**Sweep не вызывает `free()` — пулит объекты по типу** (`core/gc.odin:76-99`):
даже при корректном mark/sweep RSS (resident memory) процесса растёт под
устойчивой нагрузкой ("миллион аллокаций в цикле"), потому что "логически
освобождённые" байты не гарантируют, что ОС быстро вернёт память под
СЛЕДУЮЩУЮ аллокацию того же размера. Решение — типизированные free-list'ы:
следующий `gc_new` того же типа переиспользует объект из пула вместо
`malloc`. **Требование к вызывающему коду**: переиспользовать `resize(&obj.elements, n)`,
а НЕ `obj.elements = make(...)` — второе убивает переиспользуемый backing
buffer пула.

**`gc_protect`/`gc_unprotect`** (`core/gc.odin:128`) — временно
"заякоривают" значение, которое СНЯТО со стека, но ещё НЕ встроено в новый
объект (или новый объект ещё не положен обратно на стек). В этом окне
значение невидимо для mark-фазы (не на `vm.stack`, не в поле нового
объекта) — без `gc_protect` sweep мог бы освободить его микросекундами
позже. Нужно КАЖДЫЙ раз, когда `gc_new` вызывается ДО того, как предыдущий
промежуточный результат попал обратно на стек (например построение
`Error_Value` с двумя `gc_new_string` внутри).

**Обработка циклов — бесплатно, в отличие от reference counting**: `mark_value`
сам по себе cycle guard — уже помеченный объект не обходится повторно
(`if h.marked do return`), поэтому циклические структуры (self-referential
`Aggregate_Value` и т.п.) не зацикливают mark-фазу. Недостижимые циклы
собираются нормально — reference counting не мог бы собрать их без
отдельного cycle collector.

**Два разных аллокатора нативно vs wasm** (`core/gc.odin:12-40`):
`vm_heap_allocator()` — `runtime.heap_allocator()` (настоящий malloc/free
ОС) нативно, но НЕ РЕАЛИЗОВАН на `js_wasm32` (падает `unimplemented()`
panic'ом при первом вызове) — там `runtime.default_wasm_allocator()`.
`vm_gc_allocator()` — на нативной платформе тот же настоящий heap
(переиспользование пула между запусками VM реально возвращает память ОС);
на wasm — `context.allocator` (растущая арена, сбрасываемая целиком между
вызовами `panos_run`) — там `pool_release` физически не освобождает
(`Dynamic_Arena.Free` не реализован), поэтому единственный способ не течь
между запусками — `reset()` всей арены разом.

## Точки входа для типичной правки

Новый heap-управляемый вариант `Value` — трогать ВСЕ места сразу (иначе
компилятор Odin укажет на непокрытый exhaustive switch, кроме
`get_header`/`value_size`/`pool_release`, которые НЕ exhaustive по
умолчанию — легко забыть):

| Шаг | Файл/функция |
|---|---|
| 1. Struct с `GC_Header` первым полем | `core/compiler.odin` (рядом с остальными `*_Value`-структурами) |
| 2. Добавить вариант в `Value` union | `core/compiler.odin` |
| 3. Free-list в `GC_State` + инициализация | `core/gc.odin:61` (`GC_State`), `new_gc_state` |
| 4. Диспетч в `gc_new` | `core/gc.odin:158` |
| 5. Case в `get_header` | `core/gc.odin:228` — возвращает `&val.header` |
| 6. Case в `mark_value` | `core/gc.odin:270` — рекурсивно пометить вложенные `Value`-поля |
| 7. Case в `value_size` | учёт байт при `bytes_allocated` |
| 8. Case в `pool_release` | очистка/финализация (закрыть файл/сокет, если внешний ресурс) + `append` в свой free-list |

# Platform-split (native/wasm)

## Что

Семь пар файлов в `core/`, каждая пара реализует ОДНУ и ту же функциональность
двумя разными способами — нативно (`#+build !js`) и под wasm (`#+build js`):

| Нативная часть | WASM-часть | Что реализует |
|---|---|---|
| `file_value_native.odin` | `file_value_wasm.odin` | Файловые значения (`File_Value`) |
| `module_loader_native.odin` | `module_loader_wasm.odin` | Чтение файла модуля с диска |
| `resolver_import_native.odin` | `resolver_import_wasm.odin` | Разрешение пути импорта (`resolve_existing_import_path`) |
| `vm_compress_native.odin` | `vm_compress_wasm.odin` | Архивация/сжатие (tar/gz) |
| `vm_ffi_native.odin` | `vm_ffi_wasm.odin` | FFI (вызов внешних C-функций через libffi) |
| `vm_http_native.odin` | `vm_http_wasm.odin` | HTTP(S)-клиент |
| `vm_io_native.odin` | `vm_io_wasm.odin` | Файловый ввод-вывод |

Точка входа wasm-сборки — `wasm/main.odin`, экспортирует `panos_run "c"`
(и `panos_check`/`panos_hover`/`panos_complete` для интерактивного демо —
см. `docs/src/assets/interactive.js`).

## Зачем

Разделение существует, потому что часть стандартной библиотеки Odin
(`core:os`, файловая система, реальные сокеты) **недоступна на цели
`js_wasm32` НА ЭТАПЕ КОМПИЛЯЦИИ**, не как runtime-ошибка — импорт `core:os`
под `js_wasm32` падает compile-time panic'ом. Единый файл с рантайм-веткой
`if is_wasm { ... } else { ... }` физически не собрался бы под wasm, если
бы содержал хоть один импорт `core:os` где-либо в теле функции —
компилятор Odin проверяет ВСЕ импорты модуля независимо от того,
достижима ли конкретная ветка кода в рантайме.

## Почему так, а не иначе

**`#+build !js`/`#+build js` — раздельные ФАЙЛЫ, а не `when ODIN_OS` внутри
одного файла**: build-теги Odin работают на уровне файла компиляции —
файл с `#+build js` просто не участвует в сборке под нативную цель, и
наоборот. Это единственный способ гарантировать, что `core:os` НИКОГДА не
попадёт в wasm-сборку (в отличие от `when`-условия, которое всё равно
требует, чтобы ОБЕ ветки скомпилировались синтаксически на всех целях).

**Модульная система (см. [модульная система](./module-system.md)) сама
разделена этой же парой** (`resolver_import_native.odin`/`_wasm.odin`) —
разрешение пути импорта на нативной платформе читает переменные окружения
и файловую систему (`os.exists`), на wasm — общий контракт "нет
файловой системы" распространяется и сюда: WASM-вариант не имеет ФС и
ВСЕГДА отвечает "не найдено" (тот же контракт, что и для отсутствующего
файла на диске у нативной версии).

## Точки входа для типичной правки

| Изменение | Файл/функция |
|---|---|
| Изменить поведение файлового I/O | ОБЕ части пары: `vm_io_native.odin` И `vm_io_wasm.odin` — забыть wasm-часть означает молчаливое расхождение поведения между нативной сборкой и демо в браузере |
| Добавить новую platform-specific возможность | новая пара файлов `имя_native.odin`/`имя_wasm.odin` с соответствующими `#+build`-тегами — по образцу существующих 7 пар |
| Изменить точку входа wasm-демо | `wasm/main.odin` — `panos_run`/`panos_check`/`panos_hover`/`panos_complete` |
| Проверить, что новый код не тянет `core:os` под wasm | `just build-wasm` (см. [тулчейн и тестирование](./toolchain-and-testing.md)) — единственный надёжный способ проверить: сборка просто упадёт compile-time, если platform-контракт нарушен |

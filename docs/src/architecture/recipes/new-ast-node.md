# Рецепт: новый вид AST-узла (Expr/Stmt)

Пример: добавить новое выражение (аналогично тому, как устроены уже
существующие `Try_Expr`/`Spawn_Expr`).

## Файлы по порядку (pipeline order)

1. `core/parser.odin` — struct декларация + вариант union + функция парсинга + `expr_span`
2. `core/resolver.odin` — резолв внутренних под-выражений/имён
3. `core/type_cheker.odin` — вывод типа
4. `core/compiler.odin` — кодогенерация (может переиспользовать существующие opcode, не обязательно заводить новый)
5. `core/ast_clone.odin` — клонирование для мономорфизации (только если новый узел может встретиться внутри generic-функции)

## Шаги

1. **`core/parser.odin`**: описать struct нового узла (обязательное поле
   `span: Span`, остальные — под конкретную семантику). Добавить вариант
   `^Новый_Expr` в union `Expr` (см. [парсер](../parser.md) — `Expr` union,
   `parser.odin:571`). Написать `parse_новый_expr` (или встроить в
   существующий `nud`, если это префиксная форма). Добавить case в
   `expr_span` (`parser.odin:2746`) — БЕЗ этого AST-узел не сможет
   участвовать в LSP/диагностиках, которые вычисляют span выражения.
   Если нужен отладочный вывод — добавить case в `print_ast`.

2. **`core/resolver.odin`**: если новый узел содержит под-выражения с
   именами (например `Ident_Expr` внутри) — добавить case в функцию обхода
   резолвера (там же, где `#partial switch` по `Expr` для `Binary_Expr`/
   `Call_Expr`/и т.п.), рекурсивно резолвящий вложенные под-выражения.

3. **`core/type_cheker.odin`**: добавить case `^Новый_Expr` в switch внутри
   `infer_expr` (`type_cheker.odin:5836`, см.
   [тайпчекер](../type-checker.md)), написать `infer_новый_expr`,
   вернуть выведенный `^Type` (или `report(...)` при ошибке — см.
   `report()`, `type_cheker.odin:1044`).

4. **`core/compiler.odin`**: добавить case `^Новый_Expr` в `compile_expr`
   (там же, где `case ^Binary_Expr:`, `compiler.odin:907`) — либо
   переиспользовать существующие opcode (если семантика сводится к уже
   существующим операциям), либо завести новый (см.
   [рецепт нового бинарного оператора](./new-binary-operator.md) для
   процедуры добавления opcode).

5. **`core/ast_clone.odin`** (только если применимо): если новый узел может
   встретиться внутри bounded generic-функции — добавить case клонирования
   в `clone_expr` (см. [дженерики](../generics-and-monomorphization.md) —
   любое новое поле AST-узла, которое участвует в мономорфизации, ДОЛЖНО
   быть скопировано здесь, иначе клон потеряет его).

## Проверка

- `odin test ./core` — существующие e2e-тесты не должны сломаться.
- Написать новый e2e-тест через `run_code` (`core/e2e_test.odin`, см.
  [тулчейн и тестирование](../toolchain-and-testing.md)), конкретно
  упражняющий новый узел — резолв + тайпчек + компиляция + исполнение.
- `just debug-file <path>` на `.ps`-файле, использующем новую конструкцию —
  проверить diagnostic-вывод при некорректном использовании (например
  несовместимый тип внутри нового узла).

# Рецепт: новый бинарный оператор

Пример: гипотетический оператор возведения в степень `**`. Затрагивает
ВСЕ стадии pipeline — новый бинарный оператор не бывает "мелкой" правкой.

## Файлы по порядку (pipeline order)

1. `core/token.odin` — новый `TokenKind` (только если оператор не собирается из уже существующих символов)
2. `core/lexer.odin` — токенизация символа/сочетания (только если новый, см. выше)
3. `core/parser.odin` — precedence (`infix_bp`) + диспетчеризация автоматическая
4. `core/type_cheker.odin` — вывод типа результата (`infer_binary_expr`)
5. `core/compiler.odin` — новый `Opcode` + эмиссия
6. `core/vm.odin` — обработчик opcode в главном цикле

## Шаги

1. **Если оператор — новый символ** (не комбинация уже лексируемых
   символов): добавить вариант в `TokenKind` (`core/token.odin`), добавить
   `case` в `next_token_lex` (`core/lexer.odin`) — см.
   [лексер](../lexer.md) → "Точки входа для типичной правки". Если
   оператор собирается из уже существующих токенов — этот шаг не нужен.

2. **Добавить precedence** в `infix_bp` (`core/parser.odin:2561`, см.
   [парсер](../parser.md)): выбрать пару `(lbp, rbp)` согласно таблице
   существующих ярусов — **не менять существующие значения**, разрывы
   между ярусами оставлены щедрыми специально для таких вставок. Право-
   ассоциативные операторы (как `**` обычно) — `rbp < lbp`. Диспетчеризация
   в `parse_expr` автоматическая — отдельная функция парсинга не нужна,
   `new_bin_op` создаст `Binary_Expr` с новым `TokenKind` в поле `op`.

3. **Тайпчек**: `infer_binary_expr` (`core/type_cheker.odin:4170`, см.
   [тайпчекер](../type-checker.md)) — добавить `case` в
   `#partial switch e.op`, вывести/проверить типы операндов через
   `prune_type(infer_expr(ctx, e.left/e.right))`, решить итоговый тип
   (например `Целое`/`Число` по аналогии с существующими арифметическими
   операторами), при несовместимости — `report(ctx, e.span, ...)`.

4. **Объявить opcode**: `Opcode :: enum u8` (`core/compiler.odin:284`, см.
   [компилятор и VM](../compiler-and-vm.md)) — новый вариант, например
   `Power`.

5. **Эмиссия компилятора**: `compile_expr`, кейс `^Binary_Expr`
   (`core/compiler.odin:907`, внутренний `#partial switch e.op` начиная
   со строки ~984) — `case .НовыйToken: emit_opcode(ctx, .Power)`. Если
   выбор opcode зависит от статического типа (как `Int_Divide`/`Divide`) —
   проверка `ctx.tc.node_types[e.left] == TY_INT`.

6. **Обработчик VM**: `execute`, `#partial switch opcode`
   (`core/vm.odin:1485`+) — новый `case .Power:`, снять операнды со
   `vm.stack` через `pop`, вычислить, положить результат через `append`.

## Проверка

- `odin test ./core` — весь набор тестов.
- Добавить e2e-тест через `run_code` (`core/e2e_test.odin`, см.
  [тулчейн и тестирование](../toolchain-and-testing.md)) — конкретное
  выражение с новым оператором, проверка результата.
- `just debug-file <path>` на `.ps`-файле с новым оператором — проверить,
  что выражение компилируется и печатает ожидаемый результат.
- Если оператор поддерживает и `Целое`, и `Число` — тест на ОБА случая
  (аналогично `Int_Divide`/`Divide` — статический выбор opcode компилятором,
  не рантайм-ветвление в VM).

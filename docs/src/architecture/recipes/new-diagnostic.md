# Рецепт: новая diagnostic-проверка

Пример: запретить конкретную конструкцию на этапе резолва ИЛИ типизации
(не изменение существующего поведения — именно НОВАЯ проверка,
репортящая diagnostic).

## Файлы по порядку (pipeline order)

1. `core/resolver.odin` ИЛИ `core/type_cheker.odin` — в зависимости от
   того, на каком этапе доступна нужная информация
2. (опционально) e2e-тест диагностики

## Шаги

1. **Решить, на каком этапе проверка возможна**: если условие проверяется
   ТОЛЬКО по именам/scope (без знания типов) — резолвер
   ([резолвер](../resolver.md)); если условие требует знания типа
   выражения — тайпчекер ([тайпчекер](../type-checker.md)). Резолвер
   выполняется раньше тайпчекера — проверка, требующая типов, физически не
   может быть в резолвере (типы ещё не выведены).

2. **В резолвере**: найти подходящее место обхода (`resolve_expr`/
   `resolve_stmt`/место резолва конкретной декларации), добавить условие,
   вызвать `report_resolve(ctx, span, format, ..args)`
   (`core/resolver.odin:489`) — repортит `Diagnostic` в `ctx.diagnostics`,
   НЕ паникует (accumulate-not-panic — резолв продолжается после ошибки,
   собирая максимум диагностик за один проход).

3. **В тайпчекере**: найти подходящую `infer_*`/`check_*`-функцию (см.
   диспетчеризацию в `infer_expr`, `type_cheker.odin:5836`), добавить
   условие, вызвать `report(ctx, span, format, ..args)`
   (`type_cheker.odin:1044`) — форматирует сообщение, ДЕДУПЛИЦИРУЕТ (не
   повторяет тот же diagnostic на том же span), добавляет в
   `ctx.diagnostics`, возвращает `TY_POISON` — используйте результат
   `report(...)` НАПРЯМУЮ как возвращаемое значение функции, чтобы ошибка
   не каскадировала в производные диагностики (см.
   [известные грабли](../known-pitfalls.md) про error recovery паттерны).

4. **Формулировка сообщения**: diagnostics в этом проекте — на русском
   языке, формат `"Resolve Error: ..."` (резолвер) или `"Type Error: ..."`
   (тайпчекер) — сохранять этот префикс для консистентности с остальными
   diagnostics.

## Проверка

- `odin test ./core` — убедиться, что НИ ОДИН существующий тест не начал
  падать (новая diagnostic-проверка не должна ловить уже валидный код).
- Добавить e2e-тест через `check_source`/`run_code` (`core/pipeline.odin`,
  см. [тулчейн и тестирование](../toolchain-and-testing.md)): подать код,
  нарушающий новое правило, убедиться что `diags` (или
  `res_ctx.diagnostics`/`tc_ctx.diagnostics`) содержит ожидаемое сообщение.
- `just debug-file <path>` на файле с нарушением — убедиться, что
  diagnostic реально печатается пользователю (не только виден в тесте).

# Рецепт: новая stdlib-функция в существующем модуле

Пример: добавить функцию в уже существующий файловый модуль
`std/математика.ps`/`std/коллекции.ps`/`std/архив.ps`/`std/тест.ps`/
`std/флаги.ps`/`std/супервизор.ps` (не встроенный native-модуль вроде
`строки`/`фс` — см. [модульная система](../module-system.md) про разницу).

## Файлы по порядку (pipeline order)

1. `std/<модуль>.ps` — сама функция, обычный panos-код
2. (опционально) `docs/src/language/*.md` — пользовательская документация языка, если функция достаточно важна для явного упоминания

## Шаги

1. **Написать функцию как обычный panos-код** в существующем `std/*.ps`-файле.
   Экспортируемая функция начинается с `экспорт`:
   ```panos
   экспорт функ имя_функции(арг: Тип) -> ВозвратТип
       ...
   конец
   ```
   Без `экспорт` функция остаётся приватной для модуля (доступна только
   другим функциям того же файла).

2. **Докстринг** (`///`) непосредственно над декларацией — попадает в
   `Function_Decl.doc`, показывается в LSP hover (см.
   [парсер](../parser.md) про `Token.doc`). Соглашение проекта: все
   экспортируемые функции документируются.

3. **НИКАКИХ изменений в `core/` не требуется** — файловые stdlib-модули
   компилируются обычным пайплайном (лексер→парсер→резолвер→тайпчекер→
   компилятор), как любой другой panos-исходник. Разрешение пути импорта
   уже работает: `std/<модуль>.ps` резолвится через `resolve_import_path`
   (см. [модульная система](../module-system.md)) — **важно**: файловый
   путь резолвится РАНЬШЕ builtin-фолбэка (комментарий в
   `std/математика.ps`), так что имя модуля не должно совпадать с
   зарезервированным native-именем (`строки`/`фс`/`ос`/`ввод_вывод`/`сеть`).

## Проверка

- `just debug-file <путь>.ps` на тестовом `.ps`-файле, импортирующем модуль
  и вызывающем новую функцию.
- Если у модуля есть e2e-тесты в `core/e2e_modules_stdlib_test.odin` —
  добавить туда сценарий через `run_code` (см.
  [тулчейн и тестирование](../toolchain-and-testing.md)).
- `odin test ./core` — убедиться, что ничего не сломалось.

# Рецепт: новый stdlib-модуль целиком

Пример: добавить новый файл `std/имя.ps` (по образцу существующих
`std/математика.ps`, `std/коллекции.ps`, `std/архив.ps`, `std/тест.ps`,
`std/флаги.ps`, `std/супервизор.ps`) — не встроенный native-модуль (для
того нужны изменения в `core/stdlib.odin`, см.
[модульная система](../module-system.md)).

## Файлы по порядку (pipeline order)

1. `std/имя.ps` — новый файл
2. `docs/src/language/standard-library.md` — упоминание нового модуля в пользовательской документации языка
3. (опционально) `core/e2e_modules_stdlib_test.odin` — e2e-тест

## Шаги

1. **Выбрать имя модуля, НЕ совпадающее с зарезервированными native-именами**
   (`строки`/`фс`/`ос`/`ввод_вывод`/`сеть`) — файловый путь резолвится
   РАНЬШЕ builtin-фолбэка (см. [модульная система](../module-system.md)),
   так что технически конфликт не сломает резолв, но затенит доступ к
   native-модулю с тем же именем — избегать намеренно.

2. **Создать `std/имя.ps`** — обычный panos-исходник. Экспортируемые
   функции/типы — через `экспорт`:
   ```panos
   // Докстринг модуля (обычный комментарий, не /// — модули как целое
   // докстрингов не имеют, только отдельные декларации внутри).
   экспорт функ функция(арг: Тип) -> ВозвратТип
       ...
   конец

   экспорт тип Тип = структура
       поле: Число
   конец
   ```

3. **Никакой явной "регистрации" модуля не требуется** — файловые модули
   не имеют реестра (в отличие от native builtin-модулей, см.
   `ensure_builtin_module`, `core/stdlib.odin`). Первый `импорт имя` из
   любого `.ps`-файла триггерит `load_module_recursive`
   (`core/module_loader.odin`), которая находит `std/имя.ps` через
   `resolve_existing_import_path` и парсит/резолвит/типизирует его как
   обычный модуль.

4. **Добавить упоминание в пользовательскую документацию** —
   `docs/src/language/standard-library.md` (раздел "Язык" mdBook, см.
   [обзор раздела](../overview.md) про разницу между этим разделом и
   разделом "Язык") — новый модуль должен быть открываемым пользователями
   языка, не только мейнтейнерами.

## Проверка

- `just debug-file <путь>.ps` на тестовом файле с `импорт имя`.
- `odin test ./core` — полный набор тестов.
- Если модуль предназначен для широкого использования — добавить сценарий
  в `core/e2e_modules_stdlib_test.odin` (через `run_code`, см.
  [тулчейн и тестирование](../toolchain-and-testing.md)).

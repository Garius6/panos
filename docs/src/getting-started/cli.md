# Использование CLI

## Запуск файла

```sh
panos файл.ps [аргументы...]
```

Всё после имени файла — аргументы самой программы, не флаги `panos`.
Прочитать их внутри скрипта:

```panos-cli
импорт ос

функ старт() -> Строка
    пер аргументы = ос.аргументы()
    если длина(аргументы) == 0 тогда
        "нет аргументов"
    иначе
        аргументы[0]
    конец
конец
```

```sh
panos файл.ps привет мир
# -> "привет" (ос.аргументы() == ["привет", "мир"])
```

Без аргумента-файла `panos` открывает REPL.

## `-v` / `--verbose`

Флаг обязан стоять **до** имени файла (после — уже аргумент программы):

```sh
panos -v файл.ps
```

Печатает четыре секции по порядку — AST, результат типизации, ассемблерный
дамп скомпилированного байткода, и уже затем обычный вывод исполнения.
Полезно, чтобы посмотреть, во что разворачивается конкретная конструкция
языка, или разобраться в диагностике компилятора; для обычного запуска не
нужен.

## Поиск модулей

`импорт имя` (не путь в кавычках) ищет файл `имя.ps` по очереди в четырёх
местах, использует первое найденное:

1. рядом с импортирующим файлом (тот же каталог)
2. в подкаталоге `модули/` рядом с импортирующим файлом
3. в каталоге, на который указывает переменная окружения `PANOS_STDLIB`
4. в каталоге `std/` — там же, где сам бинарник `panos` (пакетная
   стандартная библиотека)

Если имя совпадает со встроенным нативным модулем (`ввод_вывод`, `фс`,
`ос`, `сеть`, `строки`) и файл нигде не нашёлся — используется он.
Практическое следствие: положить файл с таким же именем в свой проект
можно, но он молча перекроет нативный модуль, если найдётся раньше него в
этом порядке поиска.

Импорт вложенного стандартного модуля (`std/сеть/http.ps`,
`std/кодирование/json.ps`) требует пути в кавычках, потому что имя с `/`
не проходит как обычный идентификатор:

```panos-cli
импорт "сеть/http" как http
импорт "кодирование/json" как json
```

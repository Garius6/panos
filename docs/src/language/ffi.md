# Внешние функции (FFI)

> `внешний`-декларации требуют `core:dynlib`-загрузку системной библиотеки
> в рантайме — **недоступны** в браузерной песочнице (тот же принцип, что
> `фс`/`ос`/`сеть` — см. [«Модули» → ограничение в
> браузере](./modules.md#ограничение-в-браузере)). Примеры не запускаются
> на странице, запускайте локально: `./panos файл.ps`.

`внешний` — настоящий *динамический* FFI: декларация грузит библиотеку и
ищет символ **в рантайме**, по имени, прочитанному из `.ps`-исходника.
Механизм — `core:dynlib` (найти библиотеку/функцию) + [libffi](https://sourceware.org/libffi/)
(вызвать найденный указатель с сигнатурой, известной только в рантайме) —
статически слинкованный вендоренный `libffi.a` (см. `external/libffi/
README.md`), не системная зависимость.

Это ЕДИНСТВЕННЫЙ способ звать внешние C-библиотеки в panos: раньше
существовал core-builtin модуль `графика` (статические Odin-биндинги
поверх `vendor:raylib`, вкомпилированные в сам `panos`) — он полностью
удалён, panos core теперь вообще не знает ни о raylib, ни о каком-либо
другом внешнем API. Эквивалент — обычная panos-библиотека
[`demos/raylib.ps`](https://github.com/Garius6/panos/blob/main/demos/raylib.ps),
написанная поверх `внешний`/`ff_структура` (см. ниже), которую может
скопировать/импортировать любой проект — без единой строчки в core.

Поддержаны примитивные целые (`Целое(N)`), числа с плавающей точкой
(`Число(N)`), строки (`КСтрока`), опаковые указатели (`Указатель(T)`) и
структуры по значению (`ff_структура`) — этого достаточно для большей
части реальных C-библиотек, включая raylib. **Явно не сделано** (см.
ROADMAP §Стадия 8 на оставшийся объём):

- Callback'и (передача panos-функции как указателя в C, `ffi_closure`).
- Восстановление после SIGSEGV в чужом коде.
- Вложенные `ff_структура` (структуры внутри структур), строки/указатели
  как поля `ff_структура` — только плоский layout из `Целое(8|32|64)`/
  `Число(32|64)`.
- Платформы, отличные от `darwin-arm64` (см. `external/libffi/README.md` —
  честно задокументированный, не скрытый пробел).

## Объявление

```panos-cli
внешний "libc" функ getpid() -> Целое(32)
внешний "libc" функ abs(x: Целое(32)) -> Целое(32)

функ старт() -> Целое
    ввод_вывод.печать(getpid())
    abs(0 - 5)
конец
```

`внешний "имя_библиотеки" функ имя(параметры) -> Целое(N)` — верхнеуровневая
декларация (наравне с `функ`/`структура`/`импорт`), без тела. `"libc"` —
логическое имя библиотеки, БЕЗ платформенного расширения/пути (резолвер сам
достраивает `libc.dylib`/`libc.so`/`libc.dll` под текущую ОС). Резолвится
через `dynlib.load_library` + `dynlib.symbol_address` **на этапе резолва**,
до typecheck/компиляции — несуществующая библиотека или несуществующий
символ даёт понятную `Resolve Error`, а не крэш в рантайме:

```
Resolve Error: библиотека 'libc' не найдена (libc.dylib)
Resolve Error: библиотека 'libc' не экспортирует символ 'xyz'
```

Объявленная функция вызывается как **любая обычная** функция —
`getpid()`, `abs(x)` — никакого нового синтаксиса на стороне вызова.
`внешний` нельзя экспортировать (`экспорт внешний ...` — синтаксическая
ошибка): у неё нет "владеющего" модуля в обычном смысле, каждый файл,
которому нужна `внешний`-функция, объявляет её сам.

## `Целое(N)` / `Число(N)` — только маршаллинг, не новый тип

`Целое(8|32|64)` — синтаксис, валидный **только** внутри
`внешний`-сигнатуры (параметры, возврат, поля `ff_структура`). Число в
скобках — ширина C ABI типа (`uint8_t`/`int32_t`/`int64_t`), которую
использует libffi при упаковке/распаковке — **`Целое(8)` маршаллится как
беззнаковый `unsigned char`** (0..255), тот же тип, что C-библиотеки
обычно используют для байтовых полей (например, компоненты `Color`).
Для остального языка (арифметика, унификация типов, другие вызовы)
параметр/возврат — обычный `Целое`, БЕЗ ширины: `Целое(32)` — не
отдельный тип, не `Целое`-с-тегом, ширина нигде не хранится за пределами
самой `внешний`-декларации/`ff_структура`.

`Число(32)`/`Число(64)` — тот же принцип для чисел с плавающей точкой
(`float`/`double`) — на panos-стороне обычный `Число` (всегда 64-битный
IEEE 754 double), ширина только определяет, как значение упаковывается
для конкретного C ABI.

```panos-cli
внешний "libc" функ abs(x: Целое(32)) -> Целое(32)

функ квадрат_модуля(n: Целое) -> Целое
    пер m = abs(n) // n — обычный Целое, результат тоже
    m * m
конец
```

## `КСтрока` — строки

`КСтрока` — тоже синтаксис, валидный только внутри `внешний`-сигнатуры
(как `Целое(N)`), но маршаллит `Строка` в/из C `char*`:

```panos-cli
внешний "libc" функ strlen(s: КСтрока) -> Целое(64)
внешний "libc" функ strdup(s: КСтрока) -> КСтрока

функ старт() -> Строка
    ввод_вывод.печать(strlen("hello")) // 5
    strdup("привет") // копия через C и обратно, без искажений UTF-8
конец
```

Параметр: панос `Строка` копируется в null-terminated C-буфер **на время
самого вызова** (borrowed convention) — если C-функция сохраняет
указатель для использования ПОСЛЕ своего возврата, это вне гарантий
текущего среза. Возврат: `char*` от C ВСЕГДА копируется в новую panos
`Строка` — panos никогда не заимствует чужую C-память для строк,
независимо от того, кто ей на самом деле владеет.

## `Указатель(T)` — опаковые указатели

`Указатель(T)` — единственная из трёх FFI-форм типа, которая является
**настоящим panos-типом** (не только синтаксисом внутри `внешний`): его
можно хранить в переменной, передавать между функциями. `T` — чисто
фантомный параметр (даёт типобезопасность — `Указатель(Файл)` и
`Указатель(Сокет)` не взаимозаменяемы для тайпчекера), сам указатель
никогда не разыменовывается panos-стороной в этом срезе — нет ни
чтения полей, ни арифметики над адресом, только opaque handle (годится
для `FILE*`-подобных API — открыл/передал обратно/закрыл, без прямого
доступа к содержимому).

```panos-cli
внешний "libc" функ malloc(размер: Целое(64)) -> Указатель(Целое) свой
внешний "libc" функ getenv(имя: КСтрока) -> Указатель(Целое)
```

Возвращаемый `Указатель(T)` принимает опциональный постфикс владения:

- **`чужой`** (default, если суффикс не указан) — panos никогда не
  освобождает эту память. Безопасный выбор для указателей, которыми
  управляет сама C-библиотека (как `getenv` выше — эта память живёт в
  окружении процесса, её нельзя освобождать вручную).
- **`свой`** — panos берёт память под своё управление: когда
  `Указатель(T)` становится недостижим, GC освобождает её через libc
  `free()` (тот же finalizer-паттерн, что у `Файл`/`Сокет` — момент
  освобождения непредсказуем, это fallback, а не замена явному
  управлению памятью).

Постфикс владения указывается только на ВОЗВРАТЕ — у параметров вопрос
владения не возникает (panos просто передаёт уже имеющийся у него
указатель, ничего не приобретая).

## `ff_структура` — структуры по значению (C ABI)

Некоторые C-функции принимают/возвращают маленькие структуры **по
значению**, не по указателю (raylib's `Vector2Add(Vector2, Vector2) ->
Vector2`, `Fade(Color, float) -> Color` и т.п.) — `ff_структура` даёт
такому типу плоский C ABI layout, который знает, как упаковывать/
распаковывать libffi:

```panos-cli
тип Vector2 = ff_структура
    x: Число(32)
    y: Число(32)
конец

внешний "/opt/homebrew/opt/raylib/lib/libraylib.dylib" функ Vector2Add(a: Vector2, b: Vector2) -> Vector2

функ старт() -> Число
    пер r = Vector2Add(Vector2(1.0, 2.0), Vector2(3.0, 4.0))
    r.x + r.y // 10.0
конец
```

`ff_структура` — четвёртая форма `тип X = ...` рядом со `структура`/
`перечисление`/`интерфейс`. Поля — ТОЛЬКО `Целое(8|32|64)`/`Число(32|64)`
(плоский layout, без вложенных структур/строк/указателей в этом срезе).
На panos-стороне значение — обычный именованный конструктор
(`Vector2(1.0, 2.0)` или `Vector2(x = 1.0, y = 2.0)`, Стадия 36) и обычный
`.`-доступ к полям (`r.x`) — тот же путь, что у обычных структур,
`ff_структура` лишь добавляет C ABI метаданные (byte offset'ы каждого
поля, посчитанные libffi через `ffi_get_struct_offsets`) поверх того же
`Aggregate_Value`, никакого нового представления значения в GC.

`ff_структура`-тип можно использовать как тип параметра/возврата
`внешний`-функции напрямую (без скобок с шириной — сам факт, что имя
резолвится в `ff_структура`, достаточен для парсера/тайпчекера).

## Как это работает

1. **Резолв**: `dynlib.load_library("libc.dylib")` (кэшируется по
   логическому имени — несколько `внешний "libc" ...` не грузят библиотеку
   повторно) + `dynlib.symbol_address(lib, "getpid")` — даёт `rawptr` на
   реальный код функции. Если логическое имя уже содержит платформенное
   расширение (`.dylib`/`.dll`/`.so`) — например, полный путь вроде
   `/opt/homebrew/opt/raylib/lib/libraylib.dylib` для библиотек не на
   стандартных путях поиска dyld/ld.so — резолвер использует его как
   есть, без повторного добавления расширения.
2. **Компиляция**: вызов компилируется в свой опкод `.Call_Foreign` (не
   обычный `.Call` — нет `^Compiled_Function`, есть `rawptr` + C-сигнатура).
3. **Вызов**: каждый панос-аргумент упаковывается по своему marshal-виду
   — `Целое(N)` в C-целое нужной ширины, `Число(N)` в C `float`/`double`,
   `КСтрока` в свежий null-terminated C-буфер, `Указатель(T)` передаётся
   как raw-указатель напрямую, `ff_структура`-значение упаковывается в
   плоский байтовый буфер по своим offset'ам — `ffi_call` вызывает
   `fn_ptr` по реальной C-сигнатуре, результат распаковывается обратно
   тем же способом в обратную сторону. `ffi_prep_cif` (построение
   метаданных сигнатуры для libffi, включая размер/alignment составных
   типов) выполняется один раз на декларацию, лениво, при первом реальном
   вызове — не на каждый вызов.

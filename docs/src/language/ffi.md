# Внешние функции (FFI фаза B, первый срез)

> `внешний`-декларации требуют `core:dynlib`-загрузку системной библиотеки
> в рантайме — **недоступны** в браузерной песочнице (тот же принцип, что
> `фс`/`ос`/`сеть`/`графика` — см. [«Модули» → ограничение в
> браузере](./modules.md#ограничение-в-браузере)). Примеры не запускаются
> на странице, запускайте локально: `./panos файл.ps`.

`внешний` — настоящий *динамический* FFI: в отличие от [`графика`](./graphics.md)
(FFI фаза A, статические Odin-биндинги поверх `vendor:raylib`, вкомпилированные
в `panos` заранее), `внешний`-декларация грузит библиотеку и ищет символ
**в рантайме**, по имени, прочитанному из `.ps`-исходника. Механизм — `core:
dynlib` (найти библиотеку/функцию) + [libffi](https://sourceware.org/libffi/)
(вызвать найденный указатель с сигнатурой, известной только в рантайме) —
статически слинкованный вендоренный `libffi.a` (см. `external/libffi/
README.md`), не системная зависимость.

Это **первый, минимальный срез** — доказывает механизм целиком (загрузка +
маршаллинг аргументов и возврата), но покрывает только один случай.
**Явно не сделано** (см. ROADMAP §Стадия 8 на оставшийся объём):

- Строки (`КСтрока`/marshalling `Строка` <-> `char*`).
- Указатели/`Указатель(T)`, ownership-аннотации.
- Структуры по C ABI (`ff_структура`).
- Callback'и (передача panos-функции как указателя в C, `ffi_closure`).
- Восстановление после SIGSEGV в чужом коде.
- `Число(32)`/`Число(64)` (float-ширины) — сейчас только `Целое(32)`/`Целое(64)`.
- Платформы, отличные от `darwin-arm64` (см. `external/libffi/README.md` —
  честно задокументированный, не скрытый пробел).

## Объявление

```panos-cli
внешний "libc" функ getpid() -> Целое(32)
внешний "libc" функ abs(x: Целое(32)) -> Целое(32)

функ старт() -> Целое
    ввод_вывод.печать(getpid())
    abs(0 - 5)
конец
```

`внешний "имя_библиотеки" функ имя(параметры) -> Целое(N)` — верхнеуровневая
декларация (наравне с `функ`/`структура`/`импорт`), без тела. `"libc"` —
логическое имя библиотеки, БЕЗ платформенного расширения/пути (резолвер сам
достраивает `libc.dylib`/`libc.so`/`libc.dll` под текущую ОС). Резолвится
через `dynlib.load_library` + `dynlib.symbol_address` **на этапе резолва**,
до typecheck/компиляции — несуществующая библиотека или несуществующий
символ даёт понятную `Resolve Error`, а не крэш в рантайме:

```
Resolve Error: библиотека 'libc' не найдена (libc.dylib)
Resolve Error: библиотека 'libc' не экспортирует символ 'xyz'
```

Объявленная функция вызывается как **любая обычная** функция —
`getpid()`, `abs(x)` — никакого нового синтаксиса на стороне вызова.
`внешний` нельзя экспортировать (`экспорт внешний ...` — синтаксическая
ошибка): у неё нет "владеющего" модуля в обычном смысле, каждый файл,
которому нужна `внешний`-функция, объявляет её сам.

## `Целое(N)` — только маршаллинг, не новый тип

`Целое(32)`/`Целое(64)` — синтаксис, валидный **только** внутри
`внешний`-сигнатуры (параметры и возврат). Число в скобках — ширина C ABI
типа (`int32_t` vs `int64_t`), которую использует libffi при упаковке/
распаковке. Для остального языка (арифметика, унификация типов, другие
вызовы) параметр/возврат — обычный `Целое`, БЕЗ ширины: `Целое(32)` — не
отдельный тип, не `Целое`-с-тегом, ширина нигде не хранится за пределами
самой `внешний`-декларации.

```panos-cli
внешний "libc" функ abs(x: Целое(32)) -> Целое(32)

функ квадрат_модуля(n: Целое) -> Целое
    пер m = abs(n) // n — обычный Целое, результат тоже
    m * m
конец
```

## Как это работает

1. **Резолв**: `dynlib.load_library("libc.dylib")` (кэшируется по
   логическому имени — несколько `внешний "libc" ...` не грузят библиотеку
   повторно) + `dynlib.symbol_address(lib, "getpid")` — даёт `rawptr` на
   реальный код функции.
2. **Компиляция**: вызов компилируется в свой опкод `.Call_Foreign` (не
   обычный `.Call` — нет `^Compiled_Function`, есть `rawptr` + C-сигнатура).
3. **Вызов**: каждый панос-аргумент (`f64` в рантайм-представлении, как и
   обычный `Целое`/`Число`) упаковывается в буфер нужной C-ширины (`i32`/
   `i64`), `ffi_call` вызывает `fn_ptr` по реальной C-сигнатуре, результат
   распаковывается обратно в `f64`. `ffi_prep_cif` (построение метаданных
   сигнатуры для libffi) выполняется один раз на декларацию, лениво, при
   первом реальном вызове — не на каждый вызов.

# Процессы

panos поддерживает лёгковесные процессы в духе Erlang/Akka: изолированные
единицы исполнения, которые общаются **только** сообщениями через mailbox
— никакой разделяемой памяти между ними. Планировщик однопоточный и
кооперативный (без реального параллелизма), но модель изоляции та же, что
у настоящих акторов.

## Порождение процесса

```panos
тип Сообщение = перечисление
    Увеличить
конец

функ счётчик(состояние: Число) -> Пусто
    выбор получить()
        Сообщение.Увеличить -> счётчик(состояние + 1)
    конец
конец

функ старт() -> Строка
    пер proc: Процесс(Сообщение) = запусти счётчик(0)
    отправить(proc, Сообщение.Увеличить)
    отправить(proc, Сообщение.Увеличить)
    "готово"
конец
```

`запусти <вызов>` порождает новый процесс, который начинает исполнять тело
вызванной функции, и сразу возвращает типизированный хэндл `Процесс(T)` —
`T` здесь означает «этот процесс принимает сообщения типа `T`». Сам вызов
**не выполняется** в текущем процессе — только его функция и уже
вычисленные аргументы передаются новому.

`отправить(процесс, сообщение)` кладёт сообщение в FIFO-mailbox
адресата — обычная функция, не оператор, ничего не блокирует.

## Приём сообщений

`получить()` — блокирующий (точнее, кооперативно приостанавливающий текущий
процесс) builtin: если mailbox пуст, процесс засыпает до следующего
сообщения; когда оно приходит, `получить()` возвращает его, и выполнение
продолжается с того же места.

Тип `T` у `получить()` не объявляется явно — он выводится из того, как вы
разбираете результат в `выбор`. Ветки **обязаны** использовать
полную квалификацию `Тип.Вариант` (`Сообщение.Увеличить`, не голое
`Увеличить`) — только так typechecker узнаёт, какое перечисление ожидать:

```panos
тип Команда = перечисление
    Стоп
    Ход(Число)
конец

функ обработчик(пройдено: Число) -> Число
    выбор получить()
        Команда.Стоп -> пройдено
        Команда.Ход(шаг) -> обработчик(пройдено + шаг)
    конец
конец

функ старт() -> Строка
    пер proc: Процесс(Команда) = запусти обработчик(0)
    отправить(proc, Команда.Ход(5))
    отправить(proc, Команда.Стоп)
    "готово"
конец
```

Тело процесса — обычно рекурсивная функция: каждая ветка `выбор`
либо вызывает себя же (с обновлённым состоянием как явным аргументом,
никаких мутируемых полей объекта — состояние процесса это просто параметры
функции), либо завершает работу, возвращая значение (тип возврата у
`обработчик` — просто обычный тип функции, `запусти` его не читает и
никак не использует после завершения процесса).

## Ответ отправителю: `себя()`

Чтобы процесс мог ответить, ему нужен адрес отправителя. `себя()` — тоже
builtin, возвращает `Процесс(T)`-хэндл **текущего** процесса, который
можно передать как часть сообщения:

```panos
тип Запрос = перечисление
    Прочитать(Процесс(Число))
конец

функ хранилище() -> Пусто
    выбор получить()
        Запрос.Прочитать(отвечающему) -> отправить(отвечающему, 77)
    конец
конец

функ старт() -> Число
    пер proc: Процесс(Запрос) = запусти хранилище()
    отправить(proc, Запрос.Прочитать(себя()))
    получить()
конец
```

`старт()` — обычный процесс (тот самый, с которого начинается программа),
и `получить()` на его собственном mailbox работает так же, как везде:
ждёт, пока `хранилище()` пришлёт ответ, и разворачивает результат.

## Copy-on-send

Сообщение, отправленное через `отправить`, **копируется** в момент
отправки — мутация исходного значения после отправки не видна получателю,
и наоборот. Копирование рекурсивное (глубокое) и происходит автоматически;
переопределить его для конкретного типа можно через
`реализация Копируемое` (см. [«Интерфейсы прелюдии»](./prelude-interfaces.md#копируемое)).
Примитивы (`Число`/`Булево`) и строки не копируются — они и так неизменяемы,
шарить их безопасно.

## Жизненный цикл

Процесс завершается, когда его функция возвращает значение (или доходит до
конца тела). Программа в целом завершается сразу, как только вернулся
**первый** процесс — тот, что начал работу с `старт()` — не дожидаясь
остальных, даже если они ещё живы или ждут сообщений. Это то же поведение,
что у `main()` в Go: осиротевшие процессы просто перестают существовать
вместе со всей программой.

Отправка сообщения уже завершившемуся процессу — тихий no-op, не ошибка и
не `Результат`: синхронная проверка «жив ли процесс» на момент отправки
всё равно ничего не гарантирует (он может завершиться микросекундой позже),
так что `Результат` создавал бы ложное чувство надёжности.

## Дедлок

Если `получить()` блокирует процесс, а прислать ему сообщение больше
некому (все остальные процессы либо завершились, либо сами ждут), это
настоящий, а не приближённый дедлок — планировщик обнаруживает его и
завершает программу понятной ошибкой вместо зависания:

```panos
тип Сообщение = перечисление
    X
конец

функ старт() -> Число
    выбор получить()
        Сообщение.X -> 0
    конец
конец
```

```text
Runtime Error: все процессы заблокированы в ожидании сообщений (дедлок)
```

## Наблюдение за процессом

До сих пор единственным способом узнать что-то о судьбе процесса было
подождать ответное сообщение самому — падение процесса (например, через
`паника()`, деление на ноль или индекс за границей массива) молча
уносило с собой всю остальную наблюдаемую информацию. `наблюдать`/
`получить_сигнал` дают отдельный, специализированный канал именно для
этого — узнать, ЧТО случилось с другим процессом, не дожидаясь от него
делового сообщения:

```panos
импорт ввод_вывод
импорт строки

тип Задача = перечисление
    Выполнить
конец

функ рабочий() -> Пусто
    выбор получить()
        Задача.Выполнить -> паника("не справился")
    конец
конец

функ старт() -> Строка
    пер proc: Процесс(Задача) = запусти рабочий()
    наблюдать(proc)
    отправить(proc, Задача.Выполнить)

    пер (id, причина) = получить_сигнал()
    выбор причина
        Нет -> ввод_вывод.строка("процесс завершился штатно")
        Есть(текст) -> ввод_вывод.строка("процесс " + строки.из_целого(id) + " упал: " + текст)
    конец
    "программа жива"
конец
```

`наблюдать(процесс)` регистрирует ТЕКУЩИЙ процесс наблюдателем цели —
работает для `Процесс(T)` с любым `T`. `получить_сигнал()` — блокирующий
builtin (тот же кооперативный suspend/resume, что `получить()`), но на
СВОЕЙ, отдельной от `mailbox` очереди — сигналы никогда не перемешиваются
с обычными сообщениями. Тип результата фиксирован — `(Целое, Опция(Строка))`:
первый элемент — id упавшего/завершившегося процесса (сравнивайте с
`.номер()` — единственный метод на `Процесс(T)`), второй — `Нет` при
штатном завершении, `Есть(текст)` при краше, с человекочитаемой причиной.

Наблюдать можно и за уже мёртвым процессом — вместо зависания на
`получить_сигнал()` навсегда наблюдатель немедленно получает синтетический
сигнал `Есть("процесс уже не существует")`.

**Изоляция крашей.** До этой возможности ЛЮБОЙ runtime-краш (`паника()`,
деление на ноль, индекс за границей массива, ключ не найден в
`соответствие`) ронял ВСЮ программу — теперь такой краш роняет только
сам упавший процесс (если это не «процесс #0», см. ниже), остальная
программа продолжает работать; наблюдатели узнают о причине через
`получить_сигнал()`. Единственное исключение — краш **в самом `старт()`**
(«процесс #0»): он по-прежнему фатален для всей программы, там некому
наблюдать за корневым процессом.

`наблюдать`/`получить_сигнал` — строительный блок, на котором супервизия
(рестарт упавшего дочернего процесса) реализована обычным panos-кодом —
см. ниже.

## Супервизия

`std/супервизор.ps` даёт минимальную one_for_one супервизию (каждый
ребёнок рестартует независимо от остальных) поверх `наблюдать`/
`получить_сигнал` — чистая библиотека, без новых языковых конструкций.

```panos
импорт супервизор

тип Задача = перечисление
    Выполнить
конец

функ рабочий() -> Пусто
    выбор получить()
        Задача.Выполнить -> паника("не справился")
    конец
конец

функ старт_рабочего() -> Целое
    пер proc: Процесс(Задача) = запусти рабочий()
    наблюдать(proc)
    отправить(proc, Задача.Выполнить)
    proc.номер()
конец

тип МояЗадача = структура
конец

реализация супервизор.ДочерняяЗадача для МояЗадача
    функ запустить(это: МояЗадача) -> Целое
        старт_рабочего()
    конец
    функ имя(это: МояЗадача) -> Строка
        "мой-рабочий"
    конец
конец

// `запусти` требует bare-имя функции — module-qualified вызов нельзя
// спавнить напрямую, отсюда маленькая обёртка.
функ запустить_супервизора(задачи: Массив(супервизор.ДочерняяЗадача)) -> Пусто
    супервизор.супервизировать(задачи, 3)
конец

функ старт() -> Пусто
    пер задачи: Массив(супервизор.ДочерняяЗадача) = массив(МояЗадача())
    запусти запустить_супервизора(задачи)
конец
```

`ДочерняяЗадача` — [интерфейс](./interfaces.md), объявленный в
`супервизор` и реализуемый (через [`реализация Модуль.Интерфейс для
Тип`](./interfaces.md#интерфейс-из-другого-модуля)) КАЖДЫМ конкретным
типом задачи В СВОЁМ файле — разные задачи под одним супервизором могут
иметь совершенно разные типы сообщений `T`, гетерогенность как в
OTP-супервизорах, достигается через динамическую диспетчеризацию
интерфейса, а не через generic-параметр `Массива`.

`запустить()` — сама "фабрика": вызывается один раз при старте
супервизора и заново при каждом рестарте. У panos-лямбд НЕТ захвата
внешних локалей (замыканий не существует), поэтому конфигурация ребёнка
не передаётся при рестарте параметром — она "зашита" в тело `запустить()`
буквально как литеральный `запусти <функция>(...)`. Метод сам же
вызывает `наблюдать(proc)` и возвращает `proc.номер()` — id, по которому
супервизор потом сопоставляет сигналы `получить_сигнал()` с конкретным
ребёнком (сопоставление по ПОЗИЦИИ в массиве задач, не по id — id меняется
на каждый рестарт).

Штатное завершение ребёнка (`Нет` от `получить_сигнал()`) НЕ рестартует
— считается, что ребёнок сделал свою работу (transient-семантика,
единственная поддержанная политика). Краш (`Есть(причина)`) рестартует,
пока счётчик рестартов конкретного ребёнка не превысит
`макс_рестартов_на_ребёнка` (второй аргумент `супервизировать`) — тогда
сам супервизор паникует с сообщением, называющим ребёнка (`.имя()`) и
причину последнего краша — тот же "let it crash" принцип, что у
остального языка (если супервизор сам под наблюдением — это тоже будет
замечено).

`супервизировать(...)` блокирует вызывающего НАВСЕГДА (внутренний
бесконечный цикл на `получить_сигнал()`) — предназначена для запуска
через `запусти` (обёрнутую, как выше — `запусти` не спавнит
module-qualified вызовы напрямую), не для прямого вызова.

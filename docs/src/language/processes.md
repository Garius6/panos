# Процессы

panos поддерживает лёгковесные процессы в духе Erlang/Akka: изолированные
единицы исполнения, которые общаются **только** сообщениями через mailbox
— никакой разделяемой памяти между ними. Планировщик однопоточный и
кооперативный (без реального параллелизма), но модель изоляции та же, что
у настоящих акторов.

## Порождение процесса

```panos
тип Сообщение = перечисление
    Увеличить
конец

функ счётчик(состояние: Число) -> Пусто
    выбор получить()
        Сообщение.Увеличить -> счётчик(состояние + 1)
    конец
конец

функ старт() -> Строка
    пер proc: Процесс(Сообщение) = запусти счётчик(0)
    отправить(proc, Сообщение.Увеличить)
    отправить(proc, Сообщение.Увеличить)
    "готово"
конец
```

`запусти <вызов>` порождает новый процесс, который начинает исполнять тело
вызванной функции, и сразу возвращает типизированный хэндл `Процесс(T)` —
`T` здесь означает «этот процесс принимает сообщения типа `T`». Сам вызов
**не выполняется** в текущем процессе — только его функция и уже
вычисленные аргументы передаются новому.

`<вызов>` — либо функция из текущего файла (`запусти счётчик(0)`), либо
module-qualified вызов (`запусти модуль.функция(...)`) — `T` в обоих
случаях выводится из `получить()`-паттернов ВНУТРИ спавненной функции, а
не из места вызова `запусти`. Поэтому обёртка вида `функ f() -> Пусто \n
модуль.g() \n конец` (которая сама НЕ вызывает `получить()`) даёт
handle с НЕконстрейненным `T` — типобезопасность теряется незаметно
(`отправить` любого типа пройдёт тайпчек). Если нужен типобезопасный
handle на экспортированную функцию — спавньте её саму, `запусти
модуль.функция(...)`, а не обёртку вокруг неё.

`отправить(процесс, сообщение)` кладёт сообщение в FIFO-mailbox
адресата — обычная функция, не оператор, ничего не блокирует.

## Приём сообщений

`получить()` — блокирующий (точнее, кооперативно приостанавливающий текущий
процесс) builtin: если mailbox пуст, процесс засыпает до следующего
сообщения; когда оно приходит, `получить()` возвращает его, и выполнение
продолжается с того же места.

Тип `T` у `получить()` не объявляется явно — он выводится из того, как вы
разбираете результат в `выбор`. Ветки **обязаны** использовать
полную квалификацию `Тип.Вариант` (`Сообщение.Увеличить`, не голое
`Увеличить`) — только так typechecker узнаёт, какое перечисление ожидать:

```panos
тип Команда = перечисление
    Стоп
    Ход(Число)
конец

функ обработчик(пройдено: Число) -> Число
    выбор получить()
        Команда.Стоп -> пройдено
        Команда.Ход(шаг) -> обработчик(пройдено + шаг)
    конец
конец

функ старт() -> Строка
    пер proc: Процесс(Команда) = запусти обработчик(0)
    отправить(proc, Команда.Ход(5))
    отправить(proc, Команда.Стоп)
    "готово"
конец
```

Тело процесса — обычно рекурсивная функция: каждая ветка `выбор`
либо вызывает себя же (с обновлённым состоянием как явным аргументом,
никаких мутируемых полей объекта — состояние процесса это просто параметры
функции), либо завершает работу, возвращая значение (тип возврата у
`обработчик` — просто обычный тип функции, `запусти` его не читает и
никак не использует после завершения процесса).

## Ответ отправителю: `себя()`

Чтобы процесс мог ответить, ему нужен адрес отправителя. `себя()` — тоже
builtin, возвращает `Процесс(T)`-хэндл **текущего** процесса, который
можно передать как часть сообщения:

```panos
тип Запрос = перечисление
    Прочитать(Процесс(Число))
конец

функ хранилище() -> Пусто
    выбор получить()
        Запрос.Прочитать(отвечающему) -> отправить(отвечающему, 77)
    конец
конец

функ старт() -> Число
    пер proc: Процесс(Запрос) = запусти хранилище()
    отправить(proc, Запрос.Прочитать(себя()))
    получить()
конец
```

`старт()` — обычный процесс (тот самый, с которого начинается программа),
и `получить()` на его собственном mailbox работает так же, как везде:
ждёт, пока `хранилище()` пришлёт ответ, и разворачивает результат.

## Copy-on-send

Сообщение, отправленное через `отправить`, **копируется** в момент
отправки — мутация исходного значения после отправки не видна получателю,
и наоборот. Копирование рекурсивное (глубокое) и происходит автоматически;
переопределить его для конкретного типа можно через
`реализация Копируемое` (см. [«Интерфейсы прелюдии»](./prelude-interfaces.md#копируемое)).
Примитивы (`Число`/`Булево`) и строки не копируются — они и так неизменяемы,
шарить их безопасно.

## Жизненный цикл

Процесс завершается, когда его функция возвращает значение (или доходит до
конца тела). Программа в целом завершается сразу, как только вернулся
**первый** процесс — тот, что начал работу с `старт()` — не дожидаясь
остальных, даже если они ещё живы или ждут сообщений. Это то же поведение,
что у `main()` в Go: осиротевшие процессы просто перестают существовать
вместе со всей программой.

Отправка сообщения уже завершившемуся процессу — тихий no-op, не ошибка и
не `Результат`: синхронная проверка «жив ли процесс» на момент отправки
всё равно ничего не гарантирует (он может завершиться микросекундой позже),
так что `Результат` создавал бы ложное чувство надёжности.

## Дедлок

Если `получить()` блокирует процесс, а прислать ему сообщение больше
некому (все остальные процессы либо завершились, либо сами ждут), это
настоящий, а не приближённый дедлок — планировщик обнаруживает его и
завершает программу понятной ошибкой вместо зависания:

```panos
тип Сообщение = перечисление
    X
конец

функ старт() -> Число
    выбор получить()
        Сообщение.X -> 0
    конец
конец
```

```text
Runtime Error: все процессы заблокированы в ожидании сообщений (дедлок)
```

## Наблюдение за процессом

До сих пор единственным способом узнать что-то о судьбе процесса было
подождать ответное сообщение самому — падение процесса (например, через
`паника()`, деление на ноль или индекс за границей массива) молча
уносило с собой всю остальную наблюдаемую информацию. `наблюдать`/
`получить_сигнал` дают отдельный, специализированный канал именно для
этого — узнать, ЧТО случилось с другим процессом, не дожидаясь от него
делового сообщения:

```panos
импорт ввод_вывод
импорт строки

тип Задача = перечисление
    Выполнить
конец

функ рабочий() -> Пусто
    выбор получить()
        Задача.Выполнить -> паника("не справился")
    конец
конец

функ старт() -> Строка
    пер proc: Процесс(Задача) = запусти рабочий()
    наблюдать(proc)
    отправить(proc, Задача.Выполнить)

    пер (id, причина) = получить_сигнал()
    выбор причина
        Нет -> ввод_вывод.строка("процесс завершился штатно")
        Есть(текст) -> ввод_вывод.строка("процесс " + строки.из_целого(id) + " упал: " + текст)
    конец
    "программа жива"
конец
```

`наблюдать(процесс)` регистрирует ТЕКУЩИЙ процесс наблюдателем цели —
работает для `Процесс(T)` с любым `T`. `получить_сигнал()` — блокирующий
builtin (тот же кооперативный suspend/resume, что `получить()`), но на
СВОЕЙ, отдельной от `mailbox` очереди — сигналы никогда не перемешиваются
с обычными сообщениями. Тип результата фиксирован — `(Целое, Опция(Строка))`:
первый элемент — id упавшего/завершившегося процесса (сравнивайте с
`.номер()` — единственный метод на `Процесс(T)`), второй — `Нет` при
штатном завершении, `Есть(текст)` при краше, с человекочитаемой причиной.

Наблюдать можно и за уже мёртвым процессом — вместо зависания на
`получить_сигнал()` навсегда наблюдатель немедленно получает синтетический
сигнал `Есть("процесс уже не существует")`.

**Изоляция крашей.** До этой возможности ЛЮБОЙ runtime-краш (`паника()`,
деление на ноль, индекс за границей массива, ключ не найден в
`соответствие`) ронял ВСЮ программу — теперь такой краш роняет только
сам упавший процесс (если это не «процесс #0», см. ниже), остальная
программа продолжает работать; наблюдатели узнают о причине через
`получить_сигнал()`. Единственное исключение — краш **в самом `старт()`**
(«процесс #0»): он по-прежнему фатален для всей программы, там некому
наблюдать за корневым процессом.

`наблюдать`/`получить_сигнал` — строительный блок, на котором супервизия
(рестарт упавшего дочернего процесса) реализована обычным panos-кодом —
см. ниже.

## Принудительная остановка

`убить(процесс)` — единственный способ прервать ЧУЖОЙ процесс извне (до
этого процесс мог завершиться только сам — штатно или крахом):

```panos
тип Задача = перечисление
    Выполнить
конец

функ бесконечный() -> Пусто
    выбор получить()
        Задача.Выполнить -> бесконечный()
    конец
конец

функ старт() -> Строка
    пер жертва: Процесс(Задача) = запусти бесконечный()
    наблюдать(жертва)
    убить(жертва)

    пер (id, причина) = получить_сигнал()
    выбор причина
        Нет -> "неожиданно: нормально"
        Есть(текст) -> текст
    конец
конец
```

Наблюдатели убитого процесса получают тот же сигнал, что при обычном
краше — `Есть("процесс принудительно остановлен (убить())")` — убийство
неотличимо от краша с точки зрения `получить_сигнал()`, той же
изоляцией (остальная программа продолжает работать).

Убийство уже мёртвого процесса — тихий no-op (симметрично `отправить()`
на мёртвый процесс). Два случая запрещены явно, с понятной фатальной
ошибкой, а не неопределённым поведением:

- **Самоубийство** (`убить(себя())`) — `execute()` не умеет прервать
  сам себя посреди собственных инструкций, `убить()` — обычный
  синхронный builtin-вызов, простой возврат из него не остановил бы
  оставшееся тело процесса.
- **Убийство «старт()»** (процесс #0) — у корневого процесса уже особая
  семантика (крах/завершение останавливает всю программу целиком);
  убийство его из другого процесса потребовало бы прокидывать этот
  сигнал наверх в планировщик из глубины builtin-вызова, что не стоит
  сложности ради этого узкого случая.

## Двусторонняя связь

`связать(процесс)` — в отличие от `наблюдать` (одностороннее, только
уведомление), даёт ДВУСТОРОННЮЮ связь: краш ЛЮБОЙ из двух связанных
сторон каскадно останавливает и другую, не только уведомляет о нём.
Штатное завершение НЕ каскадирует — только краш (тот же принцип, что у
Erlang: `normal`-завершение не убивает связанные процессы):

```panos
тип Задача = перечисление
    Выполнить
конец

тип Пинг = перечисление
    Готово
конец

функ падающий() -> Пусто
    выбор получить()
        Задача.Выполнить -> паника("бум")
    конец
конец

функ сосед(партнёр: Процесс(Задача), родитель: Процесс(Пинг)) -> Пусто
    связать(партнёр)
    отправить(родитель, Пинг.Готово)
    выбор получить()
        Задача.Выполнить -> сосед(партнёр, родитель)
    конец
конец

функ старт() -> Строка
    пер жертва: Процесс(Задача) = запусти падающий()
    пер сосед_proc: Процесс(Задача) = запусти сосед(жертва, себя())
    наблюдать(сосед_proc)
    выбор получить()
        Пинг.Готово -> 0
    конец
    отправить(жертва, Задача.Выполнить)

    пер (id, причина) = получить_сигнал()
    выбор причина
        Нет -> "неожиданно: нормально"
        Есть(текст) -> текст
    конец
конец
```

`жертва` падает, `сосед` (связан с ней) каскадно завершается тоже —
`старт()` (наблюдает за `сосед`, не за `жертва` напрямую) получает
`Есть("связанный процесс #<id жертвы> упал: Runtime Panic: бум")`.

Зачем нужны ОБА примитива: `наблюдать`/`получить_сигнал` — "сообщи мне,
но не трогай" (Стадия 38), `связать` — "мы с тобой единое целое, если
ты упал — я тоже". В Erlang один примитив (`link`) покрывает оба случая
через `trap_exit` (опция "получать EXIT как обычное сообщение вместо
смерти"); panos обходится без этого переключателя — если нужно только
уведомление без смерти, используйте `наблюдать`, а не `связать`.

Правила, симметричные `убить`:

- **Уже мёртвая цель** — тихий no-op.
- **Самолинковка** (`связать(себя())`) — безвредна, разрешена явно (в
  отличие от `убить(себя())`).
- **«Старт()» ни как цель, ни как инициатор** — если бы root попал в
  чей-то список связей, каскад попытался бы завершить его через
  обычный путь, которым root не заканчивается (только `run_scheduler`
  умеет корректно завершить всю программу для процесса #0) — запрещено
  в обе стороны явной фатальной ошибкой.

## Супервизия

`std/супервизор.ps` даёт супервизию с тремя restart-стратегиями (Erlang
OTP терминология) и динамическим составом детей поверх `наблюдать`/
`получить_сигнал`/`получить` — чистая библиотека, без новых языковых
конструкций (кроме qualified-`запусти`, см. выше).

```panos
импорт супервизор

тип Задача = перечисление
    Выполнить
конец

функ рабочий() -> Пусто
    выбор получить()
        Задача.Выполнить -> паника("не справился")
    конец
конец

функ старт_рабочего() -> Процесс(Задача)
    запусти рабочий()
конец

// наблюдать() регистрирует ТЕКУЩИЙ процесс наблюдателем — значит
// наблюдателем должен быть ИМЕННО этот relay, не супервизор напрямую
// (см. ниже, почему). Монотипный, bare-ident запусти — без generic'ов.
функ ретранслятор(proc: Процесс(Задача), супервизор_h: Процесс(супервизор.СобытиеСупервизора)) -> Пусто
    наблюдать(proc)
    пер (id, причина) = получить_сигнал()
    отправить(супервизор_h, супервизор.СобытиеСупервизора.Сигнал(id, причина))
конец

тип МояЗадача = структура
    последний: Опция(Процесс(Задача))
конец

реализация супервизор.ДочерняяЗадача для МояЗадача
    функ запустить(это: МояЗадача, супервизор_h: Процесс(супервизор.СобытиеСупервизора)) -> Целое
        пер proc = старт_рабочего()
        запусти ретранслятор(proc, супервизор_h)
        это.последний = Опция.Есть(proc)
        proc.номер()
    конец
    функ имя(это: МояЗадача) -> Строка
        "мой-рабочий"
    конец
    функ остановить(это: МояЗадача) -> Пусто
        если это.последний.есть() тогда
            убить(это.последний.значение())
        конец
    конец
конец

функ старт() -> Пусто
    пер задачи: Массив(супервизор.ДочерняяЗадача) = массив(МояЗадача(Опция.Нет()))
    пер супервизор_h = запусти супервизор.супервизировать(задачи, супервизор.Стратегия.ОдинЗаОдного, 3)
    отправить(супервизор_h, супервизор.СобытиеСупервизора.ДобавитьЗадачу(МояЗадача(Опция.Нет())))
конец
```

`ДочерняяЗадача` — [интерфейс](./interfaces.md), объявленный в
`супервизор` и реализуемый (через [`реализация Модуль.Интерфейс для
Тип`](./interfaces.md#интерфейс-из-другого-модуля)) КАЖДЫМ конкретным
типом задачи В СВОЁМ файле — разные задачи под одним супервизором могут
иметь совершенно разные типы сообщений `T`, гетерогенность как в
OTP-супервизорах, достигается через динамическую диспетчеризацию
интерфейса, а не через generic-параметр `Массива`.

`запустить(супервизор)` — сама "фабрика": вызывается один раз при старте
супервизора, заново при каждом рестарте, и при `ДобавитьЗадачу` (см.
ниже). У panos-лямбд НЕТ захвата внешних локалей (замыканий не
существует), поэтому конфигурация ребёнка не передаётся при рестарте
параметром — она "зашита" в тело `запустить()` буквально как литеральный
`запусти <функция>(...)`. Метод спавнит ребёнка, ЗАПУСКАЕТ ЕГО СОБСТВЕННЫЙ
`ретранслятор` (см. пример) и передаёт ему `супервизор` — хэндл,
полученный аргументом. ЗАПОМИНАЕТ `proc` в `Опция(Процесс(T))`-поле своей
структуры (`это.последний = Опция.Есть(proc)` — структуры в panos
reference-типы, мутация поля через `это` видна на следующем вызове через
тот же интерфейсный хэндл) и возвращает `proc.номер()` — id, по которому
супервизор сопоставляет сигналы с конкретным ребёнком (сопоставление по
ПОЗИЦИИ в массиве задач, не по id — id меняется на каждый рестарт).
`остановить()` читает то же поле и зовёт `убить(proc)` (тихий no-op, если
`proc` уже мёртв).

**Почему `наблюдать()` вызывает relay, а не сам `запустить()`.**
`наблюдать(proc)` регистрирует наблюдателем ТЕКУЩИЙ процесс — тот, что
ВЫПОЛНЯЕТ вызов. `запустить()` выполняется синхронно ВНУТРИ процесса
супервизора, значит если бы `наблюдать(proc)` звал сам `запустить()`,
наблюдателем стал бы супервизор — а сигналы шли бы в его СИГНАЛЫ
(`получить_сигнал()`), не в обычный mailbox (`получить()`), который
`супервизировать()` слушает (см. ниже, зачем). panos не имеет `select`
поверх `получить()`/`получить_сигнал()` разом — единственный способ
супервизору узнавать И о крахах, И о внешних командах ОДНИМ каналом —
чтобы наблюдателем была ОТДЕЛЬНАЯ, PER-CHILD задача (`ретранслятор`),
которая сама зовёт `наблюдать(proc)`, ждёт РОВНО один сигнал и форвардит
его супервизору ОБЫЧНЫМ типизированным сообщением.

Штатное завершение ребёнка (`Нет` от сигнала) НЕ рестартует ничего —
считается, что ребёнок сделал свою работу (transient-семантика,
единственная поддержанная политика). Краш (`Есть(причина)`) обрабатывается
согласно `Стратегия` (второй аргумент `супервизировать`):

- **`ОдинЗаОдного`** (one_for_one) — рестартует ТОЛЬКО упавшего ребёнка.
  Лимит (`макс_рестартов`, третий аргумент) — per-child cap, свой счётчик
  у каждого ребёнка.
- **`ВсеЗаОдного`** (one_for_all) — краш любого ребёнка рестартует ВСЕХ.
  Лимит — ОБЩИЙ на весь супервизор (счётчик "сколько раз рестартовала вся
  группа", не на ребёнка).
- **`ОстальныеЗаОдним`** (rest_for_one) — рестартует упавшего ребёнка и
  всех, что объявлены в массиве задач ПОСЛЕ него (Erlang-семантика
  "зависят от более ранних") — те, что раньше, не трогаются. Лимит —
  тот же общий счётчик, что у `ВсеЗаОдного`.

Независимо от стратегии — при исчерпании лимита сам супервизор паникует
с сообщением, называющим упавшего ребёнка (`.имя()`) и причину последнего
краша — тот же "let it crash" принцип, что у остального языка (если
супервизор сам под наблюдением — это тоже будет замечено).

> В демо выше и в фикстурах, где единственный ребёнок падает НЕМЕДЛЕННО
> (сообщение-триггер отправлено синхронно, той же функцией, что его
> спавнит) — причина краха иногда видна как generic-"процесс уже не
> существует" вместо реального текста паники: worker успевает выполниться
> и упасть РАНЬШЕ, чем planировщик впервые дойдёт до его `ретранслятор`а
> (порядок обхода — по индексу спавна, worker всегда раньше своего relay).
> `наблюдать()` на уже мёртвую цель — штатный синтетический сигнал (см.
> "Наблюдение за процессом" ниже), НЕ баг — restart/лимит-логика от
> точного текста причины не зависит. В реальных воркерах, которые делают
> хоть какую-то работу перед крахом, окно гонки закрывается первым же
> `получить()`/вводом-выводом.

При групповом рестарте (`ВсеЗаОдного`/`ОстальныеЗаОдним`) супервизор
СНАЧАЛА зовёт `остановить()` (которая вызывает `убить()` на запомненном
хэндле — тихий no-op для уже упавшего ребёнка, реальная остановка для
выживших siblings), ПОТОМ `запустить()` заново — настоящий рестарт
группы, а не просто спавн свежих процессов рядом со старыми.

### Динамические дети

`супервизировать(...)` возвращает `Пусто` и блокирует вызывающего
НАВСЕГДА — но теперь её можно спавнить НАПРЯМУЮ через qualified-`запусти`
(см. выше), получая типобезопасный `Процесс(СобытиеСупервизора)`. Через
этот хэндл ЛЮБОЙ другой процесс может во время работы супервизора:

```panos
// Добавить нового ребёнка в конец списка — запускается сразу.
отправить(супервизор_h, супервизор.СобытиеСупервизора.ДобавитьЗадачу(новая_задача))

// Остановить ребёнка по его текущему индексу и перестать за ним следить.
отправить(супервизор_h, супервизор.СобытиеСупервизора.УбратьЗадачу(0))
```

`УбратьЗадачу` не удаляет запись физически (у `Массив` нет метода
удаления по индексу) — ставит tombstone (внутренний id ребёнка = -1),
после чего запись невидима для рестартов и для сопоставления будущих
сигналов. Индексы ОСТАЛЬНЫХ детей не сдвигаются.

Супервизор можно по-прежнему вызывать СИНХРОННО (`супервизор.
супервизировать(задачи, ...)` напрямую, без `запусти`) — тогда
вызывающий процесс сам становится супервизором; динамические
add/remove-команды в этом случае может слать кто угодно, кто знает его
`себя()`-хэндл (например, сам вызывающий, до того как заблокируется).

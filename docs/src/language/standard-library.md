# Стандартная библиотека

Модули из `std/` (репозиторий: `std/математика.ps`, `std/коллекции.ps` и
т.д.) — написаны на самом panos, а не встроены в компилятор. Резолвятся
как обычный `импорт` (см. [«Модули»](./modules.md)) через `PANOS_STDLIB`/
`модули/`/репозиторный `std/`. HTTP-клиент (`сеть/http`) вынесен в
[«Сеть и HTTP»](./networking.md) — он использует `сеть`, поэтому недоступен
в браузере; всё остальное на этой странице — чистые вычисления, работает и
в интерактивной песочнице.

## `математика`

```panos
импорт математика

функ старт() -> Число
    математика.степень(2, 10) + математика.квадратный_корень(16)
конец
```

`пи()`, `е()`, `квадрат(x)`, `абс(x)`, `мин(a, b)`, `макс(a, b)`,
`степень(x, n)` (целая степень, `n` может быть отрицательным),
`квадратный_корень(x)` (метод Ньютона, `x <= 0` даёт `0` — без `Опция`/
паники).

`округл_вниз(x)`/`округл_вверх(x)`/`округлить(x)` — округление к целому
значению, результат — `Число` (не `Целое`: коэрсии между ними всё ещё нет,
см. [«Число и Целое»](./basic-types.md#число-и-целое), но она и не нужна
для округления — усечение реализовано бинарным спуском по степеням
двойки, целиком в `Число`). `округлить` — половина округляется от нуля
(`округлить(2.5) == 3`, `округлить(-2.5) == -3`).

Генератор случайных чисел — `следующее(seed)`/`дробь(seed)`/
`диапазон(seed, нижняя, верхняя)` (Lehmer/Park-Miller LCG). `seed` явный:
в panos нет модульных глобальных переменных, поэтому состояние генератора
пробрасывается вызывающим кодом между вызовами, а не хранится неявно
внутри модуля; `seed` должен быть ненулевым. Первый seed обычно берут из
`время.монотонно_мс()`:

```panos
импорт математика
импорт время

функ старт() -> Число
    пер seed = время.монотонно_мс()
    математика.диапазон(seed, 1, 6)  // кубик 1..6
конец
```

Построение строк из значений — интерполяция `\(x)` прямо в литерале (см.
[«Строки» → «Интерполяция»](./basic-types.md#интерполяция)), не отдельный
stdlib-модуль.

## `коллекции`

Generic-комбинаторы над `Массив(T)` — компараторы передаются лямбдой, не
встроенным `<`, поэтому `отсортировать` работает и на типах без
`реализация Сравниваемое`:

```panos
импорт коллекции

функ старт() -> Число
    пер числа = массив(3, 1, 4, 1, 5)
    пер отсортированные = коллекции.отсортировать(числа, функ(a, b) a < b конец)
    пер сумма = коллекции.свернуть(числа, 0, функ(накопитель, x) накопитель + x конец)
    отсортированные[0] + сумма
конец
```

- `отфильтровать[T](массив, предикат: функ(T) -> Булево) -> Массив(T)`
- `отобразить[T, U](массив, преобразование: функ(T) -> U) -> Массив(U)`
- `свернуть[T, U](массив, начало: U, шаг: функ(U, T) -> U) -> U`
- `отсортировать[T](массив, меньше: функ(T, T) -> Булево) -> Массив(T)` —
  insertion sort, стабильная

## `флаги`: разбор аргументов командной строки

```panos
импорт флаги

функ старт() -> Строка
    пер конфиг = флаги.новая_конфигурация()
    конфиг.добавить_флаг(флаги.Флаг("имя", флаги.ТипФлага.тСтрока))
    пер результат = конфиг.распарсить_флаги(массив("-имя=мир"))
    результат.получить("имя", флаги.ЗначениеФлага.фСтрока("")).вСтроку()
конец
```

Объявите флаги через `Флаг(наименование, тип)` (`ТипФлага.тСтрока` /
`.тЧисло` / `.тБулево`), добавьте их в `Конфигурация` через
`.добавить_флаг`, затем разберите либо явный массив строк
(`.распарсить_флаги(аргументы)` — как выше, удобно для тестов), либо
сразу `ос.аргументы()` через `.распарсить()` (требует `импорт ос`, см.
[«Использование CLI»](../getting-started/cli.md)). Поддержаны формы
`-флаг=значение` и `-флаг значение`; для `тБулево` значение не читается —
само присутствие флага значит `истина`. Необъявленный флаг или
пропущенное значение — паника, не `Результат` (тот же принцип, что у
`кодирование/json`/`кодирование/toml` ниже: ошибка во входных данных —
критичный сбой).

## `кодирование/json`, `кодирование/toml`

Оба модуля — парсеры в дерево `Значение` с одинаковым интерфейсом доступа
и оба умеют сериализовать обратно в текст. Импортируются по пути в
кавычках (имя с `/`):

```panos
импорт "кодирование/json" как json

функ старт() -> Строка
    пер результат = json.разобрать("{\"имя\": \"panos\", \"версия\": 1}")
    если результат.ошибка() тогда
        возврат "ошибка парсинга"
    конец
    результат.значение().поле("имя").значение().как_строку().получить("?")
конец
```

`разобрать(текст) -> Результат(Значение, Ошибка)` и
`сериализовать(значение) -> Строка` — есть в обоих модулях.

Методы `Значение` — общие для обоих модулей:

- `.как_строку() / .как_число() / .как_булево() -> Опция(...)`
- `.поле(ключ) -> Опция(Значение)` — для объекта (JSON) / таблицы (TOML)
- `.элемент(индекс) -> Опция(Значение)` — для списка

Варианты `Значение` отражают форму каждого формата. JSON:
`Ноль`/`Булево(Булево)`/`Число(Число)`/`Строка(Строка)`/
`Список(Массив(Значение))`/`Объект(Соответствие(Строка, Значение))`.

TOML отдельно различает целые и дробные числа (как и сам язык panos —
`Целое`/`Число`), и добавляет дата-время как сырой текст (структурного
типа даты в языке нет):
`Строка(Строка)`/`Целое(Целое)`/`Число(Число)`/`Булево(Булево)`/
`Время(Строка)`/`Список(Массив(Значение))`/
`Таблица(Соответствие(Строка, Значение))`. Доп. методы:
`.как_целое() -> Опция(Целое)`, `.как_время() -> Опция(Строка)`;
`.как_число()` работает и для `Целое`, и для `Число` (расширяет целое
до дробного).

`кодирование/toml` — hand-written реализация TOML v1.0.0 на чистом
panos (без внешних/нативных TOML-библиотек): базовые/литеральные строки
(в т.ч. многострочные `"""···"""`/`'''···'''` с полным набором
escape-последовательностей, включая `\uXXXX`/`\UXXXXXXXX`), целые
(десятичные со знаком, `0x`/`0o`/`0b` без знака, `_`-разделители),
дробные (экспонента, `inf`/`nan` и их `±`-варианты), точечные ключи
(`a.b.c = 1`), вложенные секции (`[a.b.c]`), массивы таблиц (`[[a.b]]`),
инлайн-таблицы (`{ x = 1, y = 2 }`), многострочные массивы, кавычечные
ключи. Сознательно не поддержано (нишевые случаи, не стоящие объёма для
hand-written парсера): дата-время с пробелом вместо `T` между датой и
временем (читается как два отдельных токена — используйте `T`),
bare-ключи, начинающиеся с цифры.

## `тест`: минимальный ассерт-фреймворк

```panos
импорт тест
импорт математика

функ старт() -> Пусто
    пер отчёт = тест.новый_отчёт()
    тест.проверить(отчёт, 2 + 2 == 4, "арифметика сломана")
    тест.равны(отчёт, математика.степень(2, 3), 8, "степень считает неверно")
    тест.напечатать_сводку(отчёт)
    если отчёт.провалено > 0 тогда
        паника("тесты провалены")
    конец
конец
```

`Отчёт` копит пройдено/провалено вместо паники на первом же провале —
`проверить(отчёт, условие, сообщение)` и generic `равны[T]`/`не_равны[T]
(отчёт, a, b, сообщение)` продолжают прогон, `напечатать_сводку(отчёт)`
печатает итог + список провалившихся сообщений. Панику (если она вообще
нужна) вызывающий код добавляет сам — так CI видит полную сводку, а не
только первый провал.

`ожидать_панику(отчёт, причина, сообщение)` — assert "код должен был
запаниковать": в panos нет try/catch, единственный способ поймать панику
— посадить проверяемый код в отдельный процесс (`запусти`) и получить
причину краша через `наблюдать`/`получить_сигнал` (см.
[«Процессы»](./processes.md)):

```panos
функ падающая_операция() -> Пусто
    паника("что-то пошло не так")
конец

функ старт() -> Пусто
    пер отчёт = тест.новый_отчёт()
    пер процесс = запусти падающая_операция()
    наблюдать(процесс)
    пер (_, причина) = получить_сигнал()
    тест.ожидать_панику(отчёт, причина, "падающая_операция должна паниковать")
    тест.напечатать_сводку(отчёт)
конец
```

## `сжатие`

Тонкая обёртка над `core:compress/gzip` (Odin) — DEFLATE/Хаффман не
реализуем в чистом panos (нет битовых операторов, `Массив(T)` — не
байтовый буфер, см. обсуждение архитектуры), поэтому распаковка — core
builtin, а не stdlib-модуль:

```panos
импорт сжатие
импорт фс

функ старт() -> Строка
    выбор фс.прочитать("архив.tar.gz")
        Результат.Успех(данные) ->
            выбор сжатие.разжать_gzip(данные)
                Результат.Успех(текст) -> текст
                Результат.Неудача(ош) -> "ошибка распаковки"
            конец
        Результат.Неудача(ош) -> "не удалось прочитать файл"
    конец
конец
```

`разжать_gzip(данные) -> Результат(Строка, Ошибка)`. Только распаковка —
записи (сжатие на запись) core:compress не поддерживает. Недоступно в
браузере (WASM-сборка не тянет `core:compress/gzip`, тот транзитивно
импортирует `core:os`).

## `архив`: tar

Чистый panos (парсинг/сборка ustar-заголовков) поверх байтовых примитивов
`строки.байт`/`длина_байт`/`срез_байт`/`из_байтов` (см.
[«Строки» → «Байты»](./basic-types.md#байты)) — тот же принцип, что
`кодирование/json`/`кодирование/toml`: сам формат на panos, только
низкоуровневый доступ к байтам — core builtin.

```panos
импорт архив
импорт фс

функ старт() -> Пусто
    пер записи: Массив(архив.Запись) = массив(
        архив.Запись("привет.txt", "Привет, мир!", ложь)
    )
    фс.записать("вывод.tar", архив.собрать(записи))
конец
```

`Запись { имя: Строка, содержимое: Строка, каталог: Булево }`.
`разобрать(данные) -> Массив(Запись)`, `собрать(записи) -> Строка` —
обычные файлы/каталоги, ustar-подмножество полей (без symlink/hardlink/
устройств, GNU длинных имён, PAX-заголовков — такие записи парсятся как
обычные "файлы" с их сырым содержимым, не как метаданные). Проверено на
кросс-совместимость с настоящим `tar` (GNU и BSD, включая архивы с
AppleDouble/PAX-записями от macOS) в обе стороны.

`разобрать_tar_gz(данные) -> Результат(Массив(Запись), Ошибка)` —
удобство: `сжатие.разжать_gzip` + `разобрать` одним вызовом для `.tar.gz`.

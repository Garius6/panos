# Contract: Опкоды и байткод

Новые опкоды добавляются в конец существующего перечисления `Opcode`
(`compiler.odin`), чтобы не смещать существующие числовые значения (это
не сериализованный формат, но правило проще). Всего добавляется четыре
опкода: `Match_Tag`, `Get_Variant_Field`, `Match_Fail`, `Build_Variant`.

## 1. `Match_Tag` (u8 operand)

- **Операнд**: 1 байт — индекс константы в пуле функции, где лежит
  целочисленный тег-ожидание (`f64`, приводится к `int`, повторяет
  соглашение других целочисленных опкодов Panos).
- **Стек до**: [..., subject]
- **Стек после**: [..., subject, bool] — subject сохраняется, поверх кладётся
  результат сравнения.
- **Побочные эффекты**: нет.
- **Использование**: перед каждым ветвлением по варианту. Далее
  `Jump_If_False` уводит на следующий кейс.

## 2. `Get_Variant_Field` (u8 operand)

- **Операнд**: 1 байт — индекс поля варианта (0..N-1).
- **Стек до**: [..., subject]
- **Стек после**: [..., field_value] — subject снимается, значение поля
  кладётся.
- **Побочные эффекты**: нет.
- **Runtime error**: если `variant_field(subject, i)` вернул `ok = false`
  (несовпадение варианта, отсутствие поля) — `fmt.panicf` с русскоязычным
  сообщением, но type checker гарантирует, что этого не случится.

## 3. `Match_Fail` (без операнда)

- **Стек до**: [..., subject]
- **Стек после**: — VM останавливается.
- **Побочные эффекты**: печатает сообщение вида
  `Runtime Error: значение варианта <имя> не покрыто ни одной веткой
  выбора`, где `<имя>` восстанавливается из `variant_tag` + таблицы имён
  (сохраняется в константах функции как строка).
- **Использование**: эмитится в конце цепочки веток, если ветки `_` в
  выражении не было. Type checker обычно предотвращает достижение
  `Match_Fail`, но опкод оставлен как страховка (R7).

## 4. `Build_Variant` (u8 type_name_const, u8 tag, u8 arity)

- **Операнды**: три однобайтовых операнда — индекс константы с
  `type_name: string`, индекс варианта (`tag`), число полей (`arity`).
- **Стек до**: [..., field_0, field_1, ..., field_{arity-1}]
- **Стек после**: [..., ^Variant_Value] — снимает `arity` элементов,
  собирает их в порядке появления, кладёт указатель на новое значение.
- **Побочные эффекты**: одна аллокация `^Variant_Value` и (при
  `arity > 0`) один `[dynamic]Value` под `fields`.
- **Использование**: единый путь генерации конструктора варианта в
  T016/T017 — как для zero-field (`arity = 0`), так и для конструкторов
  с полями. Убирает развилку в T017.

## 5. Общая форма скомпилированного `выбор`

Для выражения `выбор subject Arm1 Arm2 ... Armn конец`:

```
    <compile subject>
    ; для каждой ветки i:
    Match_Tag <tag_const_i>
    Jump_If_False -> LnextArm     ; label
    <bind pattern fields via Get_Variant_Field into locals>
    Pop                            ; удалить subject-копию
    <compile arm body>             ; результат остаётся на стеке
    Jump -> Lend
LnextArm:
    Pop                            ; удалить bool
    ; ... следующая ветка ...
    ; после последней ветки:
    Match_Fail
Lend:
```

Ветка `_`:

- `Match_Tag`/`Jump_If_False` заменяются на безусловный переход в тело:
  сразу `Pop` (subject) → тело.

Ветка `Ident` (биндер без конструктора):

- То же самое, что `_`, только перед `Pop` эмитится `Set_Local` для
  привязки subject-значения к локали биндера.

Ветки с вложенными шаблонами:

- Каждый `Pattern_Constructor` внутри аргумента ветки компилируется
  рекурсивно тем же алгоритмом на копии значения соответствующего поля.
  Требуется дублирование subject через `Get_Variant_Field` +
  сохранение промежуточного значения (используются существующие
  `Set_Local`/`Get_Local` со слотом временной переменной).

## 6. Влияние на существующий байткод

- Не переопределяются существующие опкоды.
- Не меняется представление `Compiled_Function`.
- Пул констант всё ещё индексируется 1 байтом; лимит 255 констант на
  функцию сохраняется. Плановая нагрузка (десятки веток) хорошо в него
  влезает.

# Phase 0: Research — ADT и pattern-matching

Собраны решения по всем открытым точкам, отмеченным в spec.md и plan.md.
Формат каждого пункта: **Decision**, **Rationale**, **Alternatives**.

## R1. Синтаксис объявления ADT

**Decision**: `тип Имя = перечисление Вариант1[(Тип, ...)]; Вариант2 ... конец`.
Разделитель между вариантами — `;` **или** перевод строки (та же логика,
что уже используется у `перечисление`/`структура` для полей и у
`интерфейс` для методов через `consume_semicolon_or_newline`).

**Rationale**:

- Совпадает с существующей стилистикой `тип X = структура ... конец` и
  `тип X = интерфейс ... конец`; парсер уже узнаёт заголовок `тип X =` и
  диспетчеризует по третьему токену.
- Не требует новых ключевых слов и токенов: `перечисление` уже
  зарезервировано в лексере.
- Позволяет писать варианты в одну или несколько строк без ломки правил.

**Alternatives**:

- Формат `Rust-style` с фигурными скобками — язык скобок для блоков не
  использует, ввёл бы диссонанс.
- Форма `| Вариант ...` (ML-style) — потребует нового токена и уводит
  визуально от структурных объявлений.

## R2. Квалификация одноимённых вариантов

**Decision**: Квалификация выполняется точечной формой `ИмяТипа.Вариант`.
`Тип` разрешается как имя типа в текущем скоупе; после точки идёт имя
варианта. В шаблонах `выбор` квалификация допустима симметрично:
`ИмяТипа.Вариант(биндер)`.

**Rationale**:

- Существующий парсер уже разбирает `x.y` как `Property_Expr`. При типе,
  чей значение-конструктор известен резольверу, можно распознать этот же
  синтаксис как квалифицированный конструктор без нового токена.
- Не конфликтует с `::` (модульная квалификация, ключевое `импорт ... как`
  уже занимает эту нишу).

**Alternatives**:

- `Тип::Вариант` — язык уже не использует `::` в исходном тексте (только в
  символических полных именах модулей). Ввод оператора расширяет
  синтаксис без необходимости.
- Обязательная квалификация всегда — увеличит шум в 90% случаев, где имя
  однозначно.

## R3. Внутреннее представление типа ADT

**Decision**: Добавить `Type_Kind.Enum` (значение перечисления `Type_Kind`).
Список вариантов хранить в новом поле `variants: [dynamic]Type_Variant`
структуры `Type`, где

```odin
Type_Variant :: struct {
    name:   string,
    fields: [dynamic]^Type,   // может быть пустым
}
```

**Rationale**:

- Меньшее вмешательство в существующую иерархию `Type`: одно новое
  значение enum + одно новое поле.
- Не требует отдельного модуля с типами. Не создаёт нового ownership'а
  памяти — существующие Type-объекты уже разделяются по указателям.

**Alternatives**:

- Переиспользовать `elements` (уже занят кортежами) — привело бы к
  перегрузке смысла поля и путанице в тайп-чекере.
- Вынести варианты в `map[string]Type_Variant` — потеряется порядок
  объявления, а он значим для сообщений об ошибках и для стабильного
  вывода печати.

## R4. Представление значения ADT в VM

**Decision**: Ввести `Variant_Value :: struct { tag_index: int, fields:
[dynamic]Value }`; добавить `^Variant_Value` в union `Value`. Значение
хранит числовой индекс варианта (соответствует порядку объявления) и
массив полей.

**Rationale**:

- Индекс, а не строка: сравнение тегов — целочисленное, генерируется
  из константы в пуле функции.
- Тип уже связан с описанием вариантов (см. R3), поэтому напечатать имя
  по индексу можно через сохранённое поле имени типа + `variants[i].name`
  во время диагностики (для этого достаточно передавать имя типа как
  константу вместе с тегом).
- Симметрично уже существующим `^Option_Value` / `^Result_Value`, но
  унифицированно.

**Alternatives**:

- Строковый тег — простая, но каждый матч тратит string compare и лишние
  аллокации.
- Уложить в `^Aggregate_Value` — потеряется различие между кортежем и
  вариантом при диагностике и печати.

## R5. Разбор встроенных `Опция(T)` / `Результат(T, E)` (обновлено Q5)

**Decision**: Runtime-представление (`^Option_Value`, `^Result_Value`) НЕ
меняется. Но резольверный слой унифицируется с пользовательскими ADT:

- В prelude резольвера регистрируются четыре `Symbol_Kind.Enum_Variant`:
  `Нет` и `Есть` привязаны к типу `Опция`; `Успех` и `Неудача` —
  к `Результат`. Регистрация происходит один раз при создании корневого
  scope модуля, до разрешения пользовательского кода.
- Type checker наполняет поле `Type.variants` для интернированных
  `Опция(T)`/`Результат(T, E)` реальными `Type_Variant` записями с
  правильными типами полей. Никакой параллельной hardcoded-таблицы для
  match-логики не заводится: exhaustiveness/недостижимость работают
  через тот же путь, что и для пользовательских ADT.
- Компилятор для этих типов эмитит те же новые опкоды. VM для
  `^Option_Value` / `^Result_Value` возвращает тег и поле через чистые
  helper'ы `variant_tag`/`variant_field` — они мапят
  `has_value`/`is_ok` в фиксированный порядок тегов (0=Нет/Успех? см.
  ниже) без изменения структуры значений.

**Порядок тегов встроенных типов** (фиксирован в R5 и в
`variant_tag`):

- `Опция(T)`: `Нет` = 0, `Есть` = 1.
- `Результат(T, E)`: `Успех` = 0, `Неудача` = 1.

**Rationale**:

- Совпадение синтаксиса выражений и шаблонов для встроенных и
  пользовательских ADT — цель Q5. Резольверу не приходится делать
  case-специальный обход шаблонов `выбор`.
- Существующие методы (`.есть()`, `.успех()`, `?`) и все существующие
  тесты продолжают работать: их путь идёт через builtin-функции и
  оператор `?`, а не через resolver-lookup вариантов.
- Значения на стеке не мигрируют; изменения локализованы в prelude и
  небольшом слое `variant_tag`/`variant_field`.

**Alternatives**:

- Hardcoded-таблица в type checker'е (первоначальный вариант R5,
  отброшен Q5) — заводит две модели резолюции.
- Переписать `Опция`/`Результат` целиком через `^Variant_Value` — большой
  blast radius, задевает существующие методы и `?`.
- Требовать квалификации `Опция.Есть(х)` в шаблонах — противоречит
  единству синтаксиса и увеличивает шум в 100% случаев.

## R6. Проверка исчерпываемости и недостижимости (уточнено Q4)

**Decision**: Наивный алгоритм:

1. Собрать список имён вариантов ADT — из `Type.variants` (общий путь
   и для пользовательских, и для `Опция`/`Результат` — см. R5).
2. Пройти по веткам сверху вниз, помечая, какие варианты покрыты. Ветка
   с уже покрытым вариантом (без охран — их в поставке нет) — ошибка
   «недостижимая ветка».
3. Ветка `_` допустима только один раз и только последней. Любая ветка,
   встреченная после `_`, — ошибка «ветка после `_` недостижима» (Q4).
4. При наличии `_` в конце: исчерпываемость выполнена автоматически.
5. Если `_` не встретился и остались непокрытые имена — ошибка с их
   перечислением.

**Rationale**:

- В поставке нет литеральных / охранных / вложенных «покрытий»
  (вложенный конструктор считается покрытым только если внешнее имя
  совпадает и внутренний шаблон — `_` или биндер; иначе тайп-чекер не
  утверждает покрытие). Такой конкретный слепок правил делает алгоритм
  тривиально линейным по числу веток × числу вариантов.
- Явно назван как «наивный», чтобы не путать с полноценной
  matrix-based exhaustiveness (Maranget) — она overkill для текущего
  языка.

**Alternatives**:

- Полный алгоритм с матрицей — переусложнение без пользы для текущих
  шаблонов; см. Simplicity First.
- Отказ от проверки исчерпываемости — противоречит US3 и FR-007.

## R7. Опкоды (обновлено при анализе)

**Decision**: Добавить четыре новых опкода в `Opcode` (`compiler.odin`):

- `Match_Tag` — операнд: 1 байт (индекс константы с ожидаемым тегом-int).
  Вершину стека читает без снятия; кладёт булев результат сравнения
  тегов в стек (для последующего `Jump_If_False`).
- `Get_Variant_Field` — операнд: 1 байт (индекс поля). Снимает вершину
  (значение варианта), кладёт значение i-го поля.
- `Match_Fail` — без операнда. Эмитится в конце цепочки, если не было
  `_`; VM печатает русскоязычное сообщение с именем варианта.
- `Build_Variant` — три операнда u8 (`type_name_const`, `tag`, `arity`).
  Единый путь для всех конструкторов вариантов — zero-field и с полями,
  без развилки на compile-time expansion. Убирает неопределённость
  T017 и позволяет одному опкоду отвечать за создание `^Variant_Value`.

Все прыжки — существующие `Jump_If_False`, `Jump`.

**Rationale**:

- Минимальный набор: тег-компаратор + аксессор + завершающий трап.
- Не ломает существующий encoding `u8`-индексов констант.

**Alternatives**:

- Один суперопкод `Match_Case` (сравнение + условный прыжок) — короче в
  байткоде, но менее композиционен; текущие макросы прыжков (`emit_jump`,
  `patch_jump`) уже дают нужную гибкость.
- Не эмитить `Match_Fail`, полагаясь на type checker: слишком легко
  забыть в будущих расширениях (охраны, литералы); дешёвая страховка.

## R8. Порядок работ и разбиение по стадиям

**Decision**: MVP-инкремент = US1 полностью (декларация, конструктор,
проверка типов, печать) → US2 полностью (сопоставление, биндеры,
базовые ветки) → US3 полностью (исчерпываемость, недостижимость,
`Match_Fail`) → US4 (Option/Result). Внутри каждой истории — один PR/один
commit группировкой «lexer→parser→resolver→type_checker→compiler→vm→
e2e-тест», чтобы каждый шаг оставался работоспособным.

**Rationale**:

- Соответствует существующему правилу из `AGENTS.md`: «минимальные
  вертикальные срезы».
- Каждая история сама по себе — рабочий язык (см. Independent Test
  каждой истории).

**Alternatives**:

- Сначала всё в парсере, потом всё в тайп-чекере и т.д. — нарушает
  вертикальный срез и увеличивает риск лишнего кода.

## R10. Cross-module квалификация варианта (Q3)

**Decision**: Форма `модуль.Тип.Вариант(...)` работает всегда для
экспортированного ADT из импортированного модуля. Форма
`модуль.Вариант(...)` допустима, только если короткое имя варианта
однозначно среди экспортов модуля (т.е. только один экспортированный
ADT содержит вариант с этим именем).

**Rationale**:

- Модульная квалификация уже реализована через `Property_Expr` для
  импортов; тот же путь легко расширяется.
- Форма `модуль.Тип.Вариант(...)` не требует новых токенов и хорошо
  читается.
- Fallback `модуль.Вариант(...)` избегает шума в типичных случаях с
  одним ADT в модуле, но продолжает выдавать понятную ошибку на
  реальном конфликте.

**Alternatives**:

- Только полная форма всегда — избыточно, ломает существующую
  привычку `модуль.имя_функции(...)`.
- Разрешить неквалифицированное `Вариант(...)` через use-подобные
  импорты — вне поставки; расширяет модель имён и требует нового
  синтаксиса.

## R11. Формат печати ADT-значения (Q2)

**Decision**: Встроенная процедура `печать`/`строка` (сериализация в
строку) форматирует ADT-значение как:

- `ИмяВарианта(арг1, арг2, ...)` — для варианта с полями,
- `ИмяВарианта` — для варианта без полей.

Аргументы рекурсивно форматируются существующими правилами `печать`
для их типов (числа, строки, вложенные ADT, `Опция`/`Результат`).

**Rationale**:

- Совпадает с формой вызова конструктора в исходнике: пользователь
  видит то же, что писал.
- Не требует нового API форматирования: реализуется в существующем
  диспетчере `value_to_display_string` (или где сейчас лежит логика
  форматирования для `печать`).

**Alternatives**:

- `ИмяТипа.ИмяВарианта(...)` — избыточно для типичной отладки.
- Числовые теги (`#0`, `#1(3)`) — нечитаемо; отклонено в опросе.
- Запретить прямую печать ADT — усложняет пользовательский тест US1.

## R9. Ошибка при runtime-fallthrough

**Decision**: Формулировка сообщения:
`Runtime Error: значение варианта <имя_варианта или '?'> не покрыто ни
одной веткой выбора`. Инстанс `паника(...)` не переиспользуется, но
механизм остановки VM — тот же `fmt.panicf` (как во всех текущих runtime
error).

**Rationale**:

- Совпадает по стилистике с существующими сообщениями (`Runtime Error:
  индекс массива ...`).
- Не смешивает язык (`паника(...)` в исходном коде) и внутренний трап.

**Alternatives**:

- Полноценное объектное представление паники через `Никогда` — избыточно
  для текущего рантайма.

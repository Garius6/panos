# Feature Specification: ADT и pattern-matching

**Feature Branch**: `001-adt-pattern-matching`
**Created**: 2026-07-07
**Status**: Draft
**Input**: User description: "Заверши реализацию ADT и pattern-matching. Используй функциональный стиль (чистые функции и отсутствие глобальных состояний, если возможно), если возможно"

## Clarifications

### Session 2026-07-07

- Q: Синтаксис квалификации варианта при неоднозначном коротком имени → A: `Тип.Вариант` (используется существующий парсинг `x.y` симметрично в выражениях и шаблонах).
- Q: Формат печати ADT-значения через `печать(...)` → A: `ИмяВарианта(арг, ...)`; вариант без полей печатается как одно имя (`Точка`); аргументы форматируются существующими правилами `печать` для их типов.
- Q: Cross-module использование варианта импортированного ADT → A: полная форма `модуль.Тип.Вариант(...)` (например `ф.Фигура.Круг(3)`); тип-квалификатор обязателен только когда короткое имя неоднозначно в импортированном модуле; иначе допустимо `модуль.Вариант(...)`.
- Q: Позиция ветки `_` в `выбор` → A: `_` допустима только как последняя ветка; любая ветка после `_` — ошибка компиляции (недостижима).
- Q: Как resolver обрабатывает встроенные `Есть`/`Нет`/`Успех`/`Неудача` в шаблонах `выбор` → A: синтаксис совпадает с пользовательскими ADT; варианты `Опция` и `Результат` регистрируются как обычные `Enum_Variant` в prelude до старта программы, доступны и в выражениях, и в шаблонах на общих основаниях.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Объявление ADT (Priority: P1)

Разработчик, пишущий программу на Panos, объявляет собственный алгебраический тип
данных (sum type) с несколькими вариантами. Часть вариантов не несёт полезной
нагрузки, часть — принимает один или несколько типизированных параметров.
Разработчик использует конструкторы этих вариантов в выражениях и хранит их в
переменных так же свободно, как встроенные `Опция(T)`/`Результат(T, E)`.

**Why this priority**: Без пользовательских ADT язык остаётся коллекцией
встроенных sum-типов, а модель домена (состояния, события, деревья) невозможно
описать без искусственных структур и флагов. Это первый шаг, без которого
pattern-matching нечему разбирать.

**Independent Test**: Написать программу на Panos с объявлением вроде
`тип Фигура = перечисление Круг(Число); Квадрат(Число); Точка конец`,
присвоить в переменные `Круг(3)`, `Квадрат(2)`, `Точка` и вывести
их (через `печать` или другой доступный способ). Программа должна пройти
все стадии обработки без ошибок и напечатать значения, однозначно
отражающие вариант и его поля.

**Acceptance Scenarios**:

1. **Given** объявлен ADT с тремя вариантами (без полей, с одним полем, с двумя
   полями), **When** программа создаёт по одному значению каждого варианта и
   передаёт их в `печать`, **Then** программа выполняется до конца; строка
   вывода имеет форму `ИмяВарианта(арг, ...)` для вариантов с полями и
   `ИмяВарианта` для вариантов без полей.
2. **Given** тот же ADT, **When** пользователь передаёт в конструктор аргумент
   несовместимого типа (например `Круг("строка")`), **Then** программа
   отклоняется на стадии проверки типов с сообщением на русском языке, которое
   называет ожидаемый и полученный тип.
3. **Given** объявлены два ADT с одноимённым вариантом (`Круг` в `Фигура` и
   `Круг` в `Иконка`), **When** конструктор используется без квалификации,
   **Then** программа отклоняется на стадии разрешения имён с понятной
   ошибкой; квалифицированное обращение (`Фигура.Круг(1)` либо аналогичный
   выбранный синтаксис) работает корректно.

### User Story 2 — Разбор ADT через `выбор` (Priority: P1)

Разработчик получает значение ADT и разбирает его в выражении `выбор`, обрабатывая
каждый вариант отдельно и получая имена привязанных полей. Значение выражения
`выбор` можно присвоить переменной или использовать как результат функции.

**Why this priority**: ADT без pattern-matching бесполезен: пользователь вынужден
писать вспомогательные предикаты для каждого варианта. Именно `выбор` даёт язык
для реального использования новых типов.

**Independent Test**: Определить функцию `площадь(ф: Фигура) -> Число`, тело
которой состоит из одного выражения `выбор ф ... конец` с ветками
`Круг(р)`, `Квадрат(с)`, `Точка`. Вызвать функцию для каждого варианта и убедиться,
что возвращаемые числа соответствуют формулам.

**Acceptance Scenarios**:

1. **Given** функция возвращает значение выражения `выбор`, чьи ветки покрывают
   все варианты ADT, **When** функция вызвана для каждого варианта, **Then**
   возвращаемое значение соответствует телу совпавшей ветки; выражение имеет
   тип, равный общему супертипу тел веток.
2. **Given** ветка использует поле-биндер (`Круг(р)`), **When** ветка
   выполняется, **Then** внутри тела ветки переменная `р` доступна, имеет тип,
   объявленный у варианта, и не видна за пределами ветки.
3. **Given** используется шаблон-заглушка (`_`), **When** значение не совпало с
   предшествующими ветками, **Then** выполняется тело ветки `_`.
4. **Given** значение содержит вложенный ADT (например
   `Результат(Фигура, Ошибка)`), **When** шаблон вкладывает конструктор в
   конструктор (`Успех(Круг(р))`), **Then** сопоставление проходит и биндер
   `р` доступен внутри ветки.

### User Story 3 — Гарантия исчерпывающего разбора (Priority: P1)

Разработчик добавляет новый вариант в существующий ADT. Все выражения `выбор`,
которые больше не покрывают все варианты, отклоняются на стадии проверки типов
с указанием непокрытых случаев. Ветка `_` явно снимает это требование там, где
пользователь этого хочет.

**Why this priority**: Основная ценность ADT+match — компилятор ловит забытый
случай. Без исчерпывающей проверки язык теряет главную гарантию, ради которой
эти конструкции вводятся.

**Independent Test**: Взять функцию `площадь` из истории 2, добавить новый
вариант в `Фигура` и запустить pipeline. Ожидать отказ с перечислением
непокрытых вариантов. После добавления соответствующей ветки (или `_`) pipeline
проходит.

**Acceptance Scenarios**:

1. **Given** выражение `выбор` покрывает не все варианты своего ADT и не имеет
   шаблона `_`, **When** запускается проверка типов, **Then** проверка
   завершается ошибкой; сообщение перечисляет имена непокрытых вариантов на
   русском.
2. **Given** выражение `выбор` содержит две ветки с одним и тем же вариантом
   (без различающих условий), **When** запускается проверка типов, **Then**
   проверка завершается ошибкой о недостижимой ветке с указанием её позиции.
3. **Given** значение, для которого во время выполнения не нашлось подходящей
   ветки (в силу ошибки, обошедшей проверки), **When** VM исполняет `выбор`,
   **Then** исполнение завершается runtime-ошибкой с осмысленным русскоязычным
   сообщением, а не молчаливым продолжением.

---

### User Story 4 — Разбор встроенных `Опция` и `Результат` (Priority: P2)

Разработчик использует `выбор` для разбора значений встроенных типов
`Опция(T)` и `Результат(T, E)` наравне с пользовательскими ADT, без
необходимости оборачивать их или писать отдельные условия.

**Why this priority**: Эти два типа — самые частые sum-типы в коде на Panos.
Оставить их вне охвата означает поддерживать две разные модели разбора и
подорвать заявленную выгоду. Но встроенные типы уже имеют методы (`.есть()`,
`.успех()`), поэтому история отделена от P1: язык остаётся полезным и без неё.

**Independent Test**: Написать функцию, принимающую `Опция(Число)` и
возвращающую строку, разобрать её `выбор о ... конец` с ветками `Есть(х)` и
`Нет`. То же для `Результат(Строка, Ошибка)`.

**Acceptance Scenarios**:

1. **Given** значение `Опция(T)`, **When** оно разбирается через `выбор` с
   шаблонами `Есть(х)` и `Нет`, **Then** каждая ветка выполняется для
   соответствующего варианта, биндер `х` типизирован как `T`.
2. **Given** значение `Результат(T, E)`, **When** оно разбирается через
   `выбор` с шаблонами `Успех(з)` и `Неудача(о)`, **Then** биндеры имеют типы
   `T` и `E` соответственно.
3. **Given** выражение `выбор` над `Опция(T)`, покрывающее только `Есть(х)`,
   **When** запускается проверка типов, **Then** проверка отклоняет программу
   с указанием, что `Нет` не покрыт.

---

### Edge Cases

- ADT без единого варианта: запрещён на стадии парсинга или проверки типов
  с русскоязычным сообщением.
- Вариант ADT с тем же именем, что и объявленная структура или интерфейс в том
  же модуле: отклоняется на стадии разрешения имён.
- Литеральные шаблоны (числа, булевы, строки) — вне текущей поставки; попытка
  их использовать даёт ошибку парсинга.
- Экспорт ADT: варианты экспортируются вместе с типом (или все, или ни одного),
  чтобы избежать частично видимого sum-типа.
- Рекурсивный ADT (например `тип Дерево = перечисление Лист; Узел(Дерево, Дерево)
  конец`): поддерживается; проверка типов не должна зацикливаться.
- Пустой `выбор` (без веток): отклоняется на стадии парсинга.
- Значение `паника(...)` (`Никогда`) в ветке `выбор`: тип выражения `выбор`
  игнорирует такую ветку при выведении общего типа результата, как это уже
  делается в `если`.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Язык MUST допускать объявление именованного sum-типа синтаксисом,
  основанным на уже зарезервированном ключевом слове `перечисление`, с одним или
  более вариантами; каждый вариант имеет уникальное в пределах типа имя и
  фиксированное число позиционно указанных параметров.
- **FR-002**: Язык MUST допускать объявление вариантов без параметров и вариантов
  с параметрами; типы параметров задаются существующими средствами системы
  типов (примитивы, кортежи, пользовательские типы, сам объявляемый тип
  для рекурсии).
- **FR-003**: Конструктор варианта MUST быть выражением: без параметров — как
  значение, с параметрами — как вызов. Тип результата — объявленный ADT.
- **FR-004**: Проверка типов MUST отклонять программу, если аргумент конструктора
  несовместим с типом соответствующего параметра варианта; сообщение MUST
  называть ожидаемый и полученный тип и позицию.
- **FR-005**: Язык MUST поддерживать выражение `выбор subject ... конец` с одной
  или более ветками; результат `выбор` MUST быть выражением с типом, равным
  общему супертипу тел веток по существующим правилам вывода (аналогично `если`).
- **FR-006**: Ветка MUST состоять из шаблона и тела; поддерживаемые шаблоны:
  `_` (заглушка), идентификатор-биндер, конструктор с вложенными шаблонами.
  Литеральные и tuple-шаблоны — вне текущей поставки.
- **FR-007**: Проверка типов MUST требовать, чтобы все варианты ADT были покрыты
  ветками `выбор`, либо чтобы присутствовала ветка `_`; при нарушении — ошибка
  со списком непокрытых имён.
- **FR-008**: Проверка типов MUST отклонять недостижимые ветки: повторный
  вариант без различающих условий, любая ветка после `_`. `_` допустима
  только как последняя ветка выражения `выбор`.
- **FR-009**: Биндеры внутри шаблона ветки MUST быть видимы только в теле этой
  ветки и MUST иметь типы, вытекающие из объявления соответствующего варианта.
- **FR-010**: Разрешение имён MUST различать одноимённые варианты разных ADT
  через квалификацию именем типа в форме `Тип.Вариант` (симметрично в
  выражениях и в шаблонах `выбор`); неквалифицированное имя варианта
  допустимо только если оно однозначно в текущей области.
- **FR-011**: Разбор значений встроенных `Опция(T)` и `Результат(T, E)` через
  `выбор` MUST работать с шаблонами `Есть(х)`/`Нет` и `Успех(з)`/`Неудача(о)`
  с той же семантикой исчерпываемости и биндеров, что и пользовательские ADT.
  Варианты `Опция` и `Результат` MUST регистрироваться в prelude как обычные
  `Enum_Variant` символы (унифицированный синтаксис с пользовательскими
  ADT); поведение в выражениях `Есть(41)`/`Успех("ok")` сохраняется.
- **FR-012**: Экспорт ADT MUST быть все-или-ничего: экспортированный тип
  экспортирует все свои варианты. Смешанная экспозиция запрещена.
- **FR-012a**: Для импортированного модуля `м` вариант ADT MUST быть
  доступен формой `м.Тип.Вариант(...)`; форма `м.Вариант(...)` допустима,
  только если короткое имя варианта однозначно среди экспортов модуля `м`.
  Ошибка неоднозначности — на стадии разрешения имён.
- **FR-013**: VM MUST выполнять `выбор` за время, линейное относительно числа
  веток в худшем случае; порядок сопоставления — сверху вниз, первая
  совпавшая ветка выполняется.
- **FR-014**: При недостижимом промахе `выбор` во время выполнения VM MUST
  завершить исполнение runtime-ошибкой с русскоязычным сообщением,
  идентифицирующим значение и позицию `выбор`.
- **FR-015**: Все сообщения об ошибках разбора, проверки типов, компиляции и
  выполнения, добавляемые этой фичей, MUST быть на русском языке и следовать
  существующей стилистике диагностики.
- **FR-015a**: Встроенная процедура `печать(...)` MUST форматировать значение
  ADT как `ИмяВарианта(арг, ...)` для вариантов с полями и как `ИмяВарианта`
  для вариантов без полей; аргументы форматируются по существующим правилам
  для их типов. То же применяется к встроенным `Опция(T)`/`Результат(T, E)`
  (`Есть(...)`, `Нет`, `Успех(...)`, `Неудача(...)`).
- **FR-016**: Существующие тесты (`e2e_test.odin`, демонстрационные `.ps`-файлы)
  MUST продолжать проходить; новая функциональность MUST быть покрыта хотя бы
  одним сквозным тестом на каждую пользовательскую историю уровня P1.
- **FR-017**: Внутренняя реализация SHOULD следовать функциональному стилю там,
  где это не ухудшает читаемость: чистые функции стадий (без скрытого
  общего состояния), явная передача контекста через параметры, отсутствие
  новых package-level мутабельных переменных. Отклонения от стиля MUST быть
  оправданы в PLAN.

### Key Entities

- **Объявление ADT (`Enum_Decl`)**: именованный тип с упорядоченным списком
  вариантов и флагом экспорта.
- **Вариант (`Variant_Decl`)**: имя и упорядоченный список типов параметров
  (может быть пустым). Уникален в пределах своего ADT.
- **Значение ADT**: во время исполнения — метка варианта плюс значения его
  параметров; равенство определяется по метке и параметрам.
- **Выражение `выбор` (`Match_Expr`)**: субъект (произвольное выражение) и
  непустой упорядоченный список ветвей.
- **Ветка (`Match_Arm`)**: шаблон и тело (последовательность операторов, чьё
  значение — значение последнего выражения).
- **Шаблон (`Pattern`)**: `_`, биндер, конструктор с вложенными шаблонами.
- **Биндер**: имя, введённое шаблоном, видимое только в теле своей ветки, с
  типом от соответствующего параметра варианта.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% сценариев приёмки историй P1 (US1, US2, US3) проходят как
  автоматические e2e-тесты в `e2e_test.odin`.
- **SC-002**: Программа из ~30 строк на Panos, которая объявляет ADT состояний,
  реализует функцию-обработчик через `выбор` и печатает результат для трёх
  разных входов, выполняется до конца и печатает три ожидаемых значения,
  не требуя изменений в других файлах, кроме собственного.
- **SC-003**: Попытка забыть один из вариантов ADT приводит к отказу проверки
  типов с сообщением, которое читающий разработчик может понять без обращения
  к исходникам транслятора (проверяется на трёх разных типах в тестах).
- **SC-004**: Программы и сценарии, которые собирались и проходили до
  добавления фичи, продолжают собираться и проходить с тем же результатом.
- **SC-005**: Диагностические сообщения новой фичи полностью на русском языке
  (проверка: 0 англоязычных сообщений в тестах на негативные сценарии этой
  фичи).

## Assumptions

- Ключевые слова `перечисление` (объявление ADT) и `выбор` (pattern-matching)
  уже зарезервированы в лексере и не будут переименовываться в рамках этой
  фичи.
- Экспорт вариантов идёт вместе с типом; отдельного экспорта вариантов не
  требуется.
- Поставка ограничивается шаблонами: `_`, биндер, конструктор с вложенными
  шаблонами. Литеральные и tuple-шаблоны, охранные условия и or-шаблоны — вне
  поставки и могут быть добавлены позже.
- Квалификация варианта именем типа выполняется точечной формой
  `Тип.Вариант` (например `Фигура.Круг(3)`) и работает симметрично в
  выражениях и в шаблонах `выбор`. Форма зафиксирована в Q1 сессии
  клэрификации.
- Разбор через `выбор` покрывает встроенные `Опция(T)` и `Результат(T, E)`;
  существующие методы этих типов сохраняются без изменений.
- Функциональный стиль внутренней реализации — рекомендация, а не абсолютное
  требование: там, где чистая функция уступает по ясности локальному
  мутабельному состоянию, PLAN может задокументировать отклонение.
- Runtime-паника при недостижимом промахе `выбор` использует тот же механизм,
  что и `паника(...)`.

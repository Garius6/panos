# Структурный контракт глав документации архитектуры

Эта фича не выставляет API/CLI-контракт в привычном смысле — "контракт"
здесь — обязательная структура, которой должна соответствовать КАЖДАЯ
markdown-глава `docs/src/architecture/`, чтобы:
- проходить SC-003 (ответ на "почему X" всегда со ссылкой на код);
- быть узнаваемой при навигации (SC-001 — читатель знает, где искать ответ,
  не читая всю главу целиком);
- быть проверяемой по чеклисту без формального внешнего тестировщика (см.
  Clarifications spec.md).

Формат ниже — MUST для глав типа "Pipeline"/"Рантайм"/"LSP" (см.
`data-model.md`). Главы "Известные грабли" и "Рецепты" имеют СВОЙ формат
(описан отдельно ниже) — короче, т.к. не объясняют архитектуру с нуля.

## Контракт A: главы Pipeline / Рантайм / LSP

Обязательные секции, в этом порядке:

```markdown
# <Название стадии/подсистемы>

## Что

<Какие структуры данных стадия строит/потребляет — с именами полей и
типами. Минимум одна цитата конкретного файла на абзац с новым фактом.>

## Зачем

<Почему эта стадия/подсистема существует ОТДЕЛЬНО от соседних — какую
проблему решает именно она, а не предыдущая/следующая стадия.>

## Почему так, а не иначе

<Отвергнутая альтернатива, если она зафиксирована в коде (комментарий типа
"вместо X сделали Y, потому что...") или известна автору документации.
Если альтернатива НЕ известна — секция явно пишет "альтернатива не
зафиксирована", а не выдумывает её (см. Edge Cases spec.md).>

## Точки входа для типичной правки

<Таблица или список: "тип изменения" → "файл:функция". Например:
"новый бинарный оператор" → `infix_bp`/`prefix_bp` в `parser.odin`.>

## Известные ограничения / грабли (если есть)

<Ссылка на `known-pitfalls.md`, если у этой стадии/подсистемы есть
подтверждённая ловушка (не дублировать текст, только ссылка + одна строка
контекста).>
```

**Валидация чек-листом** (самопроверка автора перед тем, как считать главу
готовой):

- [ ] Каждый абзац в "Что"/"Зачем"/"Почему так" содержит хотя бы одну
      цитату файла/структуры/функции — общих фраз без ссылки на код нет
      (FR-012)
- [ ] "Почему так" либо называет конкретную отвергнутую альтернативу, либо
      явно пишет "не зафиксировано" — не придумывает обоснование
- [ ] "Точки входа" перечисляет ВСЕ файлы для типичной правки (проверено
      против Acceptance Scenarios соответствующей User Story в spec.md)
- [ ] Контент проверен ЧТЕНИЕМ текущего файла на момент написания, а не
      только памятью/старым комментарием

## Контракт B: глава "Известные грабли"

Один файл, список пунктов. Каждый пункт — фиксированная форма:

```markdown
### <Короткое имя ловушки>

**Симптом**: <что наблюдается — сегфолт, ошибка компиляции, тихий баг>

**Причина**: <конкретный механизм, с цитатой кода/паттерна>

**Безопасный паттерн**: <конкретная замена — код-пример>

**Источник**: <коммит/файл, где впервые обнаружено>
```

**Валидация**: минимум 2 пункта на момент первой публикации (FR-008/SC-004),
каждый — со всеми 4 полями заполненными (не "TBD").

## Контракт C: глава "Рецепт"

```markdown
# Рецепт: <название изменения>

## Файлы по порядку (pipeline order)

<Нумерованный список файлов в порядке прохождения pipeline — от лексера к
VM/LSP, не в произвольном порядке.>

## Шаги

<Для каждого файла из списка выше — что именно добавить/изменить (описание
МЕСТА и ФОРМЫ правки, БЕЗ готового кода-решения — рецепт учит находить
место, не копировать чужой diff).>

## Проверка

<Конкретная команда(ы), подтверждающая успех — `odin test ./core`,
`odin build ./wasm`, ручной JSON-RPC-вызов для LSP-рецептов и т.п.>
```

**Валидация**: "Файлы по порядку" ссылается только на файлы, УЖЕ описанные
в соответствующей главе Pipeline/Рантайм/LSP (рецепт не вводит новых фактов
об архитектуре — только компонует уже задокументированное в
последовательность действий, см. `data-model.md` → зависимости).

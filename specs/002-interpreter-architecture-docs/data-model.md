# Data Model: Внутренняя документация архитектуры интерпретатора

Эта фича не создаёт код и не работает с рантайм-данными интерпретатора —
"данные" здесь — сами документы: markdown-главы и связи между ними. Ниже
описаны сущности из spec.md → Key Entities в терминах конкретных файлов
(структура уже зафиксирована в `plan.md` → Project Structure; здесь —
только атрибуты и связи каждой сущности).

## Сущность: Глава Pipeline

**Представляет**: Одна стадия компиляции (лексер/парсер/резолвер/
тайпчекер/компилятор+VM) или сквозная тема pipeline (generics).

**Атрибуты** (обязательны для КАЖДОЙ главы этого типа, см.
`contracts/chapter-template.md`):
- `что` — какие структуры данных стадия строит/потребляет (с именами полей)
- `зачем` — почему стадия существует отдельно от соседних
- `почему_так` — отвергнутая альтернатива, если известна/зафиксирована
- `точки_входа` — файл+функция для типичной правки (FR-002)
- `ссылки_на_код` — минимум одна цитата конкретного файла/структуры на
  каждое утверждение (FR-012)

**Экземпляры**: `lexer.md`, `parser.md`, `resolver.md`, `type-checker.md`,
`compiler-and-vm.md`, `generics-and-monomorphization.md`

**Связи**: Каждая глава ссылается на СЛЕДУЮЩУЮ стадию pipeline (лексер →
парсер → …) и на любые рецепты, которые её затрагивают (например
`lexer.md`/`parser.md`/`type-checker.md`/`compiler-and-vm.md` все связаны с
`recipes/new-binary-operator.md`).

## Сущность: Глава Рантайм

**Представляет**: Подсистему, работающую на протяжении всего исполнения
программы, а не один проход pipeline — GC, модульная система,
platform-split.

**Атрибуты**: те же 5 полей, что у главы Pipeline (общий контракт, см.
`contracts/chapter-template.md`), плюс для `memory-and-gc.md` — явный
раздел "обязательные инварианты" (GC_Header первым полем и т.п.).

**Экземпляры**: `memory-and-gc.md`, `module-system.md`, `platform-split.md`

**Связи**: `platform-split.md` ссылается на КАЖДУЮ `_native`/`_wasm` пару
файлов, обнаруженную при глубоком проходе (см. research.md); `module-system.md`
связан с `recipes/new-stdlib-module.md`.

## Сущность: Глава LSP

**Представляет**: LSP-сервер как отдельную подсистему — единственная
глава этого типа (не разбита на под-главы, в отличие от Pipeline), т.к.
FR-006/FR-007 описывают её как один связный раздел.

**Атрибуты**: 5 общих полей + обязательный подраздел "известные
архитектурные ограничения" (FR-007) — каждое ограничение сформулировано как
ПРИНЯТЫЙ КОМПРОМИСС, не как баг, с указанием, чего не хватает для снятия.

**Экземпляры**: `lsp.md`

**Связи**: связан с `known-pitfalls.md` (оба сегфолт-кейса найдены именно в
LSP-коде) и с `recipes/new-lsp-method.md`.

---

## Сущность: Глава "Известные грабли"

**Представляет**: Список независимых пунктов-ловушек. В отличие от
остальных сущностей — НЕ делится на подстраницы (один файл, растущий
список), т.к. FR-008 не требует навигационной иерархии для 2 (пока)
пунктов.

**Атрибуты на каждый пункт** (не на файл целиком — список открытый):
- `симптом` — что наблюдается (например: сегфолт, `Unsupported_Type`)
- `корневая_причина` — конкретный механизм (прямая индексация `map[key]` в
  `for`-range на отсутствующем ключе; `json.marshal` не поддерживает
  указатели)
- `безопасный_паттерн` — конкретная замена (`v, ok := m[key]` перед циклом;
  ручная сборка `json.Value` вместо struct с полем-указателем)
- `источник` — где впервые обнаружено (коммит `85b6455` для первого пункта)

**Экземпляры**: `known-pitfalls.md` (единственный файл, минимум 2 пункта —
см. FR-008/SC-004)

**Связи**: оба текущих пункта ссылаются на `lsp.md` (место обнаружения).

## Сущность: Глава "Тулчейн и тестирование"

**Представляет**: Сборку (3 цели), `Justfile`-команды, конвенции тестов
(`run_code`/e2e vs inline-pipeline), ритуал редеплоя LSP-бинарника.

**Атрибуты**: 5 общих полей + явный раздел "чек-лист после правки" (какие
команды прогнать, куда скопировать бинарник — FR-011).

**Экземпляры**: `toolchain-and-testing.md`

**Связи**: связан со ВСЕМИ главами Pipeline/Рантайм/LSP как "что делать
после того, как правка внесена".

## Сущность: Рецепт

**Представляет**: Пошаговую инструкцию для одного типичного изменения —
6 экземпляров (FR-014). В отличие от остальных сущностей, рецепт САМ ПО
СЕБЕ не объясняет архитектуру — он ССЫЛАЕТСЯ на уже написанные главы
Pipeline/Рантайм/LSP и перечисляет порядок действий.

**Атрибуты**:
- `файлы_по_порядку` — упорядоченный (по pipeline) список файлов, которые
  нужно тронуть
- `для_каждого_файла` — что именно добавить/изменить (без вставки готового
  кода — рецепт описывает МЕСТО и ФОРМУ правки, не пишет её за читателя)
- `проверка` — какая команда подтверждает успех (`odin test ./core`,
  `odin build ./wasm`, ручной JSON-RPC вызов для LSP-рецепта и т.п.)

**Экземпляры**: `new-binary-operator.md`, `new-lsp-method.md`,
`new-stdlib-function.md`, `new-ast-node.md`, `new-diagnostic.md`,
`new-stdlib-module.md` (все внутри `recipes/`)

**Связи**: каждый рецепт ЗАВИСИТ от готовности архитектурных глав, которые
он перечисляет (см. `research.md` → таблица готовности) — рецепт не может
быть написан раньше соответствующей главы, иначе он рискует стать общими
словами без ссылки на код (нарушение FR-012).

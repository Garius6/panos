# Research: Внутренняя документация архитектуры интерпретатора

Все `[NEEDS CLARIFICATION]`-маркеры сняты в `/speckit.clarify` (см.
`spec.md` → `## Clarifications`). Ниже — не разрешение неопределённостей
Technical Context (их не осталось), а решения по ПОДХОДУ к написанию
документации плюс инвентаризация, что уже подтверждено чтением кода в этой
сессии, а что требует отдельного глубокого прохода на этапе задач.

## Decision 1: Расположение и формат

**Decision**: Новый раздел существующего mdBook — `docs/src/architecture/`,
с добавлением записи в `docs/src/SUMMARY.md` рядом с существующими разделами
"Начало работы" и "Язык".

**Rationale**: Решено пользователем в `/speckit.specify` (см. Assumptions
spec.md). У проекта уже есть работающий mdBook-пайплайн (`docs/book.toml`,
`language = "ru"`) — переиспользование не требует новой инфраструктуры.
`AGENTS.md` явно исключён — он короткий файл конвенций для LLM-агентов, а не
место для многостраничного архитектурного объяснения.

**Alternatives considered**:
- Корневой `ARCHITECTURE.md` — отклонён: не даёт межстраничной навигации/
  поиска, которые уже есть у mdBook; создаёт вторую систему документации
  рядом с существующей.
- Расширение `AGENTS.md` — отклонён пользователем явно: `AGENTS.md`
  выполняет другую роль (конвенции для агентов), смешивание с глубоким
  архитектурным объяснением сделает оба хуже.

## Decision 2: Гранулярность файлов

**Decision**: Один markdown-файл на тему, 1:1 с Key Entities/FR спеки (см.
`plan.md` → Project Structure). Единственное сознательное объединение —
`compiler-and-vm.md` (компилятор и VM документируются вместе, т.к. типичное
изменение — новый opcode — трогает оба файла как одну операцию).

**Rationale**: Simplicity First (конституция, принцип II) — гранулярность
не изобретается автором документации, а прямо считывается из уже
согласованного списка FR/Key Entities. Более крупные "тематические" главы
(например один файл "Frontend" на лексер+парсер+резолвер) хуже проходят
SC-001 ("находит файл(ы) за время чтения документации") — читателю пришлось
бы проматывать несвязанные разделы.

**Alternatives considered**:
- Один большой `ARCHITECTURE.md` на всё — отклонён: 274KB одного только
  `core/type_cheker.odin` намекает на объём материала; один файл был бы
  практически непригоден для навигации и точечного обновления (см.
  Assumptions spec.md: "точечное обновление должно быть дешёвым").
- Файл на каждую Odin-структуру/функцию (максимальная гранулярность) —
  отклонён: избыточно, увеличивает число переходов между страницами без
  выигрыша в findability.

## Decision 3: Источник контента

**Decision**: Контент пишется по РЕЗУЛЬТАТАМ ПРЯМОГО ЧТЕНИЯ актуальных
файлов на момент написания каждой главы, а не по памяти этой чат-сессии.
Факты, известные ТОЛЬКО из истории сессии/git (не зафиксированные явно в
коде), переносятся в документацию явным текстом с пометкой источника (см.
Edge Cases spec.md).

**Rationale**: Явное требование пользователя ("конкретика: что, зачем и
почему", FR-012) + Edge Case spec.md про устаревшие комментарии — код мог
измениться после того, как соответствующий комментарий был написан.

**Alternatives considered**: Пересказ существующих код-комментариев без
повторной проверки — отклонён: не гарантирует актуальность (Edge Cases
spec.md явно предупреждает об этом сценарии).

## Decision 4: Проверка качества документации

**Decision**: Двухчастная проверка — (1) `mdbook build` без ошибок
(структурная валидность, ссылки резолвятся); (2) самопроверка автором по
чеклисту "structural contract" (см. `contracts/chapter-template.md`) для
каждой главы, без формального внешнего тестировщика (см. Clarifications
spec.md, вопрос 1).

**Rationale**: SC-002/SC-003 уже переформулированы в `/speckit.clarify` как
самооценочные критерии, не формальный usability-тест с наймом человека.

**Alternatives considered**: Формальный usability-тест с реальным
тестировщиком на каждый User Story — отклонён пользователем явно в
`/speckit.clarify` (вариант B).

## Инвентаризация: что уже подтверждено, что требует глубокого прохода

Ниже — состояние знаний на момент планирования (не окончательный контент
глав). Главы, помеченные "требует глубокого прохода", нельзя писать только
по заголовочным комментариям файла — нужен полный проход исходника на
этапе задач (`/speckit.tasks` → отдельная задача на главу).

| Глава | Ключевые файлы | Статус на момент планирования |
|---|---|---|
| `lexer.md` | `core/lexer.odin`, `core/token.odin` | Подтверждено частично (структура `Token`/`Span` видена); требует глубокого прохода по `lookup_ident`/правилам токенизации |
| `parser.md` | `core/parser.odin` (102KB) | Подтверждено частично (Pratt-parser, `infix_bp`/`prefix_bp`, precedence rescale по памяти сессии); требует глубокого прохода — файл большой |
| `resolver.md` | `core/resolver.odin` (57KB) | Хорошо подтверждено — `Symbol_Store`, `Symbol_Id`, `graph.symbol_store`, `node_symbols`, `func_args` уже прочитаны и процитированы в spec.md |
| `type-checker.md` | `core/type_cheker.odin` (274KB, САМЫЙ большой файл) | ТРЕБУЕТ глубокого прохода — на момент планирования подтверждена только структура `Type`/`Type_Kind` и `report()`; основная масса логики (unification, generic instantiation, bounded traits) не вычитана построчно |
| `compiler-and-vm.md` | `core/compiler.odin` (72KB), `core/vm.odin` (93KB) | Подтверждена структура `Opcode :: enum u8` и факт GC_Header-встраивания (комментарий в начале compiler.odin); ТРЕБУЕТ глубокого прохода по полному списку opcode'ов и их VM-обработчикам |
| `generics-and-monomorphization.md` | `core/monomorphize.odin`, `core/ast_clone.odin` | Хорошо подтверждено — причина клонирования (конфликт за `ctx.node_types` при повторной типизации того же узла) уже вычитана и процитирована в spec.md |
| `memory-and-gc.md` | `core/gc.odin` (25KB) | Подтверждён только факт (GC_Header первым полем, wasm не имеет `heap_allocator()`) из заголовочного комментария; ТРЕБУЕТ глубокого прохода по алгоритму сборки (mark/sweep? generational? — не установлено) |
| `module-system.md` | `core/module_loader.odin`, `core/resolver.odin` (`resolve_import_path`) | Частично подтверждено (`Module_Graph`, `load_module_recursive`, native vs `std/`-файловые модули, порядок резолва путей в `resolver_import_native.odin`) |
| `platform-split.md` | `*_native.odin`/`*_wasm.odin` пары, `wasm/main.odin` | Подтверждён общий принцип (`#+build`, wasm без `core:os`); ТРЕБУЕТ перечислить ВСЕ такие пары файлов явно (не найдено исчерпывающе в этой сессии) |
| `lsp.md` | `lsp/*.odin` | Хорошо подтверждено — почти весь материал этой сессии: `LSP_Document`, `Resolver_Ctx`/`Type_Ctx` reuse, cross-document merge, известные ограничения rename/references |
| `known-pitfalls.md` | — | Полностью готово: 2 пункта уже точно процитированы в spec.md (FR-008) |
| `toolchain-and-testing.md` | `Justfile`, `core/pipeline.odin`, `core/e2e_test.odin` | Частично подтверждено (`run_code`, inline-pipeline vs file-based graph); список редеплой-локаций LSP-бинарника известен из истории сессии, нужно перепроверить актуальность путей |
| `recipes/*.md` (6 файлов) | пересекает все вышеперечисленные | Каждый рецепт наследует статус "требует глубокого прохода" от файлов, которые он перечисляет — рецепт нельзя написать раньше соответствующей архитектурной главы |

**Вывод для Phase 2 (`/speckit.tasks`)**: главы `resolver.md`,
`generics-and-monomorphization.md`, `lsp.md`, `known-pitfalls.md` можно
писать с высокой уверенностью прямо сейчас. Главы `type-checker.md` и
`compiler-and-vm.md` (два самых больших и наименее вычитанных файла) должны
получить отдельную под-задачу "прочитать файл целиком" ПЕРЕД написанием
текста — не в этой сессии смешивать чтение и написание, чтобы не рисковать
общими фразами вместо конкретики (нарушение FR-012).

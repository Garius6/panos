## User Scenarios & Testing *(mandatory)*

### User Story 1 — Изменение стадии pipeline без помощи LLM (Priority: P1)

Мейнтейнер panos хочет внести изменение в одну из стадий компиляции —
лексер, парсер, резолвер, тайпчекер, компилятор или VM (`core/lexer.odin`
… `core/vm.odin`, см. `AGENTS.md`). Он открывает документацию, находит
раздел про нужную стадию, понимает: какие структуры данных стадия строит и
потребляет, какие инварианты обязаны выполняться на входе/выходе, почему
стадия устроена именно так (а не иначе — какая альтернатива была отвергнута
и почему), и вносит изменение сам, не читая исходники "с нуля" и не
спрашивая LLM.

**Why this priority**: Без этого документация не решает заявленную цель
запроса — самостоятельные правки без LLM возможны только если
задокументирован именно pipeline, ядро интерпретатора.

**Independent Test**: Самопроверка автором документации — для конкретной
изолированной правки (например: добавить новый бинарный оператор, новый вид
литерала, новую diagnostic-проверку в резолвере) убедиться, что раздел
документации по нужной стадии даёт всё необходимое для её внесения
(компилируется, проходит `odin test ./core`), опираясь только на текст
документации, без обращения к LLM и без чтения несвязанных файлов.
Формальный внешний тестировщик не требуется (см. Clarifications).

**Acceptance Scenarios**:

1. **Given** документация описывает стадию резолвера, **When** читатель ищет
   ответ на вопрос "почему `Symbol_Store` — общий указатель на весь граф
   модулей, а не по одному на модуль", **Then** документация даёт прямой
   ответ со ссылкой на конкретный файл/структуру (`core/resolver.odin`,
   `graph.symbol_store`), а не общее описание "резолвер связывает имена".
2. **Given** документация описывает VM/компилятор, **When** читатель ищет,
   как добавить новый opcode, **Then** документация перечисляет ВСЕ места,
   которые нужно тронуть (объявление в `compiler.odin`, обработчик в
   `vm.odin`, при необходимости — GC-эффекты в `gc.odin`), а не только одно
   из них.
3. **Given** документация описывает generic'и, **When** читатель ищет
   объяснение мономорфизации (`core/monomorphize.odin`, `core/ast_clone.odin`),
   **Then** документация объясняет КОНКРЕТНУЮ проблему, которую решает
   клонирование AST (типизация одного и того же узла с разными
   type-параметрами перезаписывала бы `ctx.node_types` предыдущей
   инстанциации), а не просто "generic'и мономорфизируются".

### User Story 2 — Изменение LSP-сервера без помощи LLM (Priority: P2)

Мейнтейнер хочет добавить/исправить фичу LSP-сервера (`lsp/*.odin`) —
например новый LSP-метод или починку существующего хендлера. Он находит в
документации раздел про LSP: как `panos-lsp` переиспользует `core` (граф
модулей на документ, а не общий на проект), какие структуры общие с ядром
интерпретатора (`Resolver_Ctx`, `Symbol_Store`), какие известные
архитектурные ограничения уже приняты сознательно (например: `Symbol_Id` не
сравним между графами разных открытых документов; `doc.graph` строится
только из прямых импортов документа, не от reverse-dependents) — и вносит
правку, опираясь на эти факты, а не переоткрывая их заново.

**Why this priority**: LSP — второй по объёму рабочий пул кода в репозитории
после `core`; без документации архитектурных решений здесь мейнтейнер
рискует повторно наткнуться на уже известные и обойдённые ограничения (или
на уже найденный и исправленный класс багов — прямая индексация `map[key]`
внутри `for`-range на отсутствующем ключе, сегфолт, см. коммит `85b6455`).

**Independent Test**: Самопроверка автором документации — для конкретного
нового простого LSP-метода по образцу существующих (например
`textDocument/…`, переиспользующего уже посчитанные `Resolver_Ctx`/
`Type_Ctx`) убедиться, что раздел про LSP даёт всё необходимое для
реализации и регистрации (capability + dispatch) без повторного "открытия"
уже задокументированных ограничений/грабель. Формальный внешний
тестировщик не требуется (см. Clarifications).

**Acceptance Scenarios**:

1. **Given** документация описывает `LSP_Document`, **When** читатель ищет
   ответ на вопрос "почему `res_ctx` — поле-значение, а не указатель",
   **Then** документация объясняет практическое следствие (нужно брать
   `&doc.res_ctx`, чтобы передать в функции `core`, ожидающие указатель).
2. **Given** документация описывает известные баги/грабли, **When** читатель
   собирается написать `for x in someMap[key]` внутри LSP-хендлера,
   **Then** документация явно предупреждает о сегфолте на отсутствующем
   ключе и показывает безопасную форму (`v, ok := m[key]` перед циклом).
3. **Given** документация описывает межфайловые ограничения rename/references,
   **Then** документация называет точную границу (видит только СЕЙЧАС
   ОТКРЫТЫЕ документы, не весь проект на диске) и объясняет, чего не хватает
   для снятия ограничения (project-wide reverse-import индекс).

### User Story 3 — Изменение стандартной библиотеки и рантайм-подсистем (Priority: P3)

Мейнтейнер хочет добавить функцию в стандартную библиотеку (`std/*.ps`) или
поменять поведение рантайм-подсистемы — сборщик мусора (`core/gc.odin`),
загрузку модулей (`core/module_loader.odin`), FFI (`core/ffi_bindings.odin`,
`core/vm_ffi_native.odin`). Документация объясняет, какие инварианты рантайма
нельзя нарушать (например: все heap-managed варианты `Value` встраивают
`GC_Header` первым полем — почему это обязательно), и как устроена граница
platform-specific кода (`#+build` теги, `_native`/`_wasm` пары файлов).

**Why this priority**: Ниже приоритетом, чем pipeline и LSP, потому что
затрагивается реже, но без документации GC-инвариантов и platform-split
неверная правка здесь — самый дорогой класс ошибок (падение на конкретной
платформе, memory corruption).

**Independent Test**: Самопроверка автором документации — для конкретной
новой экспортируемой функции в существующем модуле `std/*.ps` ИЛИ
builtin-функции в VM убедиться, что документация даёт всё необходимое для
соблюдения GC/platform-инвариантов без падений `odin build ./wasm` и
`odin build .`. Формальный внешний тестировщик не требуется (см.
Clarifications).

**Acceptance Scenarios**:

1. **Given** документация описывает GC, **When** читатель добавляет новый
   heap-managed вариант `Value`, **Then** документация объясняет обязательное
   требование (первое поле — `GC_Header`) и почему его нарушение ломает сборщик.
2. **Given** документация описывает platform-split, **When** читатель видит
   пару файлов вроде `module_loader_native.odin`/`module_loader_wasm.odin`,
   **Then** документация объясняет правило: WASM не имеет ФС/`os.*` — импорт
   `core:os` падает compile-time panic'ом под `js_wasm32`.

### Edge Cases

- Что делать, если решение в коде задокументировано неполно (комментарий в
  коде объясняет "что", но не "почему")? Документация должна явно отмечать
  такие пробелы, а не заполнять их предположениями автора документации.
- Что если архитектурное решение со временем устарело (например ограничение
  снято в более поздней правке, а комментарий в коде не обновлён)?
  Документация должна отражать ТЕКУЩЕЕ состояние кода на момент написания,
  проверенное чтением актуальных файлов, а не пересказывать старые
  комментарии без проверки.
- Что делать с решениями, зафиксированными только в истории git/сессий
  чата, но не в коде и не в `AGENTS.md`/`specs/`? Такие решения должны быть
  перенесены в документацию явным текстом с указанием, что источник —
  практика реализации, а не код-комментарий.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Документация ДОЛЖНА описывать каждую стадию pipeline
  (`lexer.odin`, `parser.odin`, `resolver.odin`, `type_cheker.odin`,
  `compiler.odin`, `vm.odin`) с ответом на три вопроса по каждой: что стадия
  строит/потребляет (конкретные структуры данных и их поля), зачем эта
  стадия существует отдельно от соседних, почему она устроена именно так
  (с указанием отвергнутой альтернативы, если она зафиксирована в коде или
  известна автору документации).
- **FR-002**: Документация ДОЛЖНА перечислять для каждой стадии, В КАКИХ
  ФАЙЛАХ и функциях искать точку входа для типичной правки (например: "новый
  бинарный оператор — `infix_bp`/`prefix_bp` в `parser.odin`, ветка в
  `infer_binary_expr` в `type_cheker.odin`, opcode в `compiler.odin` и
  `vm.odin`").
- **FR-003**: Документация ДОЛЖНА объяснять устройство generic'ов:
  bounded traits, мономорфизация (`core/monomorphize.odin`), клонирование
  AST (`core/ast_clone.odin`) — с конкретной причиной клонирования (типизация
  одного узла с разными type-параметрами конфликтует за `ctx.node_types`).
- **FR-004**: Документация ДОЛЖНА объяснять модель памяти рантайма: GC
  (`core/gc.odin`) — какие `Value`-варианты управляются GC, обязательный
  инвариант `GC_Header` первым полем, и почему `runtime.heap_allocator()` не
  годится под wasm.
- **FR-005**: Документация ДОЛЖНА объяснять систему модулей: `Module_Graph`,
  разрешение путей импорта (`resolve_import_path`), встроенные vs
  файловые (`std/*.ps`) модули, порядок загрузки (`load_module_recursive`).
- **FR-006**: Документация ДОЛЖНА описывать LSP-подсистему (`lsp/*.odin`)
  отдельным разделом: связь с `core` (какие структуры переиспользуются
  напрямую — `Resolver_Ctx`, `Type_Ctx`, `Symbol_Store`), модель документа
  (`LSP_Document`, `Module_Graph` на документ, а не на проект), протокольный
  слой (`lsp/protocol/lsp_types.odin` — автогенерирован, не редактируется
  вручную).
- **FR-007**: Документация ДОЛЖНА явно перечислять известные архитектурные
  ограничения LSP как принятые компромиссы (не как баги): rename/references
  видят только текущий открытый документ + другие открытые документы, не
  весь проект на диске; `Symbol_Id` не сравним между графами разных
  документов (сравнение объявлений идёт по паре путь+byte-span).
- **FR-008**: Документация ДОЛЖНА содержать раздел "известные грабли" минимум
  с двумя подтверждёнными живым тестом пунктами: (1) прямая индексация
  `map[K][dynamic]V` внутри `for`-range на отсутствующем ключе приводит к
  сегфолту — безопасная форма `v, ok := m[key]` перед циклом; (2)
  `core:encoding/json.marshal` не поддерживает поля-указатели
  (`Unsupported_Type`) — обнаружено на `proto.SelectionRange.parent`
  (`^SelectionRange`), обходной путь — сборка ответа вручную как дерево
  `json.Value` вместо маршалинга структуры с полем-указателем.
- **FR-009**: Документация ДОЛЖНА объяснять platform-split (`_native`/`_wasm`
  пары файлов, `#+build` теги) и причину: у WASM нет ФС и `core:os` падает
  compile-time panic'ом.
- **FR-010**: Документация ДОЛЖНА описывать конвенции тестирования: e2e-тесты
  через `run_code` (`core/pipeline.odin`, `core/e2e_test.odin`), отличие
  inline-pipeline (без файлового I/O) от полного `Module_Graph`-пайплайна с
  диска, и почему часть LSP-тестов не может использовать file-based std
  модули (нет полной загрузки графа в изолированном тестовом пути).
- **FR-011**: Документация ДОЛЖНА описывать сборку/тулчейн: три сборочных
  цели (`odin build .`, `odin build ./lsp`, wasm), `Justfile`-команды,
  ритуал редеплоя LSP-бинарника (все известные места установки).
- **FR-012**: Каждое архитектурное решение в документации ДОЛЖНО быть
  привязано к конкретному файлу/структуре/функции — общие фразы без ссылки
  на код запрещены (соответствует явному требованию пользователя).
- **FR-013**: Документация ДОЛЖНА жить как новый раздел существующего mdBook
  (`docs/src/`, например `docs/src/architecture/`), отдельно от раздела
  `language/` (пользовательская документация языка) — не в `AGENTS.md` и не
  как отдельный набор файлов вне `docs/`.
- **FR-014**: Документация ДОЛЖНА включать практические "рецепты" —
  пошаговые инструкции для типичных изменений, минимум 6: (1) новый бинарный
  оператор/литерал; (2) новый LSP-метод по образцу существующих; (3) новая
  экспортируемая функция стандартной библиотеки в существующем модуле;
  (4) новый вид AST-узла (выражение или statement); (5) новая
  diagnostic-проверка (резолвер или тайпчекер); (6) новый stdlib-модуль
  целиком (`std/*.ps`, а не функция в существующем) — каждый рецепт
  перечисляет ВСЕ файлы, которые нужно тронуть, в порядке следования
  pipeline. Список открытый — дополнения новыми рецептами в будущем не
  требуют пересмотра этой спеки.

### Key Entities

- **Раздел документации "Pipeline"**: главы по каждой стадии компиляции
  (лексер → парсер → резолвер → тайпчекер → компилятор → VM), с общей
  структурой "что/зачем/почему" на главу.
- **Раздел документации "Рантайм"**: GC, модель `Value`, FFI, модульная
  система, platform-split.
- **Раздел документации "LSP"**: модель документа/графа, переиспользуемые
  структуры ядра, протокольный слой, известные ограничения.
- **Раздел документации "Известные грабли"**: список подтверждённых багов/
  ловушек с безопасным паттерном для каждой (растёт со временем, не
  статичен).
- **Раздел документации "Тулчейн и тестирование"**: сборка, тесты, редеплой.
- **Раздел документации "Рецепты"**: пошаговые инструкции для 6 типичных
  изменений (новый оператор, новый LSP-метод, новая stdlib-функция, новый
  вид AST-узла, новая diagnostic-проверка, новый stdlib-модуль целиком),
  каждая — с полным списком затрагиваемых файлов в порядке pipeline; список
  открытый, растёт со временем.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Мейнтейнер, ранее не работавший с конкретной стадией pipeline,
  находит файл(ы) и функцию(и) для типичной точечной правки за время
  чтения документации (без чтения исходников "с нуля" и без вопроса к LLM).
- **SC-002**: Автор документации, самостоятельно проверяя каждый раздел по
  критерию "справился бы новичок, имея только этот текст" (без формального
  внешнего тестировщика — см. Clarifications), подтверждает для минимум 2
  из 3 независимых точечных правок (по одной на каждый из User Story 1/2/3),
  что раздел документации достаточен для правки с первой попытки без
  обращения к LLM.
- **SC-003**: На вопрос "почему X устроено так, а не иначе" (для любого из
  решений, перечисленных в Functional Requirements) документация даёт ответ
  со ссылкой на конкретный файл/структуру в 100% случаев — общих фраз без
  привязки к коду быть не должно.
- **SC-004**: Раздел "известные грабли" содержит минимум 2 задокументированные
  ловушки с воспроизводимым сценарием и безопасным паттерном для каждой
  (сегфолт на `map[key]` в `for`-range и `json.marshal`/поля-указатели,
  FR-008) на момент первой публикации документации.

## Assumptions

- Аудитория документации — мейнтейнер panos с уровнем знаний, достаточным
  для чтения кода на Odin, но без контекста конкретных решений этого
  репозитория (то есть документация не обучает Odin/языкам компиляции с
  нуля, а объясняет решения ИМЕННО panos).
- Документация описывает АКТУАЛЬНОЕ состояние кода на момент написания;
  поддержание её в синхронизации с будущими изменениями — отдельная
  практика (не предмет этой фичи), но структура документации (разбиение на
  файлы по подсистемам) должна делать точечное обновление дешёвым.
- Существующая пользовательская документация языка (`docs/src/language/*.md`,
  синтаксис/семантика panos-программ) НЕ входит в объём этой фичи — предмет
  этой фичи — ВНУТРЕННЯЯ архитектура интерпретатора для мейнтейнеров, не
  синтаксис языка для его пользователей.
- Документация живёт как новый раздел существующего mdBook (`docs/book.toml`,
  `docs/src/`) — рядом с разделом `language/`, но отдельным разделом
  (например `docs/src/architecture/`), НЕ смешиваясь с пользовательской
  документацией языка. `AGENTS.md` не расширяется — он остаётся коротким
  файлом конвенций для LLM-агентов, не местом для глубокого архитектурного
  объяснения.
- LSP-подсистема (`lsp/*.odin`) входит в объём как полноценный раздел
  (User Story 2 в полном объёме, не вырезается).
- Документация включает практические "рецепты" (минимум 6, см. FR-014) в
  дополнение к описанию существующей архитектуры — не только референс "что
  уже есть", но и прямая инструкция "как это менять".

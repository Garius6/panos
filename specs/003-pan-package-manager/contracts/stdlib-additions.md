# Contract: новые builtin'ы в core (prerequisite для pan)

**Feature**: `003-pan-package-manager`

Эти три примитива — единственное изменение в `core/` (Odin), которое тянет за
собой фича (research.md п.8, п.11). Всё остальное — panos-код в
`../panosiki/pan/`. Изменения — строго additive: новые procs в существующих
нативных модулях `ос`/`фс`, без изменения поведения уже существующих функций.

## `ос.выполнить`

**Сигнатура (panos)**:

```
функ выполнить(программа: Строка, аргументы: Массив(Строка), рабочая_директория: Строка) -> Результат(РезультатПроцесса, Ошибка)

тип РезультатПроцесса = структура
    код_завершения: Число
    stdout: Строка
    stderr: Строка
конец
```

**Поведение**:
- Спавнит `программа` как дочерний процесс с аргументами `аргументы`
  (позиционные, без shell-интерполяции — не через `sh -c`, чтобы не тянуть
  shell-инъекции при передаче URL/путей из манифеста).
- Рабочая директория дочернего процесса — ровно `рабочая_директория`
  (абсолютный или относительный путь), не CWD родительского процесса.
- Блокирующий вызов: возвращается после завершения дочернего процесса
  (внутри VM это допустимо синхронно для CLI-инструмента pan — в отличие от
  actor-model non-blocking I/O описанного в `docs/src/architecture/compiler-and-vm.md`,
  которое актуально для длительных программ на VM, а не для короткоживущего
  CLI-процесса).
- `код_завершения` — реальный exit code процесса; `stdout`/`stderr` — вся
  собранная разделённая по потокам вывод.
- Если `программа` не найдена в `PATH` или не может быть запущена — возвращает
  `Ошибка`, не `Результат` с кодом завершения.

**Используется в**: `pan add`/`pan install` (git clone/checkout),
`pan run` (спавн `panos <точка_входа>`).

**Реализация**: native-only (аналогично `resolver_import_native.odin` —
`#+build !js`, недоступен в wasm-сборке; wasm-версия pan не предусмотрена
данной фичей).

## Directory-ops в `фс`

**Сигнатуры (panos)**:

```
функ создать_директорию(путь: Строка) -> Результат(Число, Ошибка)   // рекурсивно, аналог mkdir -p
функ список_директории(путь: Строка) -> Результат(Массив(Строка), Ошибка)
функ удалить_директорию(путь: Строка) -> Результат(Число, Ошибка)   // рекурсивно, аналог rm -rf
```

Возвращаемое `Число` в двух первых — не несёт смысла (всегда `0` при успехе),
по той же причине, что `фс.записать` возвращает байты, а не `Результат(Пусто,
...)`: `Пусто` в этой системе типов — только тип возврата, не конструируемое
значение (см. `core/parser.odin:195`), так что `Результат(Пусто, Ошибка)`
пришлось бы собирать без реального значения — проще переиспользовать уже
принятую в стандартной библиотеке конвенцию с `Число`.

**Поведение**:
- `создать_директорию`: создаёт директорию и все недостающие родительские;
  успех и если директория уже существует (идемпотентно).
- `список_директории`: возвращает имена непосредственных элементов директории
  (файлы и поддиректории на один уровень, без рекурсии) — этого достаточно для
  обхода дерева зависимостей самим panos-кодом pan через рекурсивные вызовы.
- `удалить_директорию`: рекурсивно удаляет директорию со всем содержимым;
  успех, если директории не существует (идемпотентно) — нужно для `pan remove`
  (очистка более не нужных записей в `модули/`) и работы с кэшем.

**Используется в**: раскладка `модули/` и локального кэша (`pan add`/
`pan install`), очистка `модули/` при `pan remove`.

**Реализация**: native-only, как и `ос.выполнить` — расширяет уже
существующий модуль `фс` (`core/stdlib.odin:209-213`), не создаёт новый
модуль.

## `ос.завершить`

**Сигнатура (panos)**:

```
функ завершить(код: Число) -> Никогда
```

**Поведение**:
- Немедленно завершает ТЕКУЩИЙ процесс (`os.exit`, Odin-сигнатура `-> !`,
  никогда не возвращается) с переданным кодом завершения.
- Статический тип возврата — `Никогда` (`TY_NEVER`, тот же тип, что у
  `паника`), поэтому вызов допустим как значение любой ветки/tail-выражения
  независимо от ожидаемого типа контекста.

**Используется в**: `pan run` — единственный способ пробросить дочернему
`panos`-процессу его РЕАЛЬНЫЙ exit code (0 или нет) в качестве exit code
самого `pan`, вместо приближения через `паника` (которая давала ненулевой,
но не тот же самый код).

**Реализация**: native-only (`core/vm_process_native.odin`); в WASM-сборке
паникует как недоступное (браузер не даёт скрипту явно завершать свой
процесс/страницу таким образом) — тот же паттерн, что `ос.выполнить`.

**Почему добавлен уже после первичной реализации фичи**: изначально пункт
"точный exit code" был принят как известное ограничение v1 (не было в
исходном списке из двух builtin'ов, согласованном как Option A). Пользователь
явно попросил закрыть именно этот гэп — добавление одного builtin'а вместо
костыля на panos-уровне (там просто неоткуда взять примитив "завершить
процесс с кодом").

## Гейт для Constitution Check

Все три примитива — минимально необходимая, additive поверхность (Principle II
Simplicity First: не добавляется ничего сверх того, что реально нужно
конкретным сценариям выше — ни прав доступа, ни асинхронности, ни
кросс-платформенных абстракций сверх того, что даёт сам Odin `core:os`).

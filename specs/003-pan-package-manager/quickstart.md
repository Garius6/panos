# Quickstart: pan

**Feature**: `003-pan-package-manager`

Сценарий ниже прогоняет все четыре User Story из `spec.md` подряд — это же
основа для acceptance-тестов при реализации.

```bash
mkdir мой-проект && cd мой-проект

# US1 — инициализация
panos /путь/до/pan/start.ps init
# → появились pan.toml (имя=мой-проект, версия=0.1.0) и start.ps

# US2 — добавление зависимости
panos /путь/до/pan/start.ps add https://github.com/panosiki/pan.git ^1.0.0
# → pan.toml и pan.lock обновлены, модули/pan.ps на месте

# US3 — воспроизводимая установка (например, после git clone проекта коллегой)
rm -rf модули ~/.pan/cache
panos /путь/до/pan/start.ps install
# → все зависимости из pan.lock получены заново, сами версии не пересчитаны

# US4 — запуск
panos /путь/до/pan/start.ps run
# → эквивалентно тому, что дал бы прямой `panos start.ps`, но с гарантированно
#   верным резолвом модули/ независимо от текущей поддиректории

cd некоторая/вложенная/поддиректория
panos /путь/до/pan/start.ps run
# → тот же результат (SC-005) — pan сам нашёл корень пакета вверх по дереву
```

**Примечание**: после того как сам `pan` соберётся в исполняемый `panos`-пакет
и будет доступен как `pan` в `PATH` (вне рамок этой фичи — вопрос дистрибуции),
вызовы выше сокращаются до `pan init`/`pan add`/`pan install`/`pan run`.

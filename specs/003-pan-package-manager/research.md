# Research: Pan — пакетный менеджер для panos

**Feature**: `003-pan-package-manager` | **Date**: 2026-07-21

Все пункты ниже разрешены по итогам `/speckit.clarify` и последующего обсуждения
(design discussion до `/speckit.plan`) — NEEDS CLARIFICATION в Technical Context
не остаётся.

## 1. Модель разрешения версий: единая версия vs сосуществование

- **Decision**: Единая разрешённая версия на canonical-имя пакета во всём графе
  зависимостей (Cargo-style). Конфликт двух разных версий одного пакета — хард-ошибка.
- **Rationale**: Проще резолвер и проще раскладка `модули/` — один физический путь
  на имя пакета, без версионированных под-путей. Соответствует уже принятой модели
  импортов panos (`импорт "имя"` — один модуль на путь).
- **Alternatives considered**: npm-style сосуществование нескольких версий одного
  пакета в графе — отклонено: требует версионированных путей в `модули/`
  (`модули/имя@1.0.0/`), что резолвер (`core/resolver_import_native.odin`) сегодня
  не поддерживает и добавлять ради этого — переусложнение для v1.

## 2. Синтаксис ограничения версии

- **Decision**: Semver-диапазоны (`^`, `~`, `>=`) поверх git-тегов; точный
  коммит/ветка — частный случай без дальнейшего разрешения.
- **Rationale**: Стандарт де-факто (Cargo/npm), необходим для реального управления
  транзитивными конфликтами при единой версии на граф.
- **Alternatives considered**: Только точный пин (тег/ветка/коммит) — отклонено:
  слишком жёстко в сочетании с единой версией на весь граф (любое обновление
  транзитивной зависимости ломало бы сборку).

## 3. Источник имени пакета/импорта

- **Decision**: Canonical-имя объявляет сама зависимость в своём манифесте;
  потребитель переопределяет его алиасом только при коллизии имён.
- **Rationale**: Устраняет вопрос "чьё имя главнее", совпадает с тем, как единая
  версия на граф резолвит имена глобально (одно имя = один пакет = одна версия).
- **Alternatives considered**: Имя выбирает потребитель при `pan add --as` —
  отклонено: усложняет резолв транзитивных зависимостей (два потребителя могут
  дать одному и тому же пакету разные локальные имена, граф перестаёт быть
  однозначным по имени).

## 4. Формат манифеста и lock-файла

- **Decision**: TOML. Манифест — `pan.toml`, lock-файл — `pan.lock`.
- **Rationale**: В проекте уже есть `std/кодирование/toml.ps` с разбором и
  сериализацией (`разобрать`/`сериализовать`, generic `разобрать_в[T]`/
  `сериализовать_из[T]`), включая already-used прецедент в самом
  `../panosiki/pan/start.ps` (`импорт "кодирование/toml" как томл`). Готовое
  решение снимает вопрос выбора формата и парсера.
- **Alternatives considered**: JSON — отклонено, TOML читаемее для
  человеко-редактируемого манифеста и уже импортирован в проекте; собственный
  формат — отклонено как ненужное изобретение при готовом стандарте.

## 5. Источник и способ получения зависимостей

- **Decision**: Git-репозитории (URL + тег/ветка/коммит), получаемые через
  `git clone`/`git checkout`, вызываемые как внешний процесс из panos-кода pan.
- **Rationale**: Общий механизм (любой git-хостинг, включая приватные/локальные
  репозитории), совпадает с уже принятой моделью версий (semver поверх git-тегов).
  Требует нового builtin `ос.выполнить` — но он в любом случае нужен отдельно для
  `pan run` (см. п. 7), так что переход на HTTP-tarball не убирает эту
  необходимость, а лишь добавляет отдельный путь получения зависимостей.
- **Alternatives considered**: Скачивание архива по HTTP
  (`https://.../archive/<ref>.tar.gz` + `сеть.http_запрос` +
  `архив.разобрать_tar_gz`, уже доступны в stdlib) — рассмотрено как способ
  избежать нового builtin'а для git; отклонено, так как (а) `ос.выполнить`
  всё равно обязателен для `pan run`, экономии на общем количестве новых
  примитивов нет; (б) URL-конвенция архивов различается между git-хостингами
  (GitHub/GitLab/self-hosted), что менее общее решение, чем `git clone` по
  произвольному URL.

## 6. Модель кэша зависимостей

- **Decision**: Единый локальный кэш на машине пользователя (например
  `~/.pan/cache/<host>/<org>/<repo>/<commit-or-tag>`), переиспользуемый между
  проектами; `модули/` в каждом пакете — production-копия/раскладка из кэша.
- **Rationale**: Избегает повторного скачивания одной и той же зависимости для
  разных локальных проектов (аналогично `~/.cargo/registry`).
- **Alternatives considered**: По-проектный кэш — отклонено, ведёт к дублированию
  скачиваний и дискового пространства без выгоды для однопользовательского
  инструмента.

## 7. Запуск пакета (`pan run`) и контроль рабочей директории

- **Decision**: `pan run` — единственная команда запуска (без отдельного
  `pan build` — у `panos` нет отдельного шага "собрать в артефакт", компиляция и
  исполнение происходят одним проходом внутри бинаря, см. `main.odin: run_file`).
  `pan run` ищет корень пакета (директорию с `pan.toml`), поднимаясь вверх по
  дереву от текущей директории (аналогично поиску `.git`/`Cargo.toml`), и
  спавнит бинарь `panos` дочерним процессом с рабочей директорией, явно
  установленной в найденный корень пакета.
- **Rationale**: Резолв `модули/` в `core/resolver_import_native.odin:12`
  завязан на CWD процесса `panos`, а не на путь entry-файла. Явный контроль cwd
  дочернего процесса — единственный способ гарантировать корректный резолв
  независимо от того, из какой поддиректории пакета вызван `pan run`, не трогая
  сам механизм резолва в `core/`.
- **Alternatives considered**: Генерировать Justfile-обёртку с рецептами сборки
  (`just build`/`just debug-file`) — отклонено пользователем: лишний
  генерируемый артефакт, который может разъехаться с манифестом, и лишняя
  индирекция вместо прямого вызова бинаря.

## 8. Недостающие примитивы стандартной библиотеки (core)

- **Decision**: Добавить в `core/` три новых builtin'а как prerequisite для
  этой фичи:
  1. `ос.выполнить(программа, аргументы, рабочая_директория) -> Результат((код,
     stdout, stderr), Ошибка)` — спавн внешнего процесса с контролем cwd и
     захватом вывода/кода завершения. Нужен для (а) `git clone`/`git checkout`
     при получении зависимостей, (б) спавна дочернего `panos` в `pan run`.
  2. Directory-ops в `фс`: рекурсивное создание директории (`mkdir -p`),
     список содержимого директории, рекурсивное удаление директории. Нужны для
     раскладки локального кэша и `модули/` (в т.ч. вложенных путей вида
     `модули/группа/имя.ps`) и для `pan remove` (удаление более не нужных
     закэшированных исходников).
  3. `ос.завершить(код) -> Никогда` — немедленное завершение процесса с
     заданным кодом (`os.exit`). Добавлен отдельно, по явному запросу
     пользователя закрыть гэп "точный exit code `pan run` не пробрасывается" —
     без него `pan run` мог только приблизительно сигнализировать неуспех
     через `паника` (ненулевой код, но не тот же самый).
- **Rationale**: Подтверждённый research-агентом факт — на момент начала фичи
  `ос` предоставляет только `аргументы`/`окружение`/`установить_окружение`/
  `удалить_окружение` (`core/stdlib.odin:214-238`), `фс` — только
  `есть`/`прочитать`/`записать`/`открыть` (`core/stdlib.odin:209-213`). Ни
  спавна процессов, ни директорий как понятия (создание/список/удаление), ни
  явного завершения с кодом не существовало. Без этих примитивов "написать pan
  на panos" физически невозможно (первые два) либо возможно лишь приближённо
  (третий, exit code) — сама фича требует минимального расширения
  интерпретатора.
- **Alternatives considered**: Обойтись без `pan run` (отменить решение п. 7,
  оставить только раскладку `модули/`) — отклонено пользователем явно (see
  Clarifications в spec.md). HTTP-tarball вместо git (см. п. 5) не устраняет
  нужду в `ос.выполнить`, так как он нужен и для `pan run` — рассмотрено и
  отклонено как не снижающее реальную сложность.

## 9. Тестирование

- **Decision**: Новые builtin'ы в `core/` тестируются через существующий
  `odin test ./core` (`just test`), по аналогии с остальными нативными модулями
  (см. `core/e2e_modules_stdlib_test.odin` как прецедент теста резолва модулей).
  Сам `pan` (panos-код в `../panosiki/pan/`) тестируется через `std/тест.ps` —
  уже существующий тестовый фреймворк panos, доступный любому `.ps`-проекту.
- **Rationale**: Переиспользование существующей инфраструктуры тестирования по
  обе стороны границы (core vs userland panos-код), никакого нового тестового
  фреймворка не требуется.

## 10. Расположение entry-файла и структура пакета pan

- **Decision**: Default entry-файл нового пакета — `start.ps` (уже сложившийся
  прецедент в `../panosiki/pan/start.ps`, а не изобретение нового имени).
  Сам pan как программа — многофайловый panos-пакет (не один гигантский
  `start.ps`), с внутренними относительными импортами между файлами
  (манифест/семвер/граф/кэш/git — отдельные модули), собранный в
  `../panosiki/pan/`.
- **Rationale**: Существующий `start.ps` уже содержит CLI-парсинг флагов
  (`Конфигурация.распарсить_флаги`) — логичная точка расширения, а не файл под
  замену. Разбивка на модули соответствует масштабу задачи (резолвер графа,
  TOML-маппинг, кэш, git-обёртка — самостоятельные, независимо тестируемые
  куски).
- **Alternatives considered**: Один монолитный `start.ps` на всё — отклонено,
  не соответствует масштабу логики (граф зависимостей + кэш + git + TOML +
  CLI) и затрудняет unit-тестирование отдельных частей через `std/тест.ps`.

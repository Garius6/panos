# Tasks: Pan — пакетный менеджер для panos

**Input**: Design documents from `/specs/003-pan-package-manager/`
**Prerequisites**: plan.md, spec.md, research.md, data-model.md, contracts/, quickstart.md

**Tests**: Не запрошены явно в spec.md — отдельных TDD-блоков "Tests for User
Story N" нет. Регрессионное покрытие двух новых core-builtin'ов включено
внутрь Foundational-задачи (T007), т.к. без него невозможно доверять
инфраструктуре, от которой зависят 3 из 4 user story.

**Organization**: Задачи сгруппированы по user story из spec.md. Работа идёт в
**двух репозиториях**: `core/` в этом репозитории (panos) и
`../panosiki/pan/` (отдельный git-репозиторий — сам pan, на panos).

## Format: `[ID] [P?] [Story] Description`

- **[P]**: можно выполнять параллельно (разные файлы, нет зависимости от незавершённых задач)
- **[Story]**: US1..US4 из spec.md
- Пути — точные, относительно корня соответствующего репозитория (указано в каждой задаче)

## Path Conventions

- Core-изменения (Foundational): `core/*.odin` в этом репозитории (`/Users/gaidar/dev/panos/`)
- Сам pan: `../panosiki/pan/*.ps` (репозиторий `/Users/gaidar/dev/panosiki/pan/`)

## Phase 4: User Story 2 - Добавление и удаление зависимости (Priority: P1)

**Goal**: `pan add <источник> [версия]` клонирует зависимость, разрешает граф (единая версия на имя, semver-диапазоны), фиксирует в `pan.toml`/`pan.lock`, раскладывает в `модули/`; `pan remove <имя>` отменяет.

**Independent Test**: `pan add <git-url>` → зависимость в манифесте+lock+`модули/`; `pan remove <имя>` → всё убрано (quickstart.md, US2).

### Implementation for User Story 2

- [X] T011 [P] [US2] Реализовать `../panosiki/pan/семвер.ps`: тип `Семвер` (major/minor/patch), `разобрать_версию(Строка) -> Результат(Семвер, Ошибка)`, `разобрать_ограничение(Строка) -> Результат(Ограничение, Ошибка)` (операторы `^`/`~`/`>=` плюс точный коммит/ветка как частный случай — research.md п.2), `удовлетворяет(семвер: Семвер, ограничение: Ограничение) -> Булево`, `наибольшая_подходящая(кандидаты: Массив(Строка), ограничение: Ограничение) -> Опция(Строка)`
- [X] T012 [P] [US2] Реализовать `../panosiki/pan/гит.ps` поверх `ос.выполнить` (T004): `клонировать(источник: Строка, путь_кэша: Строка) -> Результат(Пусто, Ошибка)`, `переключиться_на(путь: Строка, реф: Строка) -> Результат(Пусто, Ошибка)` (`git checkout`), `список_тегов(источник: Строка) -> Результат(Массив(Строка), Ошибка)` (`git ls-remote --tags`)
- [X] T013 [US2] Реализовать `../panosiki/pan/граф.ps`: `разрешить_граф(манифест: Манифест) -> Результат(Массив(Lock_Запись), Ошибка)` — единая разрешённая версия на canonical-имя во всём графе (research.md п.1), обнаружение конфликта версий (называет обе цепочки, FR-005) и циклов как хард-ошибки до любой записи на диск; использует `семвер.ps` (T011) и `гит.ps` (T012) — зависит от T011, T012
- [X] T014 [US2] Реализовать `../panosiki/pan/кэш.ps`: `путь_кэша(источник: Строка, коммит: Строка) -> Строка` (`~/.pan/cache/<host>/<org>/<repo>/<commit>`), `разложить_в_модули(записи: Массив(Lock_Запись), корень_пакета: Строка) -> Результат(Пусто, Ошибка)` через `фс.создать_директорию` (T005) + копирование содержимого (`фс.прочитать`+`фс.записать`, нет отдельного примитива копирования файла) — зависит от T005, T013
- [X] T015 [US2] Реализовать подкоманду `add` в `../panosiki/pan/start.ps`: разбор `<источник> [версия]`, upward-поиск корня пакета (проверка `фс.есть("pan.toml")` вверх по дереву от текущей директории), `граф.разрешить_граф` + `кэш.разложить_в_модули` + `манифест.сохранить` (с добавленной зависимостью) + запись `pan.lock`; при любой ошибке — ничего не менять (all-or-nothing, `contracts/cli.md`) — зависит от T013, T014
- [X] T016 [US2] Реализовать подкоманду `remove <имя>` в `../panosiki/pan/start.ps`: удаление записи из манифеста, пересчёт графа без неё, обновление `pan.lock`, удаление более не нужных записей из `модули/` через `фс.удалить_директорию` (T005) — зависит от T015 (тот же файл)
- [X] T017 [US2] Обработать коллизию canonical-имён при `add` в `граф.ps`/`start.ps`: если имя уже занято зависимостью с другим источником — явная ошибка с подсказкой синтаксиса алиаса, ничего не менять (edge case, spec.md) — зависит от T013, T015

## Dependencies & Execution Order

### Phase Dependencies

- **Setup (Phase 1)**: без зависимостей
- **Foundational (Phase 2)**: зависит от Setup; блокирует US2, US3, US4 — **не блокирует US1**
- **US1 (Phase 3)**: зависит только от Setup — может вестись параллельно с Foundational
- **US2 (Phase 4)**: зависит от Foundational (T004, T005)
- **US3 (Phase 5)**: зависит от Foundational (T005) и от `кэш.ps`/`граф.ps` из US2 (T013, T014) — переиспользует, не дублирует разрешение графа
- **US4 (Phase 6)**: зависит от Foundational (T004) — независима от US2/US3 по коду, но по смыслу полезна после того, как есть что запускать (US1)
- **Polish (Phase 7)**: после всех выбранных user story

### Parallel Opportunities

- T001 (Setup) — единственная параллелизуемая задача Setup
- T004, T005, T006 (Foundational) — разные файлы `core/`, без взаимной зависимости
- T008 (US1) — параллельно с любой из Foundational (T003-T007)
- T011, T012 (US2) — разные файлы (`семвер.ps`, `гит.ps`), без взаимной зависимости
- T018, T021 (US3/US4) — разные файлы/функции, без взаимной зависимости, можно вести одновременно с окончанием US2

## Implementation Strategy

### MVP First (User Story 1)

1. Phase 1: Setup
2. Phase 3: User Story 1 (можно начинать сразу после Setup, не дожидаясь Foundational)
3. **STOP и проверить**: `pan init` работает независимо от остального

### Incremental Delivery

1. Setup → US1 (MVP, `pan init`)
2. Foundational (можно параллельно с US1) → разблокирует US2/US3/US4
3. US2 (`pan add`/`pan remove`) → проверить независимо
4. US3 (`pan install`) → проверить независимо
5. US4 (`pan run`) → проверить независимо, полный quickstart.md проходит

# Contract: модуль stdlib `слог`

**Feature**: `004-gitsync-dependency-packages`

Новый файл `std/слог.ps` (не пакет, не git-репозиторий — обычный модуль
стандартной библиотеки, доступен через `импорт слог` без объявления
pan-зависимости, наравне с `фс`/`ос`/`строки`).

## API (v1)

```
экспорт функ отладка(текст: Строка) -> Пусто
экспорт функ инфо(текст: Строка) -> Пусто
экспорт функ предупреждение(текст: Строка) -> Пусто
экспорт функ ошибка(текст: Строка) -> Пусто
экспорт функ критично(текст: Строка) -> Пусто
```

## Поведение

- Каждая функция печатает ровно одну строку в stdout через
  `ввод_вывод.печать`/`ввод_вывод.строка`: `[УРОВЕНЬ] текст`, где `УРОВЕНЬ` —
  `ОТЛАДКА`/`ИНФО`/`ПРЕДУПРЕЖДЕНИЕ`/`ОШИБКА`/`КРИТИЧНО` соответственно
  (заглавными, для визуального отличия от самого сообщения).
- Без параметра "минимальный уровень"/фильтрации — все вызовы всегда что-то
  печатают (v1, см. research.md п.5).
- Без меток времени, без структурированного (JSON) вывода, без записи в
  файл — это то, что реально давал `logos` через appenders/layouts
  (`console-appender.os`, `file-appender.os`, `json-layout.os`,
  `standart-layouts.os`), сознательно не портируется в v1 (research.md п.4).
- Без побочных эффектов на глобальное состояние VM — чистые функции вывода.

## Явно вне scope v1 (не реализуется этой фичей)

- Appenders (консоль отдельно от файла, JSON-вывод).
- Layouts (форматирование сообщения, шаблоны).
- Настраиваемый порог уровня.
- Именованные логгеры (`log-manager.os`-подобная концепция — один логгер на
  модуль/класс).

Расширяется в будущей фазе, когда конкретная потребность `gitsync`
(или другого потребителя) это потребует — не заранее.

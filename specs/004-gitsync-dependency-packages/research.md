# Research: Пакеты-заготовки для зависимостей gitsync

**Feature**: `004-gitsync-dependency-packages` | **Date**: 2026-07-21

Все решения ниже разрешены в ходе `/speckit.specify` + `/speckit.clarify`
(включая проверку реальных `packagedef`/README каждой зависимости через
`gh api`, а не только предположений по названию) — NEEDS CLARIFICATION в
Technical Context не остаётся.

## 1. Состав зависимостей, получающих пакет-заготовку

- **Decision**: Из 15 рантайм-зависимостей `gitsync` пакет-заготовку в
  `panosiki/` получают 7: `tempfiles`, `v8runner`, `gitrunner`, `v8storage`,
  `cli`, `cli-selector`, `configor`. Остальные 8: `logos` → модуль stdlib
  `слог`; `json`/`strings`/`fs`/`delegate`/`opm`/`1commands` → исключены
  (покрыты существующим stdlib/языком/`pan`); `reflector` → отложена
  (блокирует отсутствующая в panos рефлексия).
- **Rationale**: Проверка РЕАЛЬНЫХ packagedef (не предположений по названию)
  показала, что `1commands` — это обёртка над запуском процессов с выводом
  через временные файлы (обход ограничения OneScript), полностью закрытая
  нативным `ос.выполнить`; `reflector` оборачивает нативную рефлексию
  рантайма OneScript, которой у panos нет вовсе; `cli` внутри зависит от
  `reflector` — 1:1 порт невозможен. `strings`/`fs` подтверждены как полное
  (не частичное) покрытие builtin `строки`/`фс`.
- **Alternatives considered**: Создать заготовку для всех 15 (буквальное
  прочтение исходного запроса "каждая зависимость — отдельный пакет") —
  отклонено пользователем явно: заготовка для функционала, которого либо уже
  нет смысла портировать (дублирует stdlib/`pan`), либо физически нельзя
  реализовать (`reflector`), была бы мусором, а не фундаментом для будущей
  работы.

## 2. Нейминг и версионирование пакетов

- **Decision**: Канонический `имя` в `pan.toml` каждого из 7 пакетов —
  латинское, совпадающее с оригинальным именем OneScript-зависимости
  (`tempfiles`, `v8runner`, `gitrunner`, `v8storage`, `cli`, `cli-selector`,
  `configor`). Версия — `0.1.0` у всех. Русское имя (`временные_файлы`,
  `запуск_1с`, `запуск_гит`, `хранилище_1с`, `командная_строка`,
  `селектор_консоли`, `конфигуратор`) фиксируется только в карте
  соответствия (`spec.md` → Key Entities) как рекомендуемый алиас для
  будущих `pan add <источник> [версия] <алиас>` — не как поле в `pan.toml`.
- **Rationale**: Латинское каноническое имя — трассируемость к
  oscript-library и единообразие с уже существующими `panosiki/pan`,
  `panosiki/panos-raylib`. Русский алиас — соответствие доминирующей в
  проекте конвенции идентификаторов на уровне кода-потребителя (`gitsync`
  на panos будет писать `импорт временные_файлы`, а не `импорт tempfiles`).
  `0.1.0` для всех — не создаёт ложного впечатления паритета с оригиналом
  (в отличие от копирования оригинальной версии в пустую заготовку).
- **Alternatives considered**: Полная русификация канонического имени —
  первоначально выбрана, затем отменена пользователем в пользу
  латиница+алиас (см. Clarifications в spec.md). Версия оригинала как
  стартовая — отклонена: вводит в заблуждение (пустая заготовка с версией
  "1.12.0" выглядит готовой).

## 3. Git-инициализация пакетов

- **Decision**: Каждый из 7 пакетов — отдельный git-репозиторий: `git init`
  + один коммит (все файлы после `pan init`) + тег `v0.1.0`. Коммит-сообщение
  — на русском (см. Constitution → Commit Messages; уже подтверждённый
  прецедент — `panosiki/panos-raylib`, коммит "Первая версия: raylib для
  panos поверх внешний/ff_структура").
- **Rationale**: Резолвер `pan` (feature 003-pan-package-manager) работает
  ТОЛЬКО через git-источник + git-теги (`граф.ps`: `гит.список_тегов` +
  `семвер.наибольшая_подходящая`) — без репозитория и тега `v0.1.0` будущий
  `pan add <путь-к-пакету> ^0.1.0` в составе сборки `gitsync` упадёт с
  "нет версии, удовлетворяющей ограничению" (реальная проверенная ошибка
  из `граф.ps`, feature 003). `pan init` САМ не делает `git init` (проверено
  по коду `команда_init` в `panosiki/pan/start.ps`) — это отдельный шаг.
- **Alternatives considered**: Отложить git-инициализацию до этапа, когда
  начнётся реальный перенос конкретного пакета — отклонено пользователем
  (см. Clarifications, Q1): без этого шага заготовка не соответствует
  собственному заявлению "готова принять портированный код" (SC-001).

## 4. Scope модуля `слог` (замена `logos`)

- **Decision**: `std/слог.ps` — новый модуль стандартной библиотеки (не
  пакет в `panosiki/`), 4-5 уровней (отладка/инфо/предупреждение/ошибка/
  критично), вывод только в stdout через `ввод_вывод.печать`. Без
  appenders (консоль/файл/JSON) и layouts — та функциональность, что
  реально имеет `logos` (подтверждено списком файлов в его `src/`:
  `levels.os`, `console-appender.os`, `file-appender.os`, `json-layout.os`,
  `standart-layouts.os`, `log-manager.os`).
- **Rationale**: `logos` — полноценный log4j-подобный фреймворк
  (уровни + множественные appenders + layouts). Порт 1:1 всего этого ради
  единственного текущего потребителя (будущий `gitsync`) — преждевременная
  генерализация (Constitution, Principle II). Минимальный набор (уровни +
  stdout) покрывает типовую потребность логирования CLI-инструмента и
  реализуется исключительно существующими примитивами (`ввод_вывод.печать`,
  `время.сейчас_мс` для меток времени) — без единого нового builtin'а в
  `core/`.
- **Alternatives considered**: Полный паритет с `logos` (appenders + layouts)
  — отклонено пользователем (Clarifications, Q2) как избыточное для v1;
  добавить только file-appender — отклонено, тот же принцип "не строить
  больше, чем нужно прямо сейчас", появится, когда реально понадобится.

## 5. Реализация `слог` — используемые примитивы

- **Decision**: Функции-константы уровня (`слог.отладка(текст)`,
  `слог.инфо(текст)`, `слог.предупреждение(текст)`, `слог.ошибка(текст)`,
  `слог.критично(текст)`) — каждая печатает `[УРОВЕНЬ] текст` через
  `ввод_вывод.печать`/`ввод_вывод.строка`; без глобального состояния
  (уровень фильтрации, минимальный порог и т.п. — не запрошено, не
  добавляется).
- **Rationale**: Простейшая реализация, полностью в пределах уже
  существующих builtin'ов (`ввод_вывод`), никаких изменений в `core/`.
- **Alternatives considered**: Настраиваемый порог уровня (например
  `слог.установить_уровень(...)`, фильтрующий что печатается) — отклонено:
  не запрошено ни пользователем, ни текущими требованиями spec.md;
  добавится в будущей фазе, если/когда понадобится `gitsync`.

## 6. Транзитивные зависимости самих 7 пакетов

- **Decision**: Транзитивные зависимости, обнаруженные в реальных
  packagedef (`fluent`, `datetime`, `semver`, `v8find`, `asserts` — не
  входящие в топ-15 самого `gitsync`), НЕ получают собственную оценку/
  заготовку в этой фиче.
- **Rationale**: Явно зафиксировано в spec.md (Edge Cases) — каждая такая
  транзитивная зависимость получает собственное решение (порт / замена
  stdlib / отложена) на этапе переноса КОНКРЕТНОГО пакета, который в ней
  нуждается, а не заранее.
- **Alternatives considered**: Оценить и создать заготовки для всех
  транзитивных зависимостей сразу — отклонено: расширяет scope этой фичи
  далеко за пределы топ-15 `gitsync`, без непосредственной необходимости
  прямо сейчас.

## 7. Тестирование

- **Decision**: `слог` тестируется через существующий `core` test suite
  (аналогично тому, как `../panosiki/pan` тестировался в feature 003 —
  запуском реального `.ps`-сценария через собранный бинарь `panos`, либо
  через `run_code` в `core/*_test.odin`, если модуль стабилен и не требует
  внешнего процесса). 7 пакетов-заготовок на этом этапе не тестируются
  содержательно (пустой скелет) — единственная проверка: `pan.toml`
  валиден, git-репозиторий с тегом `v0.1.0` существует.
- **Rationale**: Нет новой бизнес-логики в 7 заготовках, которую можно было
  бы тестировать содержательно (FR-003) — тест был бы тестом самого `pan
  init`/`git`, уже покрытых feature 003.

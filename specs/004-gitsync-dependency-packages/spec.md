# Feature Specification: Пакеты-заготовки для зависимостей gitsync

**Feature Branch**: `004-gitsync-dependency-packages`
**Created**: 2026-07-21
**Status**: Draft
**Input**: User description: "Следующий большой этап - перенос https://github.com/oscript-library/gitsync на panos. Начни с зависимостей. Каждая зависимость - отдельный пакет в директории panosiki. Используй pan для инициализации пакетов"

## Clarifications

### Session 2026-07-21

- Q: Нейминг пакетов — оставить оригинальные латинские имена OneScript-зависимостей или русифицировать? → A: Канонический `имя` в `pan.toml` — латинское (как у оригинала, для трассируемости с oscript-library и единообразия с `pan`/`panos-raylib`); русское имя фиксируется как рекомендуемый алиас для будущих `pan add <источник> [версия] <алиас>` (механизм алиаса уже есть в `pan`).
- Q: Версионирование заготовок — с `0.1.0` или с версии оригинала? → A: Все с `0.1.0` — новый независимый пакет, версия растёт по мере реального переноса.
- Уточнение пользователя (скорректировало scope, без отдельного вопроса): не
  дублировать функционал, уже покрытый panos stdlib или уже существующим
  инструментом panos — `logos` входит в stdlib (`слог`); `json`, `delegate`,
  `opm` исключены (закрыты `кодирование/json.ps`, нативными функциями-
  значениями и самим `pan` соответственно).
- Уточнение пользователя (перенос — не дословный порт, а переосмысление под
  идиомы panos; пример — reflection, которой у panos нет): по итогам
  проверки РЕАЛЬНЫХ packagedef каждой оставшейся зависимости (не только
  предположений по названию) состав скорректирован ещё раз:
  - `strings`, `fs` — подтверждено полное покрытие builtin-модулями `строки`/
    `фс` — исключены.
  - `1commands` (в реальности — "упрощение запуска процессов/командных
    файлов из oscript", а не CLI-фреймворк) — целиком заменяется нативным
    `ос.выполнить` (прямой возврат кода/stdout/stderr, без обходного пути
    через временные файлы, который был нужен OneScript). Исключена.
  - `reflector` (в реальности — "удобная работа с рефлектором OneScript") —
    оборачивает рефлексию, встроенную в рантайм OneScript; у panos рефлексии
    как языковой возможности нет вовсе (статическая типизация, никакого
    introspection API наружу). Порт невозможен без отдельной, гораздо более
    крупной работы над самим языком/core — **не входит в scope этой фичи**,
    пакет-заготовка НЕ создаётся, зависимость помечена "отложена".
  - `cli` внутри зависит от `reflector` (реального, из packagedef) — то есть
    сам фреймворк реализован через рефлексию OneScript. На panos это
    невозможно повторить как есть; пакет создаётся, но реализация в будущей
    фазе переноса ДОЛЖНА быть спроектирована заново на функциях-значениях/
    pattern matching, без потребности в рефлексии.
  - `cli-selector` — интерактивный выбор в консоли (класс `ВыборВКонсоли`,
    подтверждено `README.md`) требует посимвольного ввода без ожидания
    Enter (стрелки навигации) — у panos `ввод_вывод` умеет только построчное
    чтение (`прочитать_строку`), нет raw-terminal режима. Полноценная версия
    заблокирована отсутствующей core-возможностью; пакет создаётся, но
    scope первой версии — упрощённый выбор по номеру (доступно уже сейчас).
  - `v8storage` — проверка packagedef показала, что её реальный механизм
    строится поверх `v8runner` (то есть запуск 1С процессом), а не через
    OneScript-специфичный биндинг — портируема через уже имеющийся
    `ос.выполнить`, поэтому статус не меняется.
- Q: Нужна ли git-инициализация (репозиторий + тег) для 7 новых пакетов в
  рамках этой фичи, раз `pan` резолвит зависимости только через git-источник
  и git-теги? → A: Да — `git init` + первый коммит + тег `v0.1.0` для
  каждого из 7 пакетов входит в эту фичу, иначе заготовки нельзя было бы
  реально подключить через `pan add` до отдельного будущего шага.
- Q: Реальный `logos` — полноценный фреймворк (уровни + appenders консоль/
  файл/JSON + layouts). Какой scope у модуля `слог` v1? → A: Минимум — 4-5
  стандартных уровней + вывод только в stdout через `ввод_вывод.печать`, без
  appenders/layouts; расширяется позже по реальной потребности `gitsync`.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Заготовка пакета на каждую зависимость без готового аналога и без блокирующего разрыва возможностей (Priority: P1)

Разработчик, переносящий `gitsync` (синхронизация хранилища 1С с git-репозиторием,
исходно на OneScript) на panos, начинает с фундамента: для каждой рантайм-
зависимости оригинального пакета, у которой ЕЩЁ НЕТ покрытия ни в panos
stdlib, ни среди уже существующих инструментов panos (`pan`), И которую
реально можно реализовать средствами panos (не упирается в отсутствующую
языковую возможность вроде рефлексии), должен появиться отдельный,
самостоятельный panos-пакет — правильно расположенный, инициализированный
через `pan`, готовый принять портированный код в последующих этапах.

**Why this priority**: Без пакетов-заготовок для зависимостей невозможно ни
спланировать порядок переноса, ни (в дальнейшем) собрать сам `gitsync` как
panos-пакет, объявляющий эти зависимости через `pan add`. Это точка входа во
весь многоэтапный перенос.

**Independent Test**: Для каждой зависимости из списка со статусом "новый
пакет" или "частичное покрытие" (см. Key Entities) в директории `panosiki/`
существует отдельная директория с валидным `pan.toml` (создан через
`pan init`), не пересекающаяся по содержимому с другими такими директориями.

**Acceptance Scenarios**:

1. **Given** директория `panosiki/` без директории для зависимости `X`
   (статус не "исключена", не "вошла в stdlib" и не "отложена"), **When**
   разработчик инициализирует git-репозиторий (`git init` + первый коммит +
   тег `v0.1.0`) и выполняет `pan init` для новой директории `panosiki/X`,
   **Then** появляется валидный `pan.toml` с латинским именем (как у
   оригинала) и версией `0.1.0`; рекомендуемый русский алиас для будущих
   потребителей зафиксирован в карте соответствия (US3), а не в самом
   `pan.toml`; директория готова к `pan add` из другого пакета через тег
   `v0.1.0`.
2. **Given** 7 зависимостей gitsync со статусом "новый пакет" или
   "частичное покрытие" (см. Key Entities), **When** проход завершён,
   **Then** в `panosiki/` существует ровно 7 новых директорий-пакетов, по
   одной на такую зависимость, без пропусков и дублей.
3. **Given** зависимость со статусом "исключена" (`json`, `strings`, `fs`,
   `delegate`, `opm`, `1commands`) или "отложена" (`reflector` — блокирует
   отсутствующая в panos рефлексия), **When** проход завершён, **Then** для
   неё НЕ создан отдельный пакет в `panosiki/`.
4. **Given** уже существующая в `panosiki/` директория с именем, совпадающим
   с именем зависимости (конфликт с уже существующим пакетом), **When**
   разработчик доходит до этой зависимости, **Then** конфликт явно замечен и
   разрешён до создания (например переименованием) — не тихой перезаписью.

### User Story 2 - Логирование становится частью panos stdlib (`слог`) (Priority: P1)

Аналог зависимости `logos` (логирование) не оформляется отдельным пакетом в
`panosiki/`, а входит в стандартную библиотеку panos под именем `слог` —
доступен любой panos-программе через `импорт слог` без объявления внешней
зависимости через `pan`, наравне с уже существующими `фс`/`ос`/`строки`.

**Why this priority**: Логирование — сквозная потребность, нужная не только
`gitsync`, но и любому будущему panos-пакету (включая сам `pan`); держать его
как внешнюю зависимость каждого проекта означает лишний `pan add` там, где
достаточно `импорт`. Тот же приоритет, что у US1 — оба закладывают фундамент,
на который опирается сам перенос `gitsync`.

**Independent Test**: Любая panos-программа (без объявленных pan-зависимостей)
может написать `импорт слог` и вызвать базовые операции логирования (уровни,
сообщение) без ошибки резолва импорта.

**Acceptance Scenarios**:

1. **Given** новый `.ps`-файл без каких-либо pan-зависимостей, **When** в нём
   написано `импорт слог` и вызвана базовая операция логирования, **Then**
   импорт резолвится (модуль стандартной библиотеки, не внешний пакет) и
   операция выполняется.
2. **Given** модуль `слог`, **When** разработчик просматривает его API,
   **Then** доступны 4-5 стандартных уровней (отладка/инфо/предупреждение/
   ошибка/критично) с выводом только в stdout через `ввод_вывод.печать` —
   без appenders/layouts оригинального `logos` (полный паритет — вне scope
   этой фичи, расширяется позже по реальной потребности `gitsync`).

---

### User Story 3 - Карта соответствия «oscript-зависимость → судьба на panos» (Priority: P2)

Разработчик (и любой, кто продолжит перенос позже, включая будущие сессии с
AI-агентом) может открыть один документ и увидеть для каждой из 15
оригинальных рантайм-зависимостей: как называлась и какой версии была на
OneScript; что с ней происходит на panos — становится отдельным пакетом (и
каким именно), частью stdlib, исключена как дублирующая уже существующий
функционал, либо отложена как блокированная отсутствующей возможностью
panos; и — где применимо — что именно закрывает пересечение или что именно
блокирует перенос.

**Why this priority**: Без явной карты соответствия перенос рискует
дублировать уже существующий в panos функционал, потерять из виду, что
конкретная зависимость на panos нереализуема как есть (`reflector`), или
попытаться портировать 1:1 то, что требует переосмысления (`cli`,
`cli-selector`).

**Independent Test**: Документ карты соответствия существует и по нему можно
однозначно определить для любой из 15 исходных зависимостей: где искать её
продолжение на panos (пакет / stdlib / нигде — покрыта существующим /
отложена и почему).

**Acceptance Scenarios**:

1. **Given** список из 15 рантайм-зависимостей gitsync, **When** карта
   соответствия составлена, **Then** каждая из 15 имеет ровно одну строку с
   исходным именем/версией, судьбой на panos и (если применимо) новым именем
   пакета/модуля.
2. **Given** зависимость с частичным/полным пересечением с существующим на
   panos ИЛИ заблокированная отсутствующей возможностью panos, **When** карта
   составлена, **Then** явно указано, что именно покрывает пересечение или
   что именно блокирует перенос — не общая пометка "дублирует"/"сложно", а
   конкретная ссылка (`кодирование/json.ps`, `строки`, `фс`, `pan`,
   `кодирование/toml.ps`, `ос.выполнить`) либо конкретная недостающая
   возможность (рефлексия, посимвольный raw-terminal ввод).

---

### Edge Cases

- `logos` — не пакет, а часть stdlib (`слог`, см. US2).
- `json`, `strings`, `fs` — полностью покрываются существующими
  `кодирование/json.ps`, builtin `строки`, builtin `фс` (включая недавно
  добавленные directory-ops). Отдельные пакеты НЕ создаются.
- `delegate` — закрыт нативными функциями как значениями первого класса
  (используются, например, в `std/коллекции.ps`). Отдельный пакет НЕ
  создаётся.
- `opm` — роль пакетного менеджера уже играет сам `pan`. Отдельный пакет НЕ
  создаётся.
- `1commands` — по факту (не по названию) это обёртка над запуском внешних
  процессов и командных файлов с захватом вывода через временные файлы —
  обходной путь вокруг ограничения OneScript. panos это ограничение не
  имеет: `ос.выполнить` уже отдаёт код завершения/stdout/stderr напрямую.
  Отдельный пакет НЕ создаётся — и `v8runner`, и `gitrunner` (оба зависели от
  `1commands` в оригинале) на panos используют `ос.выполнить` напрямую.
- `reflector` — оборачивает нативную рефлексию рантайма OneScript; у panos
  такой возможности в языке нет (см. Clarifications). Пакет-заготовка НЕ
  создаётся на этом этапе — статус "отложена". Что именно из `gitsync`
  реально требует рефлексии, выясняется на этапе переноса самого `gitsync`;
  тогда и решается — редизайн конкретной логики без рефлексии или отдельная
  будущая инициатива по добавлению introspection API в сам язык.
- `cli` — реализован через `reflector` внутри оригинала (подтверждено
  packagedef) — porting 1:1 невозможен по той же причине, что у `reflector`.
  Пакет создаётся (концепция CLI-приложения нужна `gitsync`), но реализация
  в будущей фазе переноса MUST быть спроектирована заново на функциях-
  значениях и pattern matching panos, без использования рефлексии.
- `cli-selector` — интерактивный выбор в консоли требует посимвольного
  ввода без Enter (навигация стрелками); у panos `ввод_вывод` умеет только
  построчное чтение. Пакет создаётся с суженным scope первой версии — выбор
  по вводу номера пункта (реализуемо уже сейчас); полноценный интерактивный
  вариант — предмет отдельной будущей инициативы, если/когда в panos
  появится raw-terminal ввод.
- `v8storage` — механизм (по факту зависимостей в packagedef) строится
  через запуск 1С процессом (`v8runner`), а не через OneScript-специфичный
  биндинг — портируется через `ос.выполнить`, как и `v8runner`.
- `configor` — частично пересекается с `кодирование/toml.ps`, но
  конфигурационные библиотеки обычно делают больше (слияние источников,
  значения по умолчанию, поиск файла). Пакет-заготовка создаётся; при
  последующем переносе логика MUST переиспользовать `кодирование/toml.ps`
  для пересекающейся части.
- Зависимости для разработки/тестирования оригинала (`1bdd`, `1testrunner`,
  `asserts`, `coverage`) — вне рамок этой фичи (см. Assumptions); тестирование
  портированного кода использует уже существующий `std/тест.ps`.
- Что если у самих зависимостей обнаружатся собственные транзитивные
  зависимости при более глубоком изучении в момент переноса (например
  `fluent`, `datetime`, `semver`, `v8find` — уже замечены как транзитивные
  зависимости `v8storage`/`cli`/`1commands`/`v8runner` в их реальных
  packagedef, но не входят в топ-15 самого `gitsync`)? Для ЭТОЙ фичи это не
  блокирует — заготовка создаётся независимо от внутреннего устройства;
  каждая такая транзитивная зависимость получает собственную оценку
  (порт / замена stdlib / отложена) на этапе переноса конкретного пакета,
  который в ней нуждается.
- Что если имя зависимости, как оно есть у OneScript, конфликтует с уже
  занятым в `panosiki/` именем? Среди текущих зависимостей такого совпадения
  нет; если возникнет в будущем — разрешается явным алиасом/переименованием,
  а не тихой перезаписью существующей директории.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Для каждой из 7 рантайм-зависимостей `gitsync` со статусом
  "новый пакет" или "частичное покрытие" (список — Key Entities) MUST быть
  создана отдельная директория в `panosiki/`, инициализированная как
  panos-пакет через `pan init`.
- **FR-012**: Каждая из 7 директорий MUST быть отдельным git-репозиторием
  (`git init`) с первым коммитом и тегом `v0.1.0` — без этого пакет нельзя
  подключить будущим `pan add` (резолвер `pan` работает только через
  git-источник и git-теги, см. feature 003-pan-package-manager).
- **FR-002**: Каждый созданный пакет MUST иметь валидный `pan.toml` с
  латинским именем, совпадающим с оригинальным именем OneScript-зависимости,
  и версией `0.1.0`.
- **FR-003**: Ни одна из 7 заготовок MUST NOT содержать перенесённой бизнес-
  логики оригинальной зависимости на этом этапе — только пустой скелет
  (манифест + минимальная точка входа); сам перенос кода — предмет
  последующих отдельных этапов.
- **FR-004**: Зависимости для разработки/тестирования оригинала MUST NOT
  получать пакет-заготовку на этом этапе.
- **FR-005**: Зависимости, полностью покрытые существующим panos stdlib
  (`json`, `strings`, `fs`), закрытые возможностями самого языка panos
  (`delegate`) или уже существующим инструментом panos (`opm` ≈ `pan`,
  `1commands` ≈ `ос.выполнить`) MUST NOT получать отдельный пакет.
- **FR-006**: `logos` MUST NOT становиться пакетом в `panosiki/` — вместо
  этого фиксируется как задача добавления нового модуля `слог` в panos
  stdlib (см. US2).
- **FR-007**: Зависимость, чей перенос упирается в отсутствующую в panos
  языковую возможность (`reflector` — нужна рефлексия, которой у panos нет),
  MUST NOT получать пакет-заготовку на этом этапе — фиксируется в карте
  соответствия со статусом "отложена" и конкретной причиной блокировки.
- **FR-008**: Зависимости, чья оригинальная реализация опирается на
  недостающую возможность, но чья ЗАДАЧА решаема иначе средствами panos
  (`cli` — без рефлексии, через функции-значения; `cli-selector` — в
  суженном по вводу-номера scope вместо raw-terminal навигации) MUST
  получать пакет-заготовку с явной пометкой в карте соответствия о

- **FR-009**: Для каждой из 15 исходных зависимостей MUST быть

- **FR-010**: Система именования (латинское каноническое имя + русский

- **FR-011**: При конфликте имени новой заготовки с уже существующей

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Для всех 7 зависимостей `gitsync` со статусом "новый пакет"
  или "частичное покрытие" в `panosiki/` существует ровно один валидный,
  самостоятельно собираемый, готовый к `pan add` (git-репозиторий с тегом
  `v0.1.0`) пакет-заготовка на каждую — без пропущенных и без дублирующихся
  директорий.
- **SC-002**: Ни для одной из 6 исключённых зависимостей (`json`, `strings`,
  `fs`, `delegate`, `opm`, `1commands`), вошедшей в stdlib (`logos`) или
  отложенной (`reflector`) не создан отдельный пакет в `panosiki/`.
- **SC-003**: Любая panos-программа может использовать логирование через
  `импорт слог` без объявления pan-зависимости.
- **SC-004**: Открыв карту соответствия, разработчик может однозначно
  назвать судьбу, латинское имя пакета (если применимо) и рекомендуемый
  русский алиас для любой из 15 исходных зависимостей за время менее одной
  минуты, не заглядывая в исходный `packagedef` gitsync.
- **SC-005**: Для `reflector` карта соответствия называет конкретную
  недостающую возможность panos (рефлексия), а не общую формулировку
  "сложно перенести".
- **SC-006**: Начало работы над переносом любой ОДНОЙ конкретной зависимости
  в следующей сессии не требует повторного анализа gitsync целиком — только
  чтения записи для этой зависимости в карте соответствия.

## Assumptions

- Перенос — не построчный/дословный порт OneScript-кода, а переосмысление
  под возможности и идиомы panos: там, где у panos уже есть нативная
  возможность (функции как значения первого класса, ADT + pattern matching,
  `Результат`/`Опция`, `ос.выполнить`, собственный `std/тест.ps`) или готовый
  инструмент (`pan`), оригинальная OneScript-библиотека, решающая ТУ ЖЕ
  задачу другим способом из-за ограничений OneScript, не портируется как
  отдельный код. Симметрично: там, где у panos, наоборот, ЕЩЁ НЕТ нужной
  возможности (рефлексия, raw-terminal посимвольный ввод), перенос
  соответствующей зависимости либо откладывается (`reflector`), либо
  реализуется в суженном/переосмысленном виде без этой возможности
  (`cli`, `cli-selector`) — не имитируется искусственно.
- "Зависимости" в этой фиче — 15 рантайм-зависимостей `gitsync`, а не
  зависимости для разработки/тестирования оригинала — те не переносятся,
  тестирование портированного кода использует уже существующий `std/тест.ps`.
- Пакеты-заготовки размещаются непосредственно в `panosiki/` (плоско, без
  вложенной группировки по имени "gitsync") — так же, как уже существуют
  `panosiki/pan` и `panosiki/panos-raylib`.
- Решение "частичное покрытие" для `configor` и оценка scope для `cli`/
  `cli-selector` основаны на сопоставлении назначения библиотек и их
  реальных packagedef (проверены через `gh api`), а не построчного сравнения
  всего исходного кода — точный набор недостающих функций уточняется на
  этапе переноса именно этой зависимости.
- Эта фича покрывает только СОЗДАНИЕ заготовок, модуль `слог` и карту
  соответствия; перенос реальной логики каждой зависимости (включая редизайн
  `cli`/`cli-selector` и решение по `reflector`) и сама сборка `gitsync` как
  panos-пакета — предмет отдельных, последующих этапов.
- Назначения зависимостей и решения по ним в Key Entities основаны на
  реальных packagedef и README каждой зависимости (проверено через GitHub
  API), а не только на предположениях по именам — но не на построчном
  изучении всего исходного кода каждой библиотеки.

# Data Model: Устранение проблем языка panos

**Feature**: `005-language-fixes` | **Date**: 2026-07-23

Фича не вводит данных/сущностей рантайма (см. spec.md → Key Entities) —
единственные затрагиваемые "сущности" это AST-узлы компилятора
(`core/parser.odin`). Ниже — изменения их формы, а не новые типы.

## `Type_Qualified` (P1)

Существующий узел (`core/parser.odin`), представляющий тип вида
`модуль.Имя`:

```text
Type_Qualified :: struct {
    module_name: string
    name:        string
    span:        Span
    // НОВОЕ:
    params:      [dynamic]Type_Node   // пусто = не-generic (текущее поведение)
}
```

- **Пустой `params`** — семантика не меняется: `модуль.Тип` без аргументов
  резолвится так же, как сегодня (немодифицированный существующий путь в
  `type_cheker.odin`).
- **Непустой `params`** — новый путь: каждый элемент — обычный `Type_Node`
  (та же грамматика, что аргументы локального `Type_Generic`), резолвится
  как generic-инстанцирование через уже существующий
  `instantiate_type`/`decl_type_param_order`/`generic_instance_cache`
  (см. research.md §1).
- **Валидация** (типизация, не парсинг): количество аргументов должно
  совпадать с числом объявленных type-параметров экспортируемого типа;
  несовпадение — `Type Error` (тот же формат сообщения, что уже
  используется для локального `Type_Generic`, см. `core/type_cheker.odin`
  ~2655-2664). Тип не экспортирован модулем — уже существующая ошибка
  "модуль '%s' не экспортирует '%s'" (~2600-2606), путь не меняется.

## `Match_Arm` (P2)

Существующий узел (`core/parser.odin`) не меняет форму — `body` уже
`[dynamic]Stmt` до этой фичи. Меняется ТОЛЬКО парсинг (`parse_match_expr`):

- **Пометка формы** не хранится в AST отдельным полем — она не нужна
  дальше парсера: результат обеих форм (`->` однострочная и `тогда...
  конец` многострочная) — один и тот же `arm.body: [dynamic]Stmt` c 1 или
  N элементами соответственно. Type checker/compiler уже безразличны к
  тому, как ветка была написана — важна только длина `body` и то, что
  последний элемент — `Expr_Stmt` (иначе тип ветки — `Пусто`, тот же путь,
  что и сегодня для однострочной формы, если она не expression-statement,
  например `возврат ...`).

## Comma-separated списки (P3)

Не вводит новых узлов — исправление применяется к УЖЕ существующим полям
(`Call_Expr.args`, `Variant_Decl.types`, `Pattern_Constructor.args`,
`Type_Generic.params`, `Type_Tuple.elements` и др., см. research.md §3):
поведение парсинга становится терпимым к ОДНОЙ завершающей запятой перед
закрывающим токеном списка; сама структура/тип поля не меняется.

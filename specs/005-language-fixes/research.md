# Research: Устранение проблем языка panos, найденных при переносе gitsync

**Feature**: `005-language-fixes` | **Date**: 2026-07-23

Спецификация (spec.md) не оставила `[NEEDS CLARIFICATION]`-маркеров, но в
ходе технического планирования один пункт (P2) потребовал реального
архитектурного решения, не предусмотренного в spec.md буквально —
разобрано в §2 вместе с решением, принятым пользователем.

## 1. P1 — квалифицированный generic-тип как type-annotation

- **Decision**: `Type_Qualified` (AST-узел для `модуль.Имя`, `core/
  parser.odin`) получает новое поле `params: [dynamic]Type_Node`.
  `parse_type` — после чтения `модуль.Имя`, ПЕРЕД `return`, проверяет
  `peek_token(p.stream).kind == .LParen && !peek_token(p.stream).nl_before`
  (тот же guard, что уже есть у локального `Type_Generic` чуть ниже по
  файлу) — если условие истинно, разбирает список типов-аргументов той же
  функцией `parse_type` рекурсивно (с уже фиксимой в этой же фиче trailing-
  comma поддержкой, см. §3), иначе `params` остаётся пустым (сегодняшнее
  поведение — квалифицированный НЕ-generic тип, без изменений).
  `type_cheker.odin`'s `resolve_type_node`, ветка `case ^Type_Qualified`:
  после нахождения `export_sym`/его типа — если `n.params` непусто,
  выполняется ТОТ ЖЕ инстанцирующий путь, что уже написан для
  пользовательского `Type_Generic` (строки ~2650-2683 `core/type_cheker.
  odin`): проверка `ctx.decl_type_param_order[export_sym]` (иначе — "не
  generic-тип"), проверка количества аргументов, `instantiate_type` с
  подстановкой по `infer_id`, кэширование по `generic_instance_key`. Если
  `n.params` пусто — путь не меняется (текущее поведение).
- **Rationale**: Вся инфраструктура инстанцирования generic-типов уже
  существует и используется для локального `Type_Generic` — задача чисто
  механическая: (a) парсер обязан РАЗОБРАТЬ уже присутствующий в
  исходном тексте `(...)` вместо того, чтобы оставлять его непрочитанным
  (что и вызывало ложную "undefined variable" ошибку — парсер просто
  прекращал разбор типа раньше времени, и `(Строка)` парсилось отдельно
  как начало ВЫРАЖЕНИЯ тела функции); (b) тайпчекер обязан вызвать уже
  существующую функцию инстанцирования с новыми аргументами вместо
  молчаливого игнорирования. Никакой новой инфраструктуры не требуется.
- **Alternatives considered**: Отдельная, специальная функция
  инстанцирования для квалифицированных типов (не переиспользующая
  `instantiate_type`) — отклонено: чистое дублирование кода без причины,
  два generic-типа (локальный и квалифицированный) семантически ничем не
  отличаются после того, как найден их `Symbol`.

## 2. P2 — многострочное тело ветки `выбор`

- **Decision** (обсуждено с пользователем перед фиксацией — исходное
  предположение спеки "просто снять ограничение на 1 statement" оказалось
  неверным при чтении `parse_pattern`): синтаксис ветки расширяется на ДВЕ
  формы, различимые СРАЗУ после разбора шаблона по СЛЕДУЮЩЕМУ токену:
  - `Шаблон -> выражение` — существующая однострочная форма, без изменений
    (ровно один statement, `parse_stmt_into` вызывается один раз, как
    сегодня).
  - `Шаблон тогда стейтмент1 \n стейтмент2 \n ... конец` — НОВАЯ
    многострочная форма: переиспользует уже существующий токен `.Then`
    (лексема `тогда`, уже применяется в `если ... тогда`) как маркер начала
    блока и уже существующий `.End` (`конец`) как маркер конца — парсер
    `parse_match_expr` после `parse_pattern` смотрит на следующий токен:
    `Arrow` → однострочная форма (как сегодня); `Then` → читает
    `parse_stmt_into` в цикле, ПОКА следующий токен не `.End` (тот же цикл,
    что уже используют `parse_if_expr`'s `then_branch`/`else_branch` и
    `parse_while_expr`'s `body`), затем съедает СВОЙ `.End` (закрывающий
    ИМЕННО эту ветку, не весь `выбор` — вложенность `конец` уже работает
    корректно для if/while внутри тел функций, тот же механизм); иначе —
    synatx error "после шаблона ожидается '->' или 'тогда'".
  - Значение ветки в ОБЕИХ формах — её хвостовое выражение; `type_cheker.
    odin`'s `infer_match_expr` и `compiler.odin`'s `compile_match_expr`
    УЖЕ реализуют это генерически по длине `arm.body` (см. ниже) — эти
    файлы не меняются вообще.
- **Rationale**: Чтение `parse_pattern` показало, что тело ветки СЕГОДНЯ
  ограничено одним statement НЕ случайно и не только "забыли" убрать
  break — грамматика шаблона (`Идент`, опционально `.Идент` цепочка,
  опционально `(аргументы)`) синтаксически СОВПАДАЕТ с обычным
  statement-вызовом метода (`это.метод(х)` парсится как валидный шаблон
  `это.метод(х)`) — greedy-парсинг statement'ов текущей ветки до
  бесконечности не может честно отличить "ещё один statement этой же
  ветки" от "началась следующая ветка", а парсер panos — hand-written
  recursive-descent/Pratt БЕЗ backtracking (grep `checkpoint`/`rewind`/
  `save_pos` по `parser.odin`/`lexer.odin` — ноль совпадений). Пользователю
  представлены 3 варианта (новый keyword-маркер / backtracking в парсере /
  убрать P2 из фичи); выбран явный маркер, но с уточнением
  переиспользовать УЖЕ существующий токен `тогда` вместо изобретения нового
  слова — `тогда` уже семантически означает "дальше — блок стейтментов" в
  единственном другом месте языка, где это нужно (`если`), поэтому
  переиспользование не добавляет когнitivной нагрузки и не вводит новый
  токен в лексер вообще (`Then`/`тогда` уже существует как токен).
  Обоснование нулевых изменений в `compiler.odin`/бОльшей части
  `type_cheker.odin`: `compile_match_expr` (`core/compiler.odin:1606`)
  уже вызывает `compile_block(ctx, arm.body, is_val)` — тот же вызов, что
  используют тела if/while/функций, генерически по длине `arm.body`, а
  `infer_match_expr` (`core/type_cheker.odin:5562`) уже содержит цикл `for
  stmt, i in arm.body { is_last := i == len(arm.body) - 1; ... }` — оба
  места были написаны ГОТОВЫМИ к многострочным веткам ещё тогда, когда
  `Match_Arm.body` получил тип `[dynamic]Stmt` вместо одиночного `Expr`;
  единственное несделанное звено — сам `parse_match_expr`, который никогда
  не пользовался этой готовностью.
- **Alternatives considered**:
  - Backtracking/lookahead (пробовать разобрать `Шаблон -> `после каждого
    statement, откатываясь при неудаче) — отклонено пользователем: вводит
    в парсер принципиально новую возможность (checkpoint/rewind токен-
    потока), которой нигде в кодовой базе нет, с более высоким риском
    тонких багов на edge cases (что считать "похожим на начало шаблона")
    ради того, чтобы избежать одного дополнительного (уже существующего!)
    ключевого слова.
  - Новый выделенный keyword (`блок`) вместо переиспользования `тогда` —
    рассмотрено, отклонено пользователем в пользу переиспользования уже
    существующего `тогда` (тот же результат — однозначный маркер без
    backtracking, но без добавления нового слова в лексер).
  - Полностью убрать P2 из объёма фичи — отклонено пользователем: сам
    дефект (эргономика `выбор` для two-liner) реален и стоит устранения
    именно предложенным способом с приемлемой ценой.

## 3. P3 — завершающая запятая в списках через запятую

- **Decision**: Аудит ВСЕХ 17 мест `core/parser.odin`, где парсится список
  через запятую (`grep -n "kind == .Comma" core/parser.odin`), на предмет
  того, что ПОСЛЕ съедания запятой проверяется закрывающий токен списка
  (`RParen`/иначе) ПЕРЕД повторным вызовом элемент-парсера — этот паттерн
  уже применяется (safe) в: `parse_param_list` (~1610), `Array_Expr`
  литерал (~2260), `Map_Expr` литерал (~2290), `Type_Function`-параметры
  (~1759). НЕ применяется (unsafe — уже подтверждено конкретными
  сценариями в этой сессии) минимум в: список аргументов ВЫЗОВА (`Call_
  Expr.args`, ~2386), список типов варианта перечисления (`Variant_Decl.
  types`, ~1310), список аргументов шаблона-конструктора (`Pattern_
  Constructor.args`, ~2654), список аргументов generic-типа (`Type_
  Generic.params`, ~1805), список элементов тьюпл-типа (`Type_Tuple.
  elements`, ~1831). Остальные из 17 сайтов (~902, ~1131, ~1500, ~1544,
  ~1929, ~2183, ~2505, ~2510) классифицируются по тому же критерию на
  этапе `/speckit.tasks`/реализации — не в этом research.md, чтобы не
  дублировать работу, которую всё равно предстоит сделать построчным
  просмотром при реализации (Simplicity First — не разворачивать здесь
  результат построчного аудита, если он один раз делается в самой задаче).
  Каждый unsafe-сайт получает ОДИНАКОВУЮ правку: `if peek_token(p.stream).
  kind == .Comma { next_token(p.stream); if peek_token(p.stream).kind ==
  <закрывающий_токен_списка> { break } } else { break }` — тот же паттерн,
  что уже в safe-сайтах.
- **Rationale**: Инконсистентность УЖЕ существует внутри одного файла —
  часть списков через запятую безопасна для завершающей запятой, часть
  нет, без видимой причины для разницы (не разные семантические роли,
  просто разное время/автор правки). Механическая унификация к уже
  установленному паттерну — не новая фича, а устранение внутренней
  непоследовательности.
- **Alternatives considered**: Обобщённый helper-параметр `parse_comma_
  separated_list(p, closing_token, parse_element)` вместо копирования
  паттерна в каждый сайт — рассмотрено, отклонено (Simplicity First):
  сайты используют парсер по-разному (`Type_Node` vs `Expr` vs `Pattern`,
  разные пост-обработки типа named-args), обобщение потребовало бы
  generic-процедуры/callback-параметра в Odin ради 5-8 идентичных
  трёхстрочных блоков — цена абстракции выше цены дублирования здесь.

## 4. Совместимость и тестирование

- **Decision**: Все три правки проверяются существующим `core`-test suite
  (`odin test ./core`, `just test`) — новые e2e-тесты добавляются в новый
  файл `core/e2e_language_fixes_test.odin` (по аналогии с существующими
  `core/e2e_*_test.odin`), НЕ модифицируя существующие тестовые файлы.
  Полный прогон существующего набора (251+ тестов на момент написания)
  без регрессий — обязательное условие приёмки (FR-008, SC-003).
- **Rationale**: Проект уже использует этот паттерн (`run_code` через
  `core/e2e_test.odin`-подобные файлы, см. AGENTS.md) — новый файл
  изолирует новые тесты, не рискуя смешать их с уже существующими и
  не давая случайно затронуть чужие assertions при правке.
- **Alternatives considered**: Дописывать в уже существующие `e2e_*_test.
  odin` файлы по тематике (например, generic-тесты — в файл с generics) —
  отклонено: фича трогает 3 разных, не связанных друг с другом угла языка
  (типы, pattern matching, синтаксис списков) — общий новый файл под саму
  фичу нагляднее, чем распыление по трём существующим файлам.

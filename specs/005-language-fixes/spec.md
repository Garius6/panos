# Feature Specification: Устранение проблем языка panos, найденных при переносе gitsync

**Feature Branch**: `005-language-fixes`
**Created**: 2026-07-23
**Status**: Draft
**Input**: User description: "Исправь описанные выше ошибки" (список из 8 языковых проблем, найденных при портировании 6 пакетов-зависимостей gitsync — panosiki/gitrunner, tempfiles, configor, cli-selector, v8runner, v8storage)

## Контекст

Список из предыдущего сообщения содержал 8 пунктов. При подготовке этой
спецификации каждый пункт заново разобран на предмет того, является ли он
РЕАЛЬНЫМ дефектом языка (несоответствие собственному дизайну/документации,
неожиданное поведение без обходного пути) или УЖЕ КОРРЕКТНЫМ поведением
(строгая типизация, отсутствие фичи, которую никто не запрашивал). Только
реальные дефекты становятся user stories ниже; остальные разобраны в разделе
"Assumptions" с обоснованием.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Квалифицированный generic-тип как аннотация типа (Priority: P1)

Разработчик panos-пакета пишет функцию-хелпер в одном файле, которая
конструирует и возвращает значение generic-типа, объявленного в ДРУГОМ
импортированном модуле (например, `тип Селектор[T] = структура ... конец` в
модуле `console_selector.ps`, и хелпер `построить_селектор() -> выборка.
Селектор(Строка)` в тестовом файле). Сейчас это падает с ложной ошибкой
резолва ("undefined variable 'Строка'"), потому что парсер типов, встретив
`модуль.Имя`, не проверяет, следует ли за именем `(...)` с generic-
аргументами — тогда как ЛОКАЛЬНЫЙ generic-тип (`Опция(Число)`,
`Массив(Строка)`) инстанцируется нормально. Из-за этого пришлось разворачивать
7 повторяющихся блоков кода в тесте `console_selector` вместо одной
переиспользуемой функции-конструктора.

**Why this priority**: Блокирует базовый паттерн переиспользования (functions
factory) для ЛЮБОГО generic-типа, объявленного не в текущем файле — это
не редкий синтаксис, а стандартный способ строить тестовые фикстуры и
конструкторы в многофайловых panos-пакетах.

**Independent Test**: `.ps`-файл с `тип Пример[T] = структура значение: T
конец` в одном модуле и функцией `функ построить() -> модуль.Пример(Строка)`
во ВТОРОМ файле, импортирующем первый — компилируется и исполняется без
ошибок резолва/типов.

**Acceptance Scenarios**:

1. **Given** модуль `a.ps` с `экспорт тип Коробка[T] = структура значение: T
   конец`, **When** модуль `b.ps` импортирует `a.ps` и объявляет `функ
   новая() -> a.Коробка(Число) ... конец`, **Then** программа компилируется и
   `новая()` возвращает корректно типизированное значение `a.Коробка(Число)`.
2. **Given** тот же сценарий, но с несколькими generic-параметрами
   (`Соответствие`-подобный тип модуля `a.ps` с двумя type-параметрами),
   **When** `b.ps` объявляет функцию с аннотацией `a.Тип(Строка, Число)`,
   **Then** оба аргумента типа резолвятся корректно (не только первый).
3. **Given** качественный generic-тип используется как аннотация ПАРАМЕТРА
   функции (не только возвращаемого типа), **When** файл компилируется,
   **Then** ошибки резолва/типов не возникает.

### User Story 2 - Многострочное тело ветки `выбор`/`если` (Priority: P2)

Разработчик panos-пакета пишет ветку `выбор значение Шаблон -> тело ...` (или
`если ... тогда ... иначе ... конец`), где тело ветки должно выполнить
побочный эффект (например, `это.список.добавить(х)`) И вернуть значение
(например, `Результат.Успех(х)`) — то есть два выражения подряд. Сейчас это
синтаксическая ошибка: тело ветки принимает РОВНО одно выражение, а не блок
операторов с последним выражением как значением (как работают тела функций и
ветки `если`/`иначе`, взятые как отдельные блоки). Из-за этого пришлось
переписывать несколько мест в `temp_files.ps`/`config_manager.ps` через
дополнительные переменные и промежуточные `если ... тогда возврат ... конец`
вместо естественного two-liner в ветке `выбор`.

**Why this priority**: Реже блокирует работу насмерть (есть обходной путь
через `.ошибка()`/`.значение()` вместо `выбор`, или вынос в функцию), но
делает `выбор` — основной инструмент pattern matching в языке —
неудобным для самого частого случая "записать И вернуть" и вынуждает менее
читаемый код.

**Independent Test**: `выбор значение Шаблон -> стейтмент1 \n стейтмент2
\n хвостовое_выражение` (несколько строк на одном уровне отступа внутри
одной ветки) компилируется, `стейтмент1`/`стейтмент2` выполняются по
порядку, значением ветки становится `хвостовое_выражение` — согласовано с
тем, как уже работают тела функций и одиночные блоки `если`/`иначе`.

**Acceptance Scenarios**:

1. **Given** `выбор х \n Вариант(п) -> \n стейтмент_с_побочным_эффектом(п)
   \n хвостовое_выражение(п) \n конец`, **When** код компилируется и
   выполняется, **Then** побочный эффект применяется и ветка `выбор`
   вычисляется в `хвостовое_выражение(п)`.
2. **Given** аналогичная многострочная ветка внутри `если ... тогда ...
   иначе ... конец`, где ОБЕ ветки многострочные и заканчиваются выражениями
   разных на первый взгляд форм, но одного типа, **When** код компилируется,
   **Then** ошибки "ветки возвращают разные типы" не возникает, если типы
   ХВОСТОВЫХ выражений совпадают (промежуточные операторы не участвуют в
   проверке типа ветки).
3. **Given** многострочная ветка, где хвостовое выражение одной ветки имеет
   другой тип, чем у другой ветки/веток, **When** код компилируется, **Then**
   типовая ошибка о несовпадении типов веток выдаётся так же, как раньше
   (проверка типов по хвостовому выражению не должна становиться слабее).

### User Story 3 - Завершающая запятая в списке аргументов вызова (Priority: P3)

Разработчик panos-пакета форматирует многострочный вызов функции с
несколькими аргументами и по привычке (или через авто-форматирование)
оставляет запятую после последнего аргумента перед закрывающей скобкой.
Сейчас это синтаксическая ошибка ("неожиданный токен ')'"). Разработчик
вынужден вручную убирать финальную запятую при переносе строк.

**Why this priority**: Чисто эргономическая проблема — не блокирует работу
(достаточно не ставить запятую), самая низкая цена невключения в объём.

**Independent Test**: Многострочный вызов функции с запятой сразу перед
закрывающей `)` компилируется идентично тому же вызову без завершающей
запятой.

**Acceptance Scenarios**:

1. **Given** `функция(\n  аргумент1,\n  аргумент2,\n)`, **When** код
   компилируется, **Then** ошибки нет и вызов эквивалентен `функция(аргумент1,
   аргумент2)`.
2. **Given** список параметров функции/структуры/перечисления (объявление,
   не вызов) с завершающей запятой в аналогичной позиции, **When** код
   компилируется, **Then** ошибки нет (единообразие с списком аргументов
   вызова).

---

### Edge Cases

- Что происходит, если многострочная ветка `выбор` (User Story 2) содержит
  `возврат` посреди тела — ветка после `возврат` не должна требовать
  хвостового выражения (как в обычном теле функции, `возврат` уже прерывает
  выполнение).
- Что происходит с квалифицированным generic-типом (User Story 1), если
  указанный generic-тип НЕ экспортирован из модуля — должна выдаваться
  ошибка "тип не экспортирован"/"не найден", а не путаница с "непарсенными"
  аргументами типа, как сейчас.
- Что происходит с завершающей запятой (User Story 3), если аргументов
  ноль (`функция(,)`) — это должно оставаться ошибкой (запятая без
  предшествующего аргумента), завершающая запятая допустима только ПОСЛЕ
  хотя бы одного аргумента.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Парсер типов (`parse_type`, ветка `Type_Qualified`) MUST
  проверять наличие `(` сразу после `модуль.Имя` (без переноса строки перед
  ней, по аналогии с уже существующей проверкой для локальных generic-типов)
  и, если она есть, разбирать список generic-аргументов типа тем же образом,
  что и для нативного (неквалифицированного) `Type_Generic`.
- **FR-002**: Резолвер/тайпчекер MUST резолвить каждый generic-аргумент
  квалифицированного типа (`модуль.Тип(Арг1, Арг2, ...)`) как обычный
  type-annotation, а не как выражение — тип-параметры не должны попадать в
  область видимости имён-переменных.
- **FR-003**: Квалифицированный generic-тип с аргументами MUST быть
  допустим в ЛЮБОЙ позиции, где допустим обычный generic-тип — аннотация
  параметра функции, аннотация возвращаемого типа, аннотация `пер`,
  аннотация поля структуры.
- **FR-004**: Тело ветки `выбор ... Шаблон -> тело` MUST поддерживать блок
  из нескольких операторов на одном уровне отступа (как тело функции), где
  ПОСЛЕДНЕЕ выражение блока становится значением ветки — сохраняя обратную
  совместимость с уже работающим односрочным телом ветки.
- **FR-005**: Проверка совпадения типов между ветками `выбор` (и между
  `тогда`/`иначе` внутри `если`) MUST основываться ТОЛЬКО на типе хвостового
  (последнего) выражения каждой ветки — промежуточные операторы блока не
  участвуют в выводе типа ветки, как и в телах функций.
- **FR-006**: Парсер списка аргументов ВЫЗОВА функции MUST допускать ОДНУ
  необязательную завершающую запятую перед закрывающей `)`, если список
  содержит хотя бы один аргумент.
- **FR-007**: То же самое послабление (FR-006) MUST применяться единообразно
  к спискам параметров при ОБЪЯВЛЕНИИ функции/структуры/перечисления —
  разработчик не должен помнить, где завершающая запятая разрешена, а где
  нет.
- **FR-008**: Ни одно из изменений FR-001…FR-007 не MUST менять поведение
  уже компилирующегося сегодня кода (полная обратная совместимость) —
  проверяется прогоном существующего `core`-тестового набора без
  регрессий.

### Key Entities

Фича не вводит новых данных/сущностей рантайма — это исключительно
расширения грамматики/тайпчекера компилятора (`core/parser.odin`,
`core/resolver.odin`, `core/type_cheker.odin`).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Разработчик может объявить и использовать функцию-конструктор
  generic-типа из другого модуля (`модуль.Тип(Аргумент)` как type-annotation)
  без обходных путей (дублирования кода, отказа от аннотации типа) — на
  практике: 7 повторяющихся блоков в `panosiki/cli-selector/
  test_console_selector.ps` сводятся к 1 функции-конструктору без потери
  компилируемости.
- **SC-002**: Разработчик может написать ветку `выбор`/`если` с побочным
  эффектом и возвращаемым значением в ДВЕ строки внутри одной ветки, без
  выноса в отдельную функцию — на практике: хотя бы одно из мест-обходов в
  `panosiki/tempfiles/temp_files.ps` или `panosiki/configor/
  config_manager.ps`, переписанных в этой сессии из-за ограничения,
  переписывается обратно в естественную двухстрочную форму и компилируется.
- **SC-003**: Существующий `just test` (весь `core`-набор, 251+ тестов) MUST
  проходить без регрессий после внесения изменений.
- **SC-004**: Компиляция файла с завершающей запятой в списке аргументов
  вызова завершается успешно в 100% случаев, где тот же список без
  завершающей запятой уже компилировался успешно.

## Assumptions

- User Story 1 (квалифицированные generic-типы) и User Story 2
  (многострочные ветки) — это исправления РЕАЛЬНЫХ дефектов (расхождение с
  уже работающим для похожих случаев поведением: локальные generic-типы
  парсятся с аргументами, тела функций уже поддерживают блок-с-хвостовым-
  выражением). User Story 3 (завершающая запятая) — чистое эргономическое
  расширение, не дефект.
- Не входит в объём (пункты из исходного списка, разобранные как НЕ
  дефекты, а корректное/сознательное поведение — фиксировать НЕ требуется):
  - **Нет top-level констант** — минималистичный, сознательный дизайн (см.
    AGENTS.md: верхний уровень — только `функ`/`тип`), а не пробел.
  - **`Целое`/`Число` без неявного приведения** — сознательная строгая
    типизация; явные касты `Целое(x)`/`Число(x)` уже существуют и
    достаточны.
  - **Нет builtin random/uuid** — отсутствующая ФИЧА (feature request), не
    дефект существующего поведения.
  - **`строки.в_число` возвращает `Результат`, а не `Опция`** — согласовано
    с остальным API модуля `строки` (проверка через `Результат`, где ошибка
    содержательна — "не число", а не просто "было/не было") — это было
    ошибочное ожидание при портировании, а не дефект языка.
  - **Generic `реализация Тип` без повторения `[T]`/`(T)` у receiver** —
    единообразно и уже документировано практикой `core/prelude.odin`
    (`Опция`/`Результат`) — не дефект, а недостаточно очевидное для
    новичка (в контексте этого проекта) соглашение.
- Реализация FR-001…FR-007 затрагивает только `core/parser.odin`,
  `core/resolver.odin`, `core/type_cheker.odin` (и, при необходимости,
  `core/compiler.odin` для генерации байткода многострочных веток) —
  никаких изменений в `std/`-модулях или уже существующих
  `panosiki/`-пакетах эта фича не требует (они уже написаны в рамках
  текущих ограничений и продолжат работать).
- LSP (`lsp/`) и автодополнение (`core/completion.odin`) не гарантированно
  обновляются этой фичей — если многострочные ветки/квалифицированные
  generic-типы потребуют изменений в LSP для корректных
  подсказок/диагностики, это фиксируется отдельной фичей после базовой
  поддержки в компиляторе.

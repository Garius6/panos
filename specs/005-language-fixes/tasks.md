во втором файле, импортирующем первый, — компилируется и исполняется без
ошибок резолва/типов.

## Phase 4: User Story 2 - Многострочное тело ветки `выбор`/`если` (Priority: P2)

**Goal**: Ветка `Шаблон тогда стейтмент1 \n стейтмент2 \n ... конец`
компилируется — побочные эффекты выполняются по порядку, значением ветки
становится её хвостовое (последнее) выражение; однострочная форма
(`Шаблон -> выражение`) продолжает работать без изменений.

**Independent Test**: `выбор х \n Вариант(п) тогда \n стейтмент_с_побочным_
эффектом(п) \n хвостовое_выражение(п) \n конец \n конец` компилируется,
исполняется, побочный эффект применяется, значение ветки — хвостовое
выражение.

### Tests for User Story 2 ⚠️

> Пишутся ПЕРВЫМИ, должны ПАДАТЬ до реализации ниже.

- [X] T007 [US2] Написать e2e-тесты многострочной ветки `выбор` в
  `core/e2e_language_fixes_test.odin`: (1) `Шаблон тогда ... конец` —
  побочный эффект + хвостовое значение (Acceptance Scenario 1); (2) один
  `выбор` с ОДНОЙ веткой в форме `тогда...конец` и ДРУГОЙ в форме `->`,
  типы хвостовых значений совпадают — компилируется (Acceptance Scenario
  2); (3) типы хвостовых значений ВЕТОК не совпадают — `Type Error`
  выдаётся так же, как раньше (Acceptance Scenario 3, регрессия
  строгости проверки типов); (4) `возврат` посреди многострочной ветки —
  не требует хвостового выражения после себя (Edge Case); (5) та же
  многострочная форма работает и для `если...тогда...иначе...конец` (обе
  ветки многострочные — FR-005 распространяется и на `если`, не только
  `выбор`)

### Implementation for User Story 2

- [X] T008 [US2] В `parse_match_expr` (`core/parser.odin`) — после
  `parse_pattern`, ПЕРЕД сегодняшним безусловным `expect(p, .Arrow)`:
  посмотреть следующий токен; если `.Arrow` — существующий путь без
  изменений (один `parse_stmt_into`); если `.Then` — съесть токен, затем

  `.End` (закрывает конкретную ветку, не весь `выбор` — вложенность
  `конец` уже корректно работает для if/while внутри тел функций); иначе
  — `report_parse` с сообщением "после шаблона ожидается '->' или
  'тогда'"
- [X] T009 [US2] Прогнать `just test` — тесты T007 проходят, регрессий
  нет; ПОДТВЕРДИТЬ (не предполагать), что `infer_match_expr`

## Dependencies & Execution Order

### Phase Dependencies

- **Setup (Phase 1)**: без зависимостей
- **Foundational (Phase 2)**: пусто — не блокирует user story
- **US1 (Phase 3)**: зависит только от Setup
- **US2 (Phase 4)**: зависит только от Setup — независима от US1 (другая
  функция парсера, `core/type_cheker.odin` не трогается вообще)
- **US3 (Phase 5)**: зависит только от Setup — независима от US1/US2
  (другие участки `parser.odin`, `core/type_cheker.odin` не трогается)
- **Polish (Phase 6)**: после всех выбранных user story

### Within Each User Story

- Тесты (T002/T007/T010) пишутся и ПАДАЮТ до соответствующей реализации
- US1: T003 → T004 → T005 (struct-поле → парсинг → типизация,
  последовательная зависимость) → T006
- US2: T008 (единственная задача реализации) → T009
- US3: T011-T015 (5 независимых по коду, но одного файла правок,
  выполняются последовательно во избежание накладывающихся diff'ов) →
  T016 (аудит остального) → T017

### Parallel Opportunities

- Между US1/US2/US3 — теоретически параллельны (разные функции/участки),
  но обе трогают `core/parser.odin` — при solo-исполнении естественно
  последовательно, [P] намеренно не проставлен ни на одной паре
  задач US1/US2/US3 (тот же файл, риск конфликта diff'ов важнее
  теоретической независимости)
- T018/T019 (Polish) — разные файлы/активности (ручной прогон vs правка
  AGENTS.md), можно параллельно

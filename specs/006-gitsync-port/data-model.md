# Data Model: Портирование ядра gitsync на panos

**Feature**: `006-gitsync-port` | **Date**: 2026-07-23

## VERSION (файл, не тип panos)

Простой текстовый файл в рабочем git-каталоге — ровно один текстовый
токен, число (например `3`). Отсутствие файла означает "ничего ещё не
синхронизировано" (эквивалент `0`).

- **Чтение**: `фс.есть(путь)` → если нет, `0`; если есть — `фс.
  прочитать(путь)` → `строки.в_число(строки.обрезать(текст))`.
- **Запись**: `фс.записать(путь, строки.из_числа(N))` — ТОЛЬКО после
  успешного коммита версии N (см. spec.md Edge Cases/SC-003).
- **Валидация** (`set-version`): значение MUST быть неотрицательным
  целым — отрицательное/нечисловое отклоняется как ошибка (FR-007).

## AUTHORS (TOML-файл, не тип panos)

```toml
[Администратор]
имя = "Admin User"
email = "admin@example.com"

["Вася Иванов"]
имя = "Vasily Ivanov"
email = "vasily@example.com"
```

- Ключ верхнего уровня — имя пользователя хранилища 1С (как есть, точное
  совпадение с тем, что вернёт `v8storage`/`v8runner` — в текущей
  суженной реализации этих пакетов имя пользователя коммита не
  извлекается автоматически из вывода платформы, см. Assumptions ниже).
- `имя`/`email` — отображаемое имя и адрес для git (`user.name`/
  `user.email`).
- Отсутствие записи для пользователя — не ошибка (FR-005): имя
  пользователя хранилища используется как `user.name` напрямую, `user.
  email` не устанавливается для этого коммита (репозиторий использует
  ранее установленное/глобальное значение).

## `Контекст_Синхронизации` (panos-структура, `sync.ps`)

```panos
экспорт тип Контекст_Синхронизации = структура
	путь_хранилища:       Строка
	рабочий_каталог:      Строка
	пользователь_хранилища: Строка
	пароль_хранилища:      Строка
	имя_расширения:        Строка   // "" = основная конфигурация, не расширение
	путь_к_платформе:      Строка   // "" = использовать значение по умолчанию ("1cv8")
конец
```

Собирается в `start.ps` из флагов `cli` + переменных окружения (FR-009,
флаг переопределяет переменную) — см. contracts/cli-surface.md за точным
соответствием имён.

## `Результат_Синхронизации` (panos-структура, `sync.ps`)

Возвращается функцией синхронизации для тестируемости (вместо печати
напрямую) — сколько версий синхронизировано, до какого номера дошли:

```panos
экспорт тип Результат_Синхронизации = структура
	синхронизировано_версий: Число
	итоговая_версия:          Число
конец
```

## Ключевые операции (без нового структурного состояния, композиция уже существующих типов)

- **`получить_известные_версии(хранилище, начиная_с)`**
  (`sync.ps`) — перебирает `начиная_с+1, начиная_с+2, ...`, вызывая
  `Хранилище.версию_в_файл(N, temp_путь)` (см. research.md §1), возвращает
  наибольший успешный N (или `начиная_с`, если версий нет).
- **`синхронизировать(контекст) -> Результат(Результат_Синхронизации, Ошибка)`**
  (`sync.ps`) — оркестрирует: чтение VERSION → цикл по новым версиям
  (research.md §3: v8storage→temp .cf→v8runner→файлы→git add .→
  установить_настройку(автор из AUTHORS)→закоммитить→запись VERSION) →
  возврат `Результат_Синхронизации`.

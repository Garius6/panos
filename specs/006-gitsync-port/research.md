# Research: Портирование ядра gitsync на panos

**Feature**: `006-gitsync-port` | **Date**: 2026-07-23

Спецификация (spec.md) не оставила `[NEEDS CLARIFICATION]`-маркеров — оба
пункта ниже не блокеры спецификации, а технические находки, сделанные при
чтении API уже реализованных пакетов (`panosiki/gitrunner`,
`panosiki/v8storage`), прежде чем писать код.

## 1. Перечисление версий хранилища без расширения v8storage

- **Decision**: `sync` не запрашивает у хранилища "список версий больше
  N" ОДНОЙ командой — вместо этого перебирает номера версий
  ПОСЛЕДОВАТЕЛЬНО, начиная с `VERSION+1`, вызывая `Хранилище.версию_в_
  файл(строки.из_числа(N), путь)` для каждого N по очереди, пока вызов не
  вернёт `Результат.Неудача` (несуществующая версия) — на этом
  остановиться, последняя успешная N и есть новый `VERSION`.
- **Rationale**: `v8storage.storage_manager.ps` (panosiki/v8storage,
  спроектирован в 004/предыдущей фиче) НАМЕРЕННО не включает
  историю/список версий — единственный способ получить его в оригинале
  (`/ConfigurationRepositoryReport`) требует ПАРСИНГА текстового вывода
  команды Конфигуратора, формат которого невозможно проверить без
  установленной платформы 1С (та же причина, по которой это уже было
  исключено из v8storage: "требует ПАРСИНГА вывода команд... риск
  непроверяемо угадать формат"). Последовательный перебор `версию_в_
  файл(N, ...)` до первой ошибки использует УЖЕ существующий, уже
  протестированный метод — самого факта "команда вернула ненулевой код
  возврата на несуществующую версию" достаточно, не нужно понимать
  СОДЕРЖИМОЕ вывода, только его успешность (`Результат.Успех`/
  `Результат.Неудача`, которые `Хранилище.выполнить_команду` уже даёт).
- **Alternatives considered**: Расширить `v8storage` методом
  `получить_последнюю_версию()`/`получить_список_версий()`, парсящим
  `/ConfigurationRepositoryReport` — отклонено: непроверяемо в песочнице
  без реальной платформы (тот же риск, что был явной причиной ИСКЛЮЧИТЬ
  эту функциональность из v8storage в первый раз), а последовательный
  перебор даёт ТОТ ЖЕ результат без риска.

## 2. Автор коммита без расширения gitrunner

- **Decision**: `sync` НЕ передаёт автора как параметр `закоммитить()` —
  вместо этого вызывает `Репозиторий.установить_настройку("user.name",
  <имя_из_AUTHORS>, РежимНастройкиGit.Локально)` и аналогично
  `"user.email"` НЕПОСРЕДСТВЕННО ПЕРЕД каждым `закоммитить()` — git
  читает `user.name`/`user.email` из локального конфига РЕПОЗИТОРИЯ в
  момент самого коммита, так что смена конфига перед каждым коммитом даёт
  разных авторов для разных версий БЕЗ отдельного `--author`-флага.
- **Rationale**: `gitrunner/git.ps` (panosiki/gitrunner) сознательно НЕ
  портировал `АвторКоммита`/`ДатаАвтораКоммита`/etc. параметры оригинала
  ("суженный scope... port сюда добавляет только 2 варианта простой + -a"
  — см. коммит gitrunner). Composition через уже существующий
  `установить_настройку` даёт ТОЧНО нужный эффект (разный автор на разный
  коммит) без единой новой строчки в уже написанном и протестированном
  пакете.
- **Alternatives considered**: Расширить `закоммитить()` необязательным
  параметром автора — отклонено: не нужно, `установить_настройку`+
  `закоммитить` уже достаточны, а расширение существующего пакета внутри
  ЭТОЙ фичи усложнило бы Constitution Check (Surgical Changes — эта фича
  не должна трогать уже готовые 7 пакетов).

## 3. Конвертация версии хранилища в файлы исходников

- **Decision**: На каждую новую версию N: (1) `v8storage.Хранилище.
  версию_в_файл(N, temp_cf)` — выгрузить версию N В .cf-файл во временный
  файл (`tempfiles.Менеджер.создать_файл("cf")`); (2) `v8runner.
  Конфигуратор` БЕЗ контекста (временная база, встроенное поведение
  `новый_конфигуратор()` — не нужен файловый/серверный контекст для этой
  операции) — `.загрузить_конфигурацию_из_файла(temp_cf)`, затем
  `.выгрузить_конфигурацию_в_файлы(рабочий_каталог)` — прямо в
  git-рабочий каталог; (3) `tempfiles.Менеджер.удалить()` — очистка
  временного .cf.
- **Rationale**: Ровно то, для чего `v8runner.новый_конфигуратор()`
  (без контекста = временная база) уже спроектирован — прямое
  переиспользование, не обход. `tempfiles` — ровно то, для чего он
  реализован (временный .cf между двумя шагами конвейера).
- **Alternatives considered**: Держать постоянную (не временную) базу для
  конвертации — отклонено: дополнительное состояние без пользы, временная
  база уже встроенный режим `v8runner`.

## 4. "git add" на весь рабочий каталог

- **Decision**: После `выгрузить_конфигурацию_в_файлы` (шаг 3 выше)
  добавляются В ИНДЕКС ВСЕ изменённые файлы каталога одним вызовом
  `Репозиторий.выполнить_команду(массив("add", "."))` — не по одному
  файлу через `добавить_файл_в_индекс`.
- **Rationale**: `добавить_файл_в_индекс` принимает ОДИН путь —
  выгрузка конфигурации в исходники создаёт МНОГО файлов (по объекту
  метаданных), перебирать их поштучно избыточно, когда `выполнить_
  команду` (уже существующий escape hatch именно для команд, не
  обёрнутых отдельным методом) даёт `git add .` за один вызов.
- **Alternatives considered**: Добавить `Репозиторий.добавить_всё()`
  как новый метод gitrunner — отклонено, то же обоснование, что §2
  (Surgical Changes — не трогаем уже готовый пакет, когда escape hatch
  уже достаточен).

## 5. Формат VERSION/AUTHORS (подтверждение решения из spec.md)

- **Decision**: `VERSION` — текстовый файл с ОДНИМ числом (например `3`),
  читается/пишется напрямую через `фс.прочитать`/`фс.записать` +
  `строки.в_число`/`строки.из_числа` — без обёртки формата. `AUTHORS` —
  TOML, ключ верхнего уровня = имя пользователя хранилища 1С, значение —
  таблица `{ имя = "...", email = "..." }`.
- **Rationale**: Уже согласовано с пользователем на этапе `/speckit.
  specify` (см. spec.md Assumptions) — здесь фиксируется КОНКРЕТНАЯ
  TOML-схема (плоская таблица построчно на пользователя, не список), т.к.
  спецификация утверждала только выбор формата (TOML vs INI), не схему.
- **Alternatives considered**: `AUTHORS` как единая строка
  `"Имя <email>"` (git-совместимый author-string) вместо раздельных
  `имя`/`email` полей — отклонено: `установить_настройку` (§2)
  принимает `user.name`/`user.email` РАЗДЕЛЬНО, разбирать единую строку
  обратно на 2 части — лишний шаг без пользы.

## 6. CLI-поверхность

- **Decision**: `panosiki/cli` (флаги.ps) уже поддерживает подкоманды
  (`новая_подкоманда(имя, синонимы)`) — `gitsync` использует ОДНУ
  `Конфигурация` с тремя подкомандами (`sync`/`init`/`set-version`, с
  синонимами `s`/—/`sv` соответственно, как в оригинале), каждая со
  своими флагами (`--storage-user`/`-u`, `--storage-pwd`/`-p`,
  `--ext`/`-e` — уже поддержанные однодефисные/синонимичные флаги из
  `cli`, реализованного/протестированного ранее в этом же проекте).
  Позиционные аргументы (путь к хранилищу, рабочий каталог) — через
  `позиционные_аргументы` в действии подкоманды.
- **Rationale**: Прямое переиспользование уже написанного и
  протестированного `cli` — ничего нового не требуется на уровне
  парсинга аргументов.
- **Alternatives considered**: Нет — `cli` уже покрывает потребность
  полностью.

## 7. Тестирование без реальной платформы 1С

- **Decision**: `fake_1cv8.sh` (тот же приём, что уже применён в
  `v8runner`/`v8storage`) расширяется одним поведением: при получении
  `-v N` в аргументах команды `/ConfigurationRepositoryDumpCfg` —
  завершается с кодом `0` (и пишет минимальное фиктивное содержимое в
  целевой файл), если `N <= ПОРОГ` (порог задаётся тестом через
  переменную окружения или отдельный аргумент), иначе — код `1`. Это
  даёт возможность e2e-тестам симулировать "хранилище с N версиями" без
  реальной платформы. Git-часть (`gitrunner`) тестируется РЕАЛЬНЫМ git,
  как и сам `gitrunner` ранее.
- **Rationale**: Тот же обоснованный компромисс, что уже принят для
  `v8runner`/`v8storage` — проверяем, что СТРОИТСЯ правильная командная
  строка/логика вокруг неё, не поведение реальной платформы (недоступно).
- **Alternatives considered**: Пропустить e2e-тесты цикла `sync` вообще
  (тестировать только `version_file.ps`/`authors_file.ps` как чистые
  функции) — отклонено: сам цикл (перебор версий до первой ошибки,
  обновление VERSION только после успешного коммита) — сердце фичи
  (FR-002/FR-003/Edge Case про прерывание), достоин e2e-проверки, а не
  только модульных тестов на файлы.

## 8. Импорт зависимостей — через `pan add`

- **Decision**: `gitsync` объявляет зависимости на `gitrunner`/`v8runner`/
  `v8storage`/`cli` в `pan.toml` (`pan add <абсолютный-путь-к-репо>
  ^0.1.1`), раскладываемые `pan`'ом в `модули/<имя>/`. Импорт в исходниках
  — путём внутрь `модули/`, к конкретному файлу пакета (не bare-имя
  пакета и не путь к каталогу): `импорт "модули/gitrunner/git.ps" как
  гит`, аналогично для `v8runner`/`configurator.ps`, `v8storage`/
  `storage_manager.ps`, `cli`/`флаги.ps`.
- **Rationale**: Проверено вживую (`pan init`+`pan add`+реальный импорт в
  тестовом каталоге) — единственный существующий тег каждого из 7 пакетов
  (`v0.1.0`) указывал на их ПЕРВОНАЧАЛЬНУЮ commit-заглушку (feature 004,
  scaffolding), реальная реализация (эта сессия) была НЕ ретегирована —
  `pan add ^0.1.0` резолвил бы старую заглушку. Исправлено ДО написания
  кода этой фичи: все 7 пакетов ретегированы `v0.1.1` (bump версии в
  `pan.toml` + коммит + тег на HEAD с реальной реализацией) — разовая,
  cheap, находится строго вне `panosiki/gitsync`, не расширяет сам пакет.
  Бонус находка: bare `импорт "gitrunner"` (без пути внутрь `модули/`) НЕ
  резолвится — resolver ищет либо файл `модули/gitrunner` (нет такого),
  либо каталог с канонiческим `индекс.ps` внутри (`модули/gitrunner/`
  содержит `start.ps` — точку входа пакета для `pan run`/`pan task`, а не
  `индекс.ps` библиотечной конвенции) — нужен явный путь до нужного .ps
  файла ЗАВИСИМОСТИ, что и используется здесь.
- **Alternatives considered**: Относительный импорт мимо `pan`
  (`импорт "../gitrunner/git.ps"`, все 7 пакетов — sibling-каталоги под
  `../panosiki/`) — рассмотрено, отклонено пользователем в пользу
  реального использования уже построенного `pan` (сам факт, что `pan
  add` для реальной пары пакетов сейчас работает end-to-end — ценная
  проверка работоспособности feature 003, а не только gitsync).

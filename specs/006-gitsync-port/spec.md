# Feature Specification: Портирование ядра gitsync на panos

**Feature Branch**: `006-gitsync-port`
**Created**: 2026-07-23
**Status**: Draft
**Input**: User description: "Портирование gitsync (git-based synchronization tool for 1C:Enterprise infobase storage, oscript-library/gitsync) на panos. Зависимости уже реализованы: panosiki/gitrunner, tempfiles, configor, cli-selector, v8runner, v8storage, cli, std/слог.ps. Нужно перенести саму бизнес-логику gitsync — цикл синхронизации git-репозитория с хранилищем конфигурации 1С."

## Контекст

Оригинал (`oscript-library/gitsync`, README проверен через `gh api`) — консольное
приложение + библиотека на OneScript, синхронизирующее хранилище конфигурации
1С с git-репозиторием. Полная функциональность v3.x значительно шире, чем
нужно для первой версии на panos — при подготовке спецификации объём
сознательно сужен (см. Assumptions), решения по каждому пункту сужения
согласованы с пользователем явно, не выбраны молча.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Синхронизация хранилища с git (Priority: P1) 🎯 MVP

Разработчик 1С вызывает `gitsync sync ПУТЬ_К_ХРАНИЛИЩУ РАБОЧИЙ_КАТАЛОГ` —
инструмент читает номер последней синхронизированной версии из файла
`VERSION` в рабочем каталоге, получает из хранилища конфигурации 1С каждую
ЕЩЁ не синхронизированную версию (от `VERSION+1` до последней версии
хранилища), для каждой версии: выгружает её в файлы исходников конфигурации
в рабочий каталог, коммитит изменения в git с автором, взятым из
соответствия `AUTHORS` (пользователь хранилища → имя+email git), обновляет
`VERSION` на номер только что закоммиченной версии. Если рабочий каталог —
git-репозиторий без коммитов, первая синхронизация коммитит с версии `1`
(или с номера, ранее установленного `set-version`).

**Why this priority**: Это и есть смысл существования gitsync — единственная
команда, ради которой создаются все остальные (init только готовит
окружение для sync, set-version — только корректирует его состояние).

**Independent Test**: Поднять bare git-репозиторий и файловое хранилище 1С
(тестовое, через уже реализованный `v8storage`) с 2-3 версиями конфигурации
— вызвать `sync`, проверить, что рабочий каталог содержит N коммитов (по
одному на версию), `VERSION` равен номеру последней версии хранилища, автор
коммитов соответствует записи `AUTHORS` для пользователя, под которым
выполнен ЭТОТ запуск `sync` (см. примечание про авторство ниже).

> **Примечание про авторство (уточнено при планировании)**: оригинальный
> gitsync сохраняет автора КАЖДОЙ ИСТОРИЧЕСКОЙ версии хранилища через MXL-отчёт
> хранилища, читаемый средствами платформы 1С изнутри (V8/COM-интероп) — у
> panos такого интеропа нет вообще (`v8runner`/`v8storage` — только
> `ос.выполнить`, командная строка). Поверсионное авторство поэтому НЕ
> входит в эту фичу (см. Assumptions) — ВСЕ коммиты одного запуска `sync`
> используют ОДНОГО автора: пользователя хранилища, под которым выполнен
> сам вызов `sync` (`--storage-user`), найденного в `AUTHORS`. Точное
> поверсионное авторство — отдельная будущая фича, если/когда понадобится.

**Acceptance Scenarios**:

1. **Given** хранилище с версиями 1..3, `VERSION` файла нет (первый запуск),
   **When** вызван `sync`, **Then** создаются 3 коммита (версии 1, 2, 3 по
   порядку), `VERSION` становится `3`, все 3 коммита — от одного автора
   (пользователь хранилища этого запуска `sync`).
2. **Given** та же настройка, но `VERSION` уже содержит `2` (версии 1-2 уже
   синхронизированы ранее), **When** вызван `sync`, **Then** создаётся
   ТОЛЬКО 1 новый коммит (версия 3), `VERSION` становится `3`.
3. **Given** хранилище БЕЗ новых версий сверх уже записанной в `VERSION`,
   **When** вызван `sync`, **Then** новых коммитов не создаётся, команда
   завершается успешно (это не ошибка — "уже всё синхронизировано").
4. **Given** пользователь хранилища, под которым выполнен `sync`,
   ОТСУТСТВУЕТ в `AUTHORS`, **When** вызван `sync`, **Then** коммиты всё
   равно создаются — автор = имя этого пользователя хранилища как есть,
   без email — не падает молча с пустым/некорректным автором.

### User Story 2 - Подготовка нового репозитория (Priority: P2)

Разработчик, у которого ещё нет локального git-репозитория для проекта,
вызывает `gitsync init ПУТЬ_К_ХРАНИЛИЩУ РАБОЧИЙ_КАТАЛОГ` — инструмент
инициализирует git-репозиторий в рабочем каталоге (`git init`, если там его
ещё нет) и создаёт заготовки `VERSION` (пусто/0 — синхронизировано ничего)
и `AUTHORS` (пустая, с комментарием про формат) в этом каталоге, если их там
ещё нет.

**Why this priority**: Нужен один раз на проект, но именно после него
можно вызывать `sync` из User Story 1 — без этого шага `sync` на пустом
каталоге падает с "не git-репозиторий".

**Independent Test**: Вызвать `init` на пустом каталоге — проверить, что
появились `.git/`, `VERSION` (с содержимым `0`), `AUTHORS` (пустой файл с
описанием формата в комментарии).

**Acceptance Scenarios**:

1. **Given** пустой каталог, **When** вызван `init`, **Then** каталог
   становится git-репозиторием с файлами `VERSION` (`0`) и `AUTHORS`
   (пустой шаблон).
2. **Given** каталог УЖЕ является git-репозиторием (повторный `init`),
   **When** вызван `init`, **Then** существующие `VERSION`/`AUTHORS` НЕ
   перезаписываются (не теряется уже накопленный прогресс синхронизации).

---

### User Story 3 - Ручная установка номера версии (Priority: P3)

Разработчик, который хочет начать синхронизацию НЕ с версии 1 хранилища (у
него уже есть история в git, начатая как-то иначе), вызывает
`gitsync set-version НОМЕР РАБОЧИЙ_КАТАЛОГ` — инструмент записывает
указанный номер в `VERSION` рабочего каталога, следующий `sync` начнёт с
`НОМЕР+1`.

**Why this priority**: Реже нужна (разовая настройка, не часть обычного
рабочего цикла), но без нехватки этой команды пользователь вынужден
редактировать `VERSION` вручную текстовым редактором — минорное неудобство,
не блокирующее основной сценарий.

**Independent Test**: Вызвать `set-version 5` в рабочем каталоге — проверить
`VERSION` содержит `5`, следующий `sync` начинает с версии 6 хранилища (не
перевыгружает 1-5).

**Acceptance Scenarios**:

1. **Given** рабочий каталог с `VERSION` = `2`, **When** вызван
   `set-version 10`, **Then** `VERSION` становится `10`.
2. **Given** тот же каталог, **When** после `set-version 10` вызывается
   `sync` на хранилище с версиями 1..12, **Then** синхронизируются только
   версии 11 и 12 (не 1..10 повторно).

---

### Edge Cases

- Что происходит, если `sync` прерывается (падает) НА СЕРЕДИНЕ выгрузки
  версии N (например, платформа 1С вернула ошибку) — `VERSION` не должен
  обновиться на N, пока коммит версии N не завершён успешно (иначе версия N
  "потеряется" — будет считаться синхронизированной, не будучи закоммиченной
  на самом деле).
- Что происходит, если рабочий каталог для `sync`/`set-version` — НЕ
  git-репозиторий (пользователь забыл `init`) — понятная ошибка
  ("выполните init"), не падение вглубь gitrunner/v8storage с
  низкоуровневым сообщением.
- Что происходит, если версия хранилища не содержит РЕАЛЬНЫХ изменений
  относительно предыдущей (например, версия создана без реальных правок
  конфигурации) — коммит всё равно создаётся (пустой diff — валидный,
  не ошибка; нумерация версий хранилища и коммитов должна оставаться
  взаимно-однозначной для последующих `sync`/`set-version`).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `sync` MUST прочитать текущий синхронизированный номер версии
  из `VERSION` рабочего каталога (`0`, если файла нет — ничего ещё не
  синхронизировано).
- **FR-002**: `sync` MUST получить список версий хранилища СТРОГО больше
  текущего синхронизированного номера, по возрастанию номера.
- **FR-003**: Для каждой такой версии `sync` MUST: выгрузить конфигурацию
  этой версии хранилища в файлы исходников в рабочий каталог, закоммитить
  изменения в git с сообщением, содержащим номер версии, и автором из
  `AUTHORS` (см. FR-005), обновить `VERSION` на номер этой версии — ТОЛЬКО
  после успешного коммита (см. Edge Cases).
- **FR-004**: `sync` MUST завершаться успешно (не ошибкой) и не создавать
  коммитов, если новых версий сверх уже синхронизированной нет.
- **FR-005**: `sync` MUST искать автора коммитов в `AUTHORS` по имени
  пользователя хранилища, под которым выполнен САМ ВЫЗОВ `sync`
  (`--storage-user`) — ОДИН автор на все коммиты одного запуска, не
  по-версионно (v8storage не даёт информации об авторе конкретной
  исторической версии — см. примечание в User Story 1); при отсутствии
  записи в `AUTHORS` — использовать
  имя пользователя хранилища как есть (без email), не падать и не
  подставлять неинформативного автора по умолчанию.
- **FR-006**: `init` MUST инициализировать git-репозиторий в рабочем
  каталоге, если он ещё не является таковым, и создать `VERSION`/`AUTHORS`
  с содержимым по умолчанию, ТОЛЬКО если эти файлы там ещё не существуют
  (см. Acceptance Scenario 2 в User Story 2).
- **FR-007**: `set-version` MUST перезаписать `VERSION` рабочего каталога
  указанным числом, отклонив (ошибка, не молчаливое исправление)
  нечисловой/отрицательный аргумент.
- **FR-008**: Все три команды (`sync`/`init`/`set-version`) MUST принимать
  путь к хранилищу и рабочий каталог как позиционные аргументы командной
  строки (через уже реализованный `panosiki/cli`), с учётными данными
  хранилища (пользователь/пароль) как флаги.
- **FR-009**: Все три команды MUST понимать переменные окружения как
  запасной источник значений для путей/учётных данных, не заданных
  флагом (соответствие `GITSYNC_*` → аргумент, см. Key Entities), той же
  семантики "флаг переопределяет переменную окружения", что в оригинале.

### Key Entities

- **Контекст синхронизации**: путь к хранилищу конфигурации 1С, рабочий
  каталог (git-репозиторий), пользователь/пароль хранилища, опционально —
  имя расширения конфигурации (не основная конфигурация, а конкретное
  расширение), опционально — путь к исполняемому файлу платформы 1С.
- **VERSION**: последний синхронизированный номер версии хранилища,
  хранится простым текстовым файлом в рабочем каталоге (число, без
  обёртки — переход с XML-формата оригинала, см. Assumptions).
  Обновляется ТОЛЬКО после успешного коммита соответствующей версии.
- **AUTHORS**: соответствие "имя пользователя хранилища 1С" →
  "отображаемое имя git" + "email", хранится TOML-файлом в рабочем
  каталоге (переход с INI-подобного формата оригинала, см. Assumptions).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Разработчик может синхронизировать N версий хранилища в N
  git-коммитов ОДНИМ вызовом `sync`, без ручного вмешательства между
  версиями.
- **SC-002**: Повторный вызов `sync` без новых версий хранилища не создаёт
  лишних коммитов и не выполняет повторную выгрузку уже
  синхронизированных версий — на практике измеримо как отсутствие diff в
  рабочем каталоге после повторного вызова.
- **SC-003**: Прерывание `sync` посреди выгрузки версии (симулированная
  ошибка платформы) не оставляет `VERSION` в состоянии, ссылающемся на
  незакоммиченную версию — следующий `sync` корректно повторяет
  прерванную версию, а не пропускает её.
- **SC-004**: `init` на уже существующем репозитории с прогрессом
  синхронизации не откатывает и не обнуляет этот прогресс (`VERSION`/
  `AUTHORS` не перезаписываются).

## Assumptions

- **Сужение объёма относительно оригинала** (согласовано с пользователем
  явно, не молчаливый выбор):
  - **Система плагинов/подписок на события** (оригинал: отдельный
    репозиторий `gitsync-plugins`, `gitsync plugins init/list/enable/
    disable/install/clear`) — НЕ входит в v1. Только ядро `sync`/`init`/
    `set-version`. Точки расширения (функции-колбэки до/после этапов
    синхронизации) допустимы как ЗАДЕЛ на будущее, если не усложняют
    основной путь — не обязательны для этой фичи.
  - **Отправка/получение из удалённого git-репозитория** (`git push`/
    `git pull`, в оригинале — отдельный плагин `sync-remote`) — НЕ
    автоматизируется этой фичей. Уже реализованный `gitrunner`
    (`panosiki/gitrunner`) даёт `отправить`/`получить` напрямую —
    пользователь вызывает их сам (или отдельным скриптом) после `sync`,
    не через встроенную в gitsync команду.
  - **Хранилище по протоколу http/tcp** — вне scope. Уже реализованный
    `v8storage` работает только с файловыми путями хранилища (см. его
    собственный suженный scope, `panosiki/v8storage`).
  - **Одновременная синхронизация нескольких хранилищ** (команда `all`) —
    не поддерживалась даже в оригинальном v3.x ("Пока не поддерживается")
    — не входит и здесь.
  - **Формат `VERSION`** — простой текстовый файл с числом, а не XML
    (`<VERSION>10</VERSION>` оригинала) — в panos нет XML-парсера, для
    одного скалярного значения он избыточен.
  - **Формат `AUTHORS`** — TOML, а не INI-подобный формат оригинала (в
    panos нет INI-парсера, есть `std/кодирование/toml.ps`) — редизайн,
    как и `configor`/`v8storage` ранее в этом проекте, не совместим с
    готовыми AUTHORS-файлами из реальных проектов на оригинальном
    gitsync.
  - **Поверсионное авторство коммитов** (найдено при планировании, не
    было очевидно на этапе `/speckit.specify`) — оригинал сохраняет
    автора КАЖДОЙ исторической версии хранилища через MXL-отчёт,
    читаемый средствами платформы 1С ИЗНУТРИ (V8/COM-интероп) —
    архитектурно недоступно `v8runner`/`v8storage` (только
    `ос.выполнить`, командная строка, никакого встроенного
    script-интерпретатора платформы). В этой фиче ВСЕ коммиты одного
    запуска `sync` — от ОДНОГО автора (пользователь хранилища самого
    запуска, см. FR-005/User Story 1). Точное поверсионное авторство —
    отдельная будущая фича (реального объёма расширения v8storage под
    MXL/COM-интероп, непроверяемая без установленной платформы 1С).
- **Зависимости уже реализованы и не меняются этой фичей**: `panosiki/
  gitrunner` (git), `panosiki/v8runner` (Конфигуратор — выгрузка в
  файлы), `panosiki/v8storage` (Хранилище — версии), `panosiki/cli`
  (флаги командной строки), `std/слог.ps` (логирование), `std/
  кодирование/toml.ps` (AUTHORS). Если `sync` потребует функциональность,
  которой у `gitrunner`/`v8storage`/`v8runner` сейчас нет (например,
  явный автор коммита — `закоммитить` этих пакетов сейчас не принимает
  автора), это РАСШИРЕНИЕ существующего пакета, определяется на этапе
  `/speckit.plan`, не переизобретение с нуля.
- **1С-специфика непроверяема в песочнице разработки** — как и
  `v8runner`/`v8storage` ранее, реального взаимодействия с платформой 1С
  не будет (нет установленной платформы) — тестируется тем же приёмом
  (подменный исполняемый файл), что уже применялось для этих двух
  пакетов; акцент тестирования — на git-стороне (`gitrunner`, реальный
  git) и на самой логике синхронизации (номера версий, AUTHORS-lookup,
  VERSION-обновление), которая тестируема без реальной 1С.
- Живёт как отдельный пакет `panosiki/gitsync` (новый git-репозиторий,
  тот же паттерн, что остальные 7 пакетов), НЕ модифицирует уже
  реализованные `gitrunner`/`v8runner`/`v8storage`/`cli`/`configor`/
  `cli-selector`/`tempfiles`/`слог`, кроме точечного расширения (см.
  выше про автора коммита), если оно понадобится.

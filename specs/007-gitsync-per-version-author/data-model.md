# Data Model: gitsync — поверсионное авторство коммитов

## `скобки.Значение` (`panosiki/скобки/скобки.ps`)

```panos
экспорт тип Значение = вариант
    Лист(Строка)
    Список(Массив(Значение))
конец
```

Универсальное дерево скобочного формата 1С — лист (голый/строковый токен
как текст) или вложенный список. Не специфично для MXL — общего назначения
представление, как `томл.Значение` для TOML.

## `скобки.ЗаписьВерсии` (`panosiki/скобки/отчёт_версий.ps`)

```panos
экспорт тип ЗаписьВерсии = структура
    номер:       Число
    автор:       Строка   // "Пользователь:" в отчёте — имя пользователя ХРАНИЛИЩА, не git-автор
    дата:        Строка   // "Дата создания:", как есть из отчёта (ДД.ММ.ГГГГ)
    время:       Строка   // "Время создания:", как есть (ЧЧ:ММ:СС)
    комментарий: Строка
конец
```

Одна запись = одна версия хранилища из диапазона отчёта. `автор` —
исходное имя пользователя хранилища, ЕЩЁ НЕ сопоставленное с `AUTHORS` (это
делает `gitsync`, не этот пакет — пакет `скобки` не знает о `AUTHORS`/git
вообще, только парсит MXL).

## `v8storage.Хранилище.отчёт_по_версиям` — новый метод

Не новая структура — метод на существующем `Хранилище` (см. research.md
§6), сигнатура:

```panos
функ отчёт_по_версиям(это: Хранилище, начиная_с: Строка, до: Строка, путь_к_файлу: Строка) -> Результат(Строка, Ошибка)
```

Возвращает вывод команды (как остальные методы `Хранилище`) — САМ MXL
пишется в `путь_к_файлу` (побочный эффект команды `1cv8`, не возврат
функции — тот же паттерн, что `версию_в_файл`).

## `gitsync.sync` — изменение потока данных

**Было (006)**: один `автор: af.Автор`, резолвленный один раз до цикла по
версиям (`контекст.пользователь_хранилища` → `authors_file.найти_автора`),
используется для ВСЕХ коммитов запуска.

**Стало**:

1. До цикла — ОДИН вызов `v8storage.отчёт_по_версиям` на весь диапазон
   известных версий → `Результат(Строка, Ошибка)` (путь к MXL).
2. Если успех — `скобки.разобрать` → `отчёт_версий.прочитать_отчёт` →
   `Массив(ЗаписьВерсии)` → индексируется в `Соответствие(Число,
   ЗаписьВерсии)` по номеру версии (для O(1) lookup внутри цикла).
3. Если отчёт недоступен ЦЕЛИКОМ (шаг 1 — Результат.Неудача) —
   `Соответствие` остаётся пустым, ВСЕ версии идут по fallback-пути (см.
   ниже) — эквивалентно поведению 006.
4. Внутри цикла по версиям, для каждой версии N:
   - если `Соответствие` содержит запись для N — резолвить автора через
     `authors_file.найти_автора(авторы, запись.автор)` (или fallback на
     `запись.автор` как есть, если в `AUTHORS` нет записи — тот же паттерн
     FR-005 из 006, но по имени пользователя ЭТОЙ версии, не контекста);
   - иначе (нет записи для N, включая случай "отчёт целиком недоступен") —
     fallback: `authors_file.найти_автора(авторы, контекст.пользователь_
     хранилища)` — прежнее (006) поведение, единый автор запуска, ТОЛЬКО
     для этой версии.

`Контекст_Синхронизации` (006) не меняется структурно — `пользователь_
хранилища` теперь используется В ДВУХ ролях: как раньше (логин в
хранилище) и как fallback-имя автора, если поверсионный отчёт недоступен
для конкретной версии.

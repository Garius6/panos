# Research: gitsync — поверсионное авторство коммитов

## §1: Флаги `/ConfigurationRepositoryReport`

**Decision**: `/ConfigurationRepositoryReport <путь> -NBegin <n> -NEnd <n>` —
БЕЗ `-GroupByObject`/`-GroupByComment` (это альтернативные режимы отчёта с
другой структурой — не тот, что понимает `yabr`'s `ЧтениеОтчетаПоВерсиямХранилища`).

**Rationale**: Реальный пример конфигурации `yabr` (`examples/ver-rep/
ver-rep2jsonfile.json`, репозиторий `oscript-library/yabr`) + реальный
образец MXL (`CR_versions_small.mxl`, тот же репозиторий) подтверждают
именно этот, "плоский" построчный формат ("Версия:"/"Пользователь:"/"Дата
создания:"/"Комментарий:"/"Изменены:"/"Добавлены:"/"Удалены:" как отдельные
записи по каждой версии) — это и есть формат по умолчанию, без флагов
группировки.

## §2: MXL-преамбула перед скобочным текстом

**Decision**: MXL-файл начинается с короткой сигнатуры `MOXCEL` +
несколько байт + таб + BOM ПЕРЕД собственно скобочным текстом (подтверждено
реальным образцом — `head -c 2000 CR_versions_small.mxl` показывает
`MOXCEL   <tab><BOM>{8,1,9,...`). Парсер MUST найти ПЕРВЫЙ символ `{` в
файле и начинать разбор скобочного формата с него — не хардкодить длину
преамбулы (более устойчиво к вариациям версии платформы).

**Alternatives considered**: Хардкодить фиксированное смещение (например,
"пропустить первые N байт") — отклонено, слишком хрупко, длина преамбулы не
гарантированно постоянна между версиями платформы.

## §3: Грамматика скобочного формата 1С

**Decision**: Рекурсивный `{...}`-формат, подтверждено реальным рабочим
кодом `oscript-library/yabr` (`ЧтениеСкобкоФайла.os`, `ПрочитатьДанныеСтроки`):

- `{` открывает вложенный список, `}` закрывает — элемент списка либо ЛИСТ
  (строка/число как текст), либо ВЛОЖЕННЫЙ список.
- Значения разделены `,`.
- Строки — в двойных кавычках `"..."`, экранирование кавычки внутри строки —
  удвоением (`""` → один `"` в значении).
- Строка внутри кавычек может физически переноситься на следующую строку
  файла (кавычки остаются "открытыми" до закрывающей) — при построчном
  чтении незакрытые кавычки склеиваются со следующей строкой файла (с
  переводом строки внутри значения) до момента закрытия.
- Нестроковые (голые) токены — тоже просто текст на panos-стороне;
  интерпретация "это число" — на совести вызывающего кода (например,
  `строки.в_число`), не парсера.

**API-дизайн** (`panosiki/скобки/скобки.ps`):

```panos
экспорт тип Значение = вариант
    Лист(Строка)
    Список(Массив(Значение))
конец

экспорт функ разобрать(текст: Строка) -> Результат(Значение, Ошибка)
```

**Rationale**: Порт этой грамматики — механическая работа (посимвольный
разбор, состояние "кавычки открыты/закрыты", рекурсия на `{`/`}`) — не
завязан на специфику MXL, применим к любому файлу 1С в скобочном формате.
ADT `Значение` (Лист/Список) — тот же паттерн, что уже устоялся в
`std/кодирование/toml.ps`'s `Значение` (Строка/Целое/Число/Булево/Время/
Список/Таблица) — знакомый вызывающему коду стиль.

## §4: MXL cell-addressing (адресация ячеек таблицы внутри дерева)

**Decision**: Порт КОНКРЕТНЫХ "магических" констант из `yabr`'s
`ЭтоЯчейкаТаблицы`/`ПолучитьОписаниеЯчейки`/`КоличествоСтрокВТаблице`
ДОСЛОВНО (не переизобретать) — уровень элементов дерева, представляющих
ячейку таблицы (`УровеньЭлементов = 4`, подтверждено реальным примером
конфигурации `yabr`, `"УровниЗаписей":[4]`), и относительные смещения
индексов (`ИндексКоличестваСтрок = 15`, `СдвигИндексаНач = 4`,
`СдвигИндекса1..3 = 1..3`) внутри контейнера уровня 1 — это
реверс-инжиниринг конкретного внутреннего формата MXL-табличного документа
1С, накопленный за годы использования `yabr` в реальных проектах — риск
переизобретения своей версии этих смещений СУЩЕСТВЕННО выше, чем риск
дословного портирования уже работающих значений.

**Известный риск** (честно, не скрывается): эти смещения — эмпирически
найденные константы конкретного внутреннего представления MXL, а не
задокументированный 1С формат. Если платформа 1С в будущей версии изменит
внутреннюю структуру MXL — константы могут перестать работать. Это
касается ОРИГИНАЛЬНОГО `yabr` ровно так же (не новый риск, привнесённый
портированием) — если/когда 1С сломает совместимость, придётся обновлять
и `yabr`, и наш порт синхронно.

## §5: Собственно чтение записи по версии (state-machine)

**Decision**: Порт `ДобавитьЗапись`-подобной логики (`yabr`'s
`ЧтениеОтчетаПоВерсиямХранилища.os`) — обходит ячейки таблицы (`§4`) по
порядку (строка/столбец), распознаёт метки `"Версия:"` → следующая ячейка =
Номер, `"Пользователь:"` → Автор, `"Дата создания:"` → Дата, `"Время
создания:"` → Время, `"Комментарий:"` → Комментарий (`"Изменены:"`/
`"Добавлены:"`/`"Удалены:"` — списки объектов, НЕ нужны для авторства,
портируются по остаточному принципу, если не усложняют — см. Simplicity
First, не блокируют MVP если пропущены).

**API-дизайн** (`panosiki/скобки/отчёт_версий.ps`):

```panos
экспорт тип ЗаписьВерсии = структура
    номер:       Число
    автор:       Строка
    дата:        Строка
    время:       Строка
    комментарий: Строка
конец

экспорт функ прочитать_отчёт(значение: скобки.Значение) -> Массив(ЗаписьВерсии)
```

## §6: Новый метод `v8storage`

**Decision**:

```panos
/// Сохраняет отчёт по версиям хранилища [начиная_с; до] в MXL-файл (/ConfigurationRepositoryReport).
функ отчёт_по_версиям(это: Хранилище, начиная_с: Строка, до: Строка, путь_к_файлу: Строка) -> Результат(Строка, Ошибка)
    это.выполнить_команду(массив("/ConfigurationRepositoryReport", путь_к_файлу, "-NBegin", начиная_с, "-NEnd", до))
конец
```

Тот же паттерн, что уже существующий `версию_в_файл` — без изменений в
существующих методах `Хранилище` (Surgical Changes).

## §7: Тестирование — синтетический fixture, не вендоринг `yabr`'s образца

**Decision**: `fake_1cv8.sh` (в `v8storage`) расширяется — при
`/ConfigurationRepositoryReport` пишет СВОЙ (написанный для этой фичи,
короткий) MXL-подобный текст в скобочном формате, СТРУКТУРНО совпадающий с
реальным образцом (`MOXCEL`-преамбула + `"Версия:"/"Пользователь:"/...`
записи на нужном уровне вложенности), но НЕ являющийся копией реального
`CR_versions_small.mxl` из `yabr` (тот файл — часть репозитория `yabr`,
MPL-2.0 — копировать его в тесты этого пакета означало бы дистрибуцию
чужого Covered Software без необходимости, когда достаточно СВОЕГО
минимального фикстура с той же структурой).

**Известное ограничение** (та же оговорка, что весь проект): фикстур
доказывает, что парсер ПРАВИЛЬНО читает скобочный формат СТРУКТУРЫ,
подтверждённой реальным образцом `yabr` — но не может доказать, что
РЕАЛЬНАЯ команда `/ConfigurationRepositoryReport` на РЕАЛЬНОЙ платформе 1С
выдаст БАЙТ-В-БАЙТ такую же структуру — это по-прежнему требует
тестирования на реальной платформе (недоступной в этой песочнице).

## §8: Fallback при недоступности отчёта (FR-004)

**Decision**: `sync.ps` вызывает `v8storage.отчёт_по_версиям(...)` ОДИН РАЗ
перед циклом по версиям (диапазон — весь набор известных версий). Если
вызов вернул `Результат.Неудача` — используется прежнее (006) поведение:
единый автор запуска для ВСЕХ версий, с сообщением в вывод. Если вызов
успешен, но КОНКРЕТНАЯ версия отсутствует в разобранном отчёте (`Массив
ЗаписьВерсии` не содержит её номер) — fallback ТОЛЬКО для этой версии
(единый автор запуска), остальные версии с найденными записями — как
обычно.

## §9: Регистронезависимый lookup в AUTHORS (FR-005)

**Decision**: `authors_file.найти_автора` (уже существующий в
`panosiki/gitsync`, из 006) MUST сравнивать имя пользователя
регистронезависимо — правка СУЩЕСТВУЮЩЕЙ функции (не новая перегрузка),
обратно совместима: существующие вызовы с точным регистром продолжают
работать, плюс теперь матчат и другой регистр. Затрагивает
`panosiki/gitsync/authors_file.ps` (пакет из 006) — единственная точка
пересечения этой фичи с уже смёрженным кодом 006, помимо самого `sync.ps`.

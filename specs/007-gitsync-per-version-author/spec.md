# Feature Specification: gitsync — поверсионное авторство коммитов

**Feature Branch**: `007-gitsync-per-version-author`
**Created**: 2026-07-23
**Status**: Draft
**Input**: User description: "Расширить panosiki/v8storage методом, оборачивающим /ConfigurationRepositoryReport (командная строка 1cv8) для генерации MXL-отчёта по версиям хранилища; добавить новый парсер скобочного формата 1С (подтверждено реальным кодом oscript-library/yabr, MPL-2.0) для извлечения автора/даты/комментария по номеру версии; подключить в panosiki/gitsync/sync.ps вместо единого автора на весь запуск. Без COM, кроссплатформенно, через ос.выполнить + текстовый парсер."

## Контекст (найдено при исследовании, не гадание)

`specs/006-gitsync-port/spec.md` (Assumptions) зафиксировал, что поверсионное
авторство НЕВОЗМОЖНО без MXL/COM-интеропа, которого у `v8runner`/`v8storage`
нет. Это оказалось НЕ ТОЧНО: `/ConfigurationRepositoryReport` (командная
строка `1cv8`, тот же паттерн `ос.выполнить`, что весь `v8storage`) генерирует
MXL-файл отчёта по версиям хранилища — а сам MXL, вопреки предположению,
не бинарный OLE-формат, а **скобочный формат** 1С (текстовая рекурсивная
`{...}`-сериализация, используемая много где в экосистеме 1С — журналы
регистрации, настройки кластера, MXL-отчёты). Подтверждено реальным рабочим
кодом `oscript-library/yabr` (MPL-2.0) — его модуль
`ЧтениеОтчетаПоВерсиямХранилища.os` построчно читает из такого MXL поля
`"Версия:"` → Номер, `"Пользователь:"` → Автор, `"Дата создания:"`/`"Время
создания:"` → Дата/Время, `"Комментарий:"` → Комментарий, `"Изменены:"`/
`"Добавлены:"`/`"Удалены:"` → списки объектов — без единого обращения к COM.

Значит поверсионное авторство достижимо тем же способом, что весь остальной
`v8storage`/`gitrunner`: `ос.выполнить` + текстовый парсер, кроссплатформенно,
без нового native-слоя.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Правильный автор каждого коммита (Priority: P1)

Пользователь синхронизирует несколько версий хранилища за один `sync`, где
разные версии были сохранены разными пользователями хранилища 1С. После
`sync` каждый git-коммит подписан ИМЕНЕМ ТОГО ПОЛЬЗОВАТЕЛЯ, который реально
сохранил ИМЕННО ЭТУ версию в хранилище (сопоставленным с `AUTHORS`), а не
единым автором всего запуска `sync` (как сейчас, см. `specs/
006-gitsync-port`).

**Why this priority**: Это единственная причина существования фичи —
исправление намеренно суженного места в 006 (единый автор на весь запуск),
теперь как отдельная, независимо тестируемая фича.

**Independent Test**: Хранилище (fake-бинарь) содержит 3 версии, сохранённые
тремя РАЗНЫМИ пользователями (согласно фиктивному MXL-отчёту) — после `sync`
`git log --format='%an %ae'` показывает 3 РАЗНЫХ автора, совпадающих с
`AUTHORS`-записями для каждого из трёх пользователей хранилища.

**Acceptance Scenarios**:

1. **Given** хранилище с версиями 1-3, сохранёнными пользователями
   `Иванов`/`Петров`/`Иванов` соответственно, и `AUTHORS` с записями для
   обоих, **When** выполняется `sync`, **Then** создаются 3 коммита, автор
   коммита версии 1 и 3 — `Иванов` (email из `AUTHORS`), автор коммита
   версии 2 — `Петров` (email из `AUTHORS`).
2. **Given** версия хранилища сохранена пользователем БЕЗ записи в
   `AUTHORS`, **When** синхронизируется эта версия, **Then** коммит всё
   равно создаётся — автор = имя этого пользователя как есть (тот же
   fallback, что уже был в 006 FR-005), без паники.
3. **Given** отчёт по версиям хранилища НЕДОСТУПЕН (`/
   ConfigurationRepositoryReport` завершился с ошибкой — например, версия
   платформы 1С не поддерживает эту команду), **When** выполняется `sync`,
   **Then** `sync` НЕ падает — используется fallback на прежнее поведение
   006 (единый автор запуска, `контекст.пользователь_хранилища`) для ВСЕХ
   версий этого запуска, с явным уведомлением в выводе `sync`, что
   поверсионное авторство недоступно в этом запуске.

---

### Edge Cases

- Что если отчёт по версиям хранилища не содержит записи для КОНКРЕТНОЙ
  синхронизируемой версии (отчёт запрошен диапазоном `-NBegin`/`-NEnd`, не
  совпадающим с реально найденными версиями)? → та же fallback-логика
  Acceptance Scenario 3, но ТОЛЬКО для этой одной версии, не для всего
  запуска (остальные версии с найденными записями синхронизируются с
  правильным автором).
- Что если пользователь хранилища из отчёта совпадает по написанию с
  `контекст.пользователь_хранилища`, но их регистр букв отличается? → 1С
  имена пользователей хранилища регистронезависимы на практике — сравнение
  для lookup в `AUTHORS` MUST быть регистронезависимым (иначе один и тот же
  реальный человек получит два разных синтетических автора в разных
  версиях).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `v8storage` MUST получить новый метод, оборачивающий
  `/ConfigurationRepositoryReport <путь> -NBegin <n> -NEnd <n>` (тот же
  `выполнить_команду`-паттерн, что остальные методы `Хранилище`) —
  генерирует MXL-файл отчёта по диапазону версий.
- **FR-002**: Новый парсер скобочного формата 1С MUST извлекать из
  сгенерированного MXL, для каждого номера версии в запрошенном диапазоне:
  имя пользователя хранилища (поле "Пользователь:"), при наличии — дату/
  время/комментарий (не обязательны для авторства, но естественная часть
  того же парсинга).
- **FR-003**: `gitsync sync` MUST для КАЖДОЙ синхронизируемой версии
  использовать автора ИЗ ЭТОГО ОТЧЁТА (найденного через `authors_file.
  найти_автора` по имени пользователя хранилища ЭТОЙ версии), а не единого
  автора запуска (замена текущего поведения из `specs/006-gitsync-port`
  FR-005).
- **FR-004**: При недоступности отчёта (ошибка команды, версия платформы не
  поддерживает флаг, версия отсутствует в отчёте) `sync` MUST не падать —
  используется fallback на `контекст.пользователь_хранилища` как единый
  автор ТОЛЬКО для затронутых версий (см. Edge Cases), с явным сообщением
  в выводе.
- **FR-005**: Сопоставление имени пользователя хранилища (из отчёта или из
  контекста) с записями `AUTHORS` MUST быть регистронезависимым.

### Key Entities

- **Отчёт по версиям хранилища**: MXL-файл (скобочный формат), диапазон
  версий, для каждой версии — пользователь/дата/время/комментарий/списки
  изменённых-добавленных-удалённых объектов.
- **Запись авторства версии**: номер версии → имя пользователя хранилища
  (используется для lookup в `AUTHORS`, как раньше единый
  `контекст.пользователь_хранилища`).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: После `sync` с версиями от разных пользователей хранилища
  `git log` показывает РАЗНЫХ авторов, соответствующих реальному
  пользователю КАЖДОЙ версии — не единого автора запуска.
- **SC-002**: Недоступность отчёта по версиям НЕ прерывает `sync` — версии
  всё равно синхронизируются (с fallback-автором), `sync` завершается
  успешно.

## Assumptions

- Формат вывода `/ConfigurationRepositoryReport` (без `-GroupByObject`/
  `-GroupByComment` — они меняют структуру отчёта) даёт вид, совместимый с
  тем, что реально парсит `yabr` — точный набор флагов уточняется на этапе
  `/speckit.plan` (research.md), непроверяемо без реальной платформы 1С
  (та же оговорка, что у всего проекта — тестируется через fake-бинарь).
- Новый парсер скобочного формата — НЕ полный универсальный порт `yabr`
  (который читает журналы регистрации/настройки кластера/PFF), а только
  тот срез, который нужен для отчёта по версиям хранилища — остальные
  форматы `yabr` вне scope этой фичи.
- Живёт как отдельный пакет `panosiki/`, отдельным от `v8storage`
  (расширяет его методом) и НЕ модифицирует `gitrunner`/`v8runner`/`cli`/
  `tempfiles` — точечное расширение `v8storage` + новый пакет-парсер +
  изменение `panosiki/gitsync/sync.ps`.
- Атрибуция `yabr` (MPL-2.0) — `NOTICE.md` в новом пакете-парсере, тот же
  паттерн, что уже сделан для 8 предыдущих портов.
- Real-platform тестирование остаётся невозможным в этой песочнице — то же
  ограничение, что весь проект (`fake_1cv8.sh`, расширенный под новую
  команду).

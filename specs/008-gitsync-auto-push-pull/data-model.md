# Data Model: gitsync — авто push/pull через --remote

## `Контекст_Синхронизации` — новое поле

```panos
экспорт тип Контекст_Синхронизации = структура
    путь_хранилища:         Строка
    рабочий_каталог:        Строка
    пользователь_хранилища: Строка
    пароль_хранилища:       Строка
    имя_расширения:         Строка
    путь_к_платформе:       Строка
    remote:                 Опция(Строка)   // НОВОЕ: Нет() — поведение 007 без изменений
конец
```

## `Результат_Синхронизации` — без изменений

```panos
экспорт тип Результат_Синхронизации = структура
    синхронизировано_версий: Число
    итоговая_версия:          Число
конец
```

Push-ошибка (FR-004) не добавляет новое поле — она превращает ВЕСЬ
вызов `синхронизировать` в `Результат.Неудача`, а не частичный успех
внутри `Результат_Синхронизации` (см. research.md §3 — VERSION/коммиты
уже сохранены к этому моменту, откат не нужен, просто сигнал ошибки
наружу).

## `sync.синхронизировать` — новый поток (поверх data-model.md `007`)

1. Проверка "это git-репозиторий" (без изменений, `006`).
2. **НОВОЕ**: если `контекст.remote.есть()` — `git pull --ff-only
   <remote> <текущая ветка>`. Ошибка — немедленный `Результат.Неудача`,
   ничего дальше не выполняется (Acceptance Scenario 3).
3. Чтение `VERSION` (ПОСЛЕ pull — см. research.md §4).
4. Резолюция авторов, цикл по версиям хранилища (без изменений, `007`).
5. **НОВОЕ**: если `контекст.remote.есть()` И `синхронизировано > 0` —
   `git push <remote> <текущая ветка>`. Ошибка — `Результат.Неудача`
   (но `VERSION`/коммиты уже сохранены на шаге 4, не откатываются).
6. `Результат.Успех(Результат_Синхронизации(...))`.

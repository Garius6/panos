# Research: gitsync — авто push/pull через --remote

## §1: Команды

**Decision**: `git pull --ff-only <remote> <ветка>` перед циклом, `git
push <remote> <ветка>` (БЕЗ `--force`) после — оба через `Репозиторий.
выполнить_команду` (escape hatch), ветка — `получить_текущую_ветку()`.

**Rationale**: `--ff-only` — уже штатный флаг `git pull`, не требует
ничего от `gitrunner`, кроме уже существующего `выполнить_команду`.
Ненулевой код возврата (расхождение/конфликт/нет такого remote) уже
превращается в `Результат.Неудача` этим же методом — ничего разбирать
дополнительно не нужно, просто проверить `.ошибка()`.

## §2: Определение "были ли новые коммиты"

**Decision**: переиспользовать УЖЕ существующий счётчик `синхронизировано`
внутри `синхронизировать` (см. `sync.ps`, тот же, что попадает в
`Результат_Синхронизации.синхронизировано_версий`) — push вызывается,
только если `синхронизировано > 0`, ПОСЛЕ завершения цикла по версиям.

**Rationale**: Не нужно отдельно спрашивать git "есть ли неотправленные
коммиты" (`git log <remote>/<ветка>..HEAD` и т.п.) — ответ уже известен
из локального состояния цикла синхронизации.

## §3: Обработка ошибки push (FR-004)

**Decision**: Push вызывается ПОСЛЕ того, как `VERSION` уже записан на
диск и функция готова вернуть `Результат.Успех` — если push падает,
`синхронизировать` возвращает `Результат.Неудача` С СООБЩЕНИЕМ ОБ
ОШИБКЕ PUSH, но НЕ трогает уже записанный `VERSION`/уже сделанные
коммиты (никакого отката).

**Rationale**: Локальная синхронизация с хранилищем 1С УЖЕ произошла
успешно к этому моменту — откатывать её из-за сетевой проблемы push
означало бы терять реально сделанную работу без причины. Пользователь
может повторить `sync --remote` позже (0 версий синхронизируется
повторно — только push retry, известный сценарий 006's "нечего
синхронизировать").

## §4: Где именно pull в потоке (до/после чтения VERSION)

**Decision**: Pull выполняется В САМОМ НАЧАЛЕ `синхронизировать`, СРАЗУ
после проверки "это git-репозиторий", ДО чтения `VERSION` с диска.

**Rationale**: Если pull подтягивает более новый `VERSION` (записанный
кем-то другим на другой машине через `push`), локальный `sync` должен
видеть АКТУАЛЬНОЕ состояние файла, а не устаревшее — иначе можно
случайно повторно "просинхронизировать" уже перенесённые версии
(дублирующиеся коммиты).

## §6: Найдено при реализации — имя ветки через symbolic-ref, не gitrunner.получить_текущую_ветку()

**Decision**: Не использовать существующий `gitrunner.получить_текущую_
ветку()` (`git rev-parse --abbrev-ref HEAD`) — вместо него `sync.ps`
получает имя ветки через собственный escape-hatch вызов `git
symbolic-ref --short HEAD`.

**Rationale**: `repo_init.ps` (006) НЕ делает начальный коммит —
`VERSION`/`AUTHORS` пишутся на диск, но HEAD остаётся "непорождённым"
(unborn) до первого реального коммита версии хранилища. `rev-parse
--abbrev-ref HEAD` требует разрешить HEAD до существующего коммита и
падает (код 128) на unborn HEAD — а именно это и есть состояние САМОГО
ПЕРВОГО запуска `sync --remote` в жизни репозитория. `symbolic-ref
--short HEAD` читает ИМЯ ветки из символической ссылки напрямую, не
требуя существования коммита — работает одинаково для unborn и
"нормального" HEAD. Не расширение `gitrunner` (тот же принцип, что
`--ff-only`) — узкий частный случай именно для этого потока.

## §7: Найдено при реализации — unborn HEAD + непустой remote = untracked-конфликт

**Decision**: Если удалённая ветка СУЩЕСТВУЕТ, а локальный HEAD
"непорождённый" (0 коммитов) — вместо `git pull --ff-only` выполняется
`git fetch <remote> <ветка>` + `git checkout -B <ветка> -f
<remote>/<ветка>` (силовой чекаут, ветка целиком принимает состояние
remote).

**Rationale**: Обнаружено при ручной проверке quickstart.md — сценарий
"второй разработчик клонирует уже существующий (непустой) remote,
делает `gitsync init` локально, затем `sync --remote`" ломался: `init`
создаёт `VERSION`/`AUTHORS` НЕОТСЛЕЖИВАЕМЫМИ (без коммита, см. §6) — а
обычный `git pull` ОТКАЗЫВАЕТСЯ, если входящий коммит перезаписал бы эти
неотслеживаемые локальные файлы ("Please move or remove them before you
merge"). Поскольку локально НЕТ НИ ОДНОГО коммита (нечего терять — сами
"непорождённые" `VERSION`/`AUTHORS` от `init` — это просто заготовка,
подлежащая замене состоянием remote в любом случае), принудительный
чекаут безопасен ИМЕННО в этом единственном случае — не применяется,
если у локальной ветки УЖЕ есть хотя бы один коммит (там остаётся
`--ff-only`, полная защита от потери данных).

## §8: Найдено при реализации — VERSION писался ПОСЛЕ коммита (баг с 006)

**Decision**: `vf.записать_версию(...)` теперь вызывается ДО
`закоммитить_версию` внутри цикла по версиям (было — после).

**Rationale**: Обнаружено при тестировании push/pull (два клона,
`git log`/`git push` реально проверяют закоммиченное содержимое, а не
только локальный диск, впервые в проекте) — старый порядок писал
`VERSION` на диск ПОСЛЕ `git add . && git commit` этой же версии, так
что `VERSION` каждой версии попадал в git-историю с ОТСТАВАНИЕМ на одну
итерацию, а `VERSION` ПОСЛЕДНЕЙ версии вообще никогда не коммитился
(оставался "грязным" изменением рабочего каталога). Раньше это было
незаметно — `vf.прочитать_версию()` всегда читает файл С ДИСКА напрямую
(не из git), поэтому для одного непрерывно существующего локального
чекаута (единственный сценарий, который тестировали 006/007) результат
был "правильным" случайно. Push/clone впервые проверяют закоммиченное
СОСТОЯНИЕ, а не файл на диске — там расхождение сразу проявилось (второй
клон получал `VERSION` на 1 меньше реального).

## §5: Тестирование — два локальных клона + bare-репозиторий

**Decision**: Как в quickstart.md — `git init --bare` как "remote",
два независимых рабочих каталога, каждый — `git clone <bare>`, оба
управляются через `sync.ps` напрямую (без CLI, как и текущие тесты
`test_sync.ps`). Весь git — настоящий, никаких fake-бинарей (это НЕ
1С-специфичная часть).

**Rationale**: Сеть/аутентификация не нужны для `file://`-путей —
локальный bare-репозиторий полностью достаточен для честной проверки
pull/push поведения, включая расхождение (два клона коммитят
независимо до синка).

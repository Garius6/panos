# Feature Specification: gitsync — авто push/pull через --remote

**Feature Branch**: `008-gitsync-auto-push-pull`
**Created**: 2026-07-23
**Status**: Draft
**Input**: User description: "Опциональный флаг --remote <имя> у gitsync sync: перед циклом синхронизации — git pull --ff-only <remote> <текущая ветка> (аборт sync с ошибкой при расхождении/конфликте, без частичного прогресса); после цикла — git push <remote> <текущая ветка> ТОЛЬКО если были новые коммиты. Без флага — поведение не меняется (только локальные коммиты, как сейчас). Использует существующий gitrunner (escape hatch выполнить_команду для --ff-only, получить_текущую_ветку для имени ветки) — без изменений в gitrunner."

## Контекст

`specs/006-gitsync-port/spec.md` (Assumptions) исключил автоматический
push/pull из scope: "пользователь вызывает `gitrunner` напрямую сам после
`sync`". Эта фича добавляет ОПЦИОНАЛЬНУЮ автоматизацию (флаг, не
поведение по умолчанию) — решение по трём развилкам зафиксировано при
обсуждении с пользователем:
1. Опционально через флаг `--remote <имя>` (не всегда включено).
2. Pull — ТОЛЬКО fast-forward (не merge/rebase) — при расхождении `sync`
   останавливается с ошибкой, без частичного прогресса.
3. Push — только если были новые коммиты (не всегда).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Синхронизация с удалённым репозиторием одной командой (Priority: P1)

Пользователь хочет, чтобы `gitsync sync --remote origin` не только
перенёс новые версии хранилища в локальный git, но и подтянул чужие
изменения перед этим, и отправил свои — одной командой, без ручных
вызовов `git pull`/`git push`.

**Why this priority**: Единственная причина фичи — устранение ручного
шага, который раньше требовал отдельного вызова `gitrunner`.

**Independent Test**: Два клона одного bare-репозитория, оба
"synced" от лица `gitsync`, с флагом `--remote origin` — после `sync` в
первом клоне видит коммит из второго клона (после pull), и наоборот
(после push).

**Acceptance Scenarios**:

1. **Given** рабочий каталог с настроенным remote `origin`, remote НЕ
   содержит новых коммитов относительно локального, **When** `sync
   --remote origin` синхронизирует 2 новые версии хранилища, **Then**
   pull перед циклом — no-op (уже актуально), после цикла — 2 новых
   коммита отправлены в `origin` (видны через `git log` в bare-репозитории).
2. **Given** remote содержит коммит, которого нет локально (fast-forward
   возможен), **When** выполняется `sync --remote origin`, **Then** перед
   циклом синхронизации локальная ветка подтягивает этот коммит (`git
   log` показывает его ДО коммитов новых версий хранилища).
3. **Given** локальная ветка и `origin` РАСХОДЯТСЯ (не fast-forward —
   есть коммиты по обе стороны), **When** выполняется `sync --remote
   origin`, **Then** `sync` завершается ошибкой ДО начала цикла по
   версиям хранilища — ни одна версия не синхронизируется, `VERSION` не
   меняется, чёткое сообщение об ошибке (не "тихий" сбой).
4. **Given** `sync --remote origin` синхронизировала 0 новых версий
   (`VERSION` уже равен максимуму хранилища), **When** цикл завершается,
   **Then** push НЕ вызывается вообще (нечего отправлять — избегаем
   лишнего сетевого вызова).
5. **Given** флаг `--remote` НЕ передан, **When** выполняется `sync`,
   **Then** поведение идентично `006`/`007` — только локальные коммиты,
   никаких сетевых вызовов git вообще.

### Edge Cases

- Push сам по себе падает (например, remote отклонил из-за политики
  ветки) — коммиты уже СДЕЛАНЫ ЛОКАЛЬНО и `VERSION` уже обновлён
  (синхронизация с хранилищем реально произошла) — `sync` в целом
  завершается ОШИБКОЙ (ненулевой код возврата CLI — см. FR-004), но БЕЗ
  отката уже сделанных локальных коммитов/VERSION (частичный успех:
  локально синхронизировано, с удалённым — нет).
- `--remote` указывает на несуществующий remote (не настроен в
  `.git/config`) — pull падает СРАЗУ (до цикла версий) с понятной
  ошибкой, тот же путь, что Acceptance Scenario 3.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `gitsync sync` MUST принимать необязательный флаг
  `--remote <имя>` (короткая форма `-r`). Без флага — поведение НЕ
  меняется относительно `007` (никаких сетевых git-вызовов).
- **FR-002**: При указанном `--remote` `sync` MUST выполнить `git pull
  --ff-only <remote> <текущая ветка>` ДО начала цикла синхронизации
  версий. Ошибка (в т.ч. расхождение/не-fast-forward) MUST остановить
  `sync` СРАЗУ — ни одна версия хранилища не обрабатывается, `VERSION`
  не изменяется (Acceptance Scenario 3).
- **FR-003**: При указанном `--remote`, ПОСЛЕ цикла синхронизации, ЕСЛИ
  было создано хотя бы 1 новый коммит, `sync` MUST выполнить `git push
  <remote> <текущая ветка>`.
- **FR-004**: Если push (FR-003) завершается ошибкой — `sync` в целом
  MUST сообщить об ошибке (ненулевой код возврата CLI) — но уже
  сделанные локальные коммиты и обновлённый `VERSION` НЕ откатываются
  (см. Edge Cases).
- **FR-005**: Имя текущей ветки для pull/push MUST определяться
  автоматически (`gitrunner.получить_текущую_ветку`), не хардкодиться
  ("main") и не запрашиваться отдельным флагом.

### Key Entities

- **Контекст_Синхронизации** (существующий, `006`) — получает новое поле
  `remote: Опция(Строка)` (`Нет()` — прежнее поведение без флага).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Пользователю больше не нужно вручную вызывать `git pull`/
  `git push` после `gitsync sync --remote <имя>` для типового сценария
  (нет расхождений) — одна команда покрывает весь цикл.
- **SC-002**: Расхождение с удалённым репозиторием обнаруживается ДО
  того, как `sync` начинает трогать версии хранилища — не смешивается с
  частично применённой синхронизацией.

## Assumptions

- Remote должен быть УЖЕ настроен пользователем (`git remote add`) —
  `sync` его не создаёт и не проверяет URL, только использует по имени.
- Аутентификация к remote (SSH-ключ/токен) — забота пользователя/git,
  вне scope этой фичи (как и везде, `gitsync` не хранит и не запрашивает
  учётные данные git отдельно от системного git).
- `gitrunner` не меняется — `--ff-only` через существующий escape hatch
  `Репозиторий.выполнить_команду`, не новый метод пакета (Surgical
  Changes — `--ff-only` не настолько частый частный случай, чтобы
  заслуживать отдельного именованного метода в общем git-пакете).
- Реального удалённого репозитория (сеть, аутентификация) недоступно в
  этой песочнице для проверки — тестируется через ДВА ЛОКАЛЬНЫХ
  git-репозитория (реальный git, тот же приём, что везде в проекте) с
  общим bare-репозиторием как "remote" — полностью реальный git,
  ничего не подделывается (в отличие от 1С-специфичных частей).

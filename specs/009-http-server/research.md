# Research: HTTP-сервер как языковая возможность panos

## §1: `external/odin-http/server.odin` — реальные ограничения моста

Прочитан целиком (`server.odin`, `handlers.odin`, `request.odin`,
`response.odin`, `body.odin`, `headers.odin`) — три находки, определяющие
весь дизайн моста:

1. **`http.serve`/`http.listen_and_serve` БЛОКИРУЮТ вызывающий поток
   НАВСЕГДА** (до shutdown) — `serve()` сам исполняет `nbio.tick()` в
   бесконечном цикле на потоке-вызывающем (плюс порождает
   `thread_count - 1` дополнительных потоков тем же способом). Нельзя
   отправлять как задачу в `vm.async_pool` (пул на 4 воркера,
   рассчитанных на КОРОТКИЕ задачи, возвращающие слот) — навсегда
   заблокированный воркер быстро исчерпал бы пул. **Decision**:
   отдельный, выделенный `thread.create_and_start` (НЕ пул) на каждый
   `сеть.http_сервер_слушать(...)`, живёт всё время жизни слушателя.

2. **`http.respond(res)` привязан к потоку** (`assert_has_td` — сверяет
   `@(thread_local) td` — "The thread you are calling from is not a
   server/handler thread") — значит `respond()` MUST вызываться С ТОГО
   ЖЕ potока odin-http, что вызвал Handler, и НЕЛЬЗЯ отложить его на
   "когда-нибудь потом с другого потока" (рассматривалась и отклонена
   идея "Handler возвращается сразу, respond() зовётся асинхронно позже
   откуда угодно" — невозможно из-за этой привязки). **Decision**:
   Handler ДОЛЖЕН синхронно заблокироваться (на канале) внутри своего
   потока odin-http, дождаться ответа от panos-кода, вызвать `respond()`
   и только тогда вернуться.

3. **Тело запроса читается АСИНХРОННО ВНУТРИ odin-http's own event loop**
   (`http.body(req, max_length, user_data, cb: Body_Callback)` —
   callback-стиль, не блокирующий синхронный вызов) — значит мост
   НЕ может блокироваться на канале сразу в самом Handler_Proc (тело
   ещё не готово, а его чтение зависит от прогресса ТОГО ЖЕ potока,
   который мост собрался бы заблокировать). **Decision**: Handler_Proc
   только РЕГИСТРИРУЕТ `http.body(...)`-callback и сразу возвращается;
   вся мостовая логика (канал + блокировка + `respond()`) — ВНУТРИ
   этого callback'а, который odin-http вызывает уже ПОСЛЕ того, как
   тело полностью получено (тем же potоком, через его собственный
   `nbio.tick()`).

## §2: Итоговый поток одного запроса

```
[odin-http поток] accept -> заголовки -> http.body(req, -1, ctx, on_body)
[odin-http поток, внутри on_body, тело уже получено]
    Bridge_Request{метод, путь, заголовки, тело, response_chan (свежий, cap=1)}
    chan.send(listener.incoming, req)   -- может заблокироваться, если
                                           потребитель (panos) не успевает —
                                           см. §4 backpressure
    chan.recv(req.response_chan)        -- БЛОКИРУЕТСЯ здесь, ждёт panos
    body_set(res, ответ.тело); response_status(res, ответ.статус)
    respond(res)
    -- возврат из on_body -> Handler_Proc -> tick() продолжает жить

[panos-сторона, независимо, в цикле пользовательского кода]
    Слушатель.принять_запрос()          -- Await_Async, submit в
                                           vm.async_pool: worker делает
                                           chan.recv(listener.incoming)
                                           (обычный блокирующий recv —
                                           ровно то, для чего существует
                                           пул, см. research §3)
    -- worker доставил Bridge_Request как ПЛОСКИЕ Odin-данные (никаких
       Value/GC на воркер-потоке — то же правило, что везде в проекте)
    -- deliver_async_result (главный поток) строит НАСТОЯЩИЙ panos
       Запрос-Value из этих данных, включая rawptr на response_chan
    Запрос.ответить(статус, тип, тело)   -- СИНХРОННЫЙ builtin (как
                                           .закрыть() у Socket) —
                                           chan.send(response_chan, ...)
                                           с cap=1, получатель уже ждёт
```

## §3: Почему `.принять_запрос()` — async builtin через существующий пул,
а не отдельный механизм

**Decision**: `.принять_запрос()` — обычный `Call_Builtin_Async`/
`Await_Async` пара (`is_async_builtin_name`, `compiler.odin`), задача
воркера — буквально один `chan.recv(listener.incoming)` (плюс
преобразование результата в плоские поля).

**Rationale**: Это ТОЧНО тот случай, для которого уже существует пул —
потенциально долгое ожидание внешнего события без блокировки
`run_scheduler`. Не нужен новый механизм — нужно только:
(а) новый вариант `Async_Result` для "запрос принят" (`vm_async.odin`,
платформонезависимая часть, чистые Odin-строки/срезы + один `rawptr` на
response_chan — тот же паттерн, что уже держит `net.TCP_Socket`
по-разному на native/wasm) и (б) обработка в `deliver_async_result`,
строящая panos `Запрос`.

## §4: Backpressure и параллелизм — реальные пределы (важно для Assumptions/SC)

- `chan.send(listener.incoming, req)` (внутри odin-http потока, §2) MUST
  быть на канале с КОНЕЧНОЙ ёмкостью (не безграничной очередью — иначе
  память растёт неограниченно, если panos-код не успевает потреблять).
  **Decision**: `chan.Chan(rawptr)` ёмкостью, например, 64 — если полон,
  `chan.send` блокирует ЭТОТ КОНКРЕТНЫЙ odin-http-поток до появления
  места (естественный backpressure, тот же принцип, что уже
  задокументирован для буферизованного `vm.async_completions`, см.
  compiler-and-vm.md).
- **Реальный потолок ОДНОВРЕМЕННО обрабатываемых (`панос-код думает`)
  запросов = `Server_Opts.thread_count`** (по умолчанию — число ядер
  минус один): пока odin-http-поток блокируется в `chan.recv(response_
  chan)` внутри `on_body`, он НЕ обслуживает СВОИ другие соединения —
  это тот же поток, что крутит `nbio.tick()` для НИХ. Значит
  одновременных "живых" запросов, которые panos реально обрабатывает
  прямо сейчас, не больше, чем потоков odin-http — сверх этого новые
  подключения просто копятся в OS accept-backlog, невидимом panos-коду.
  Это ограничение уже нижележащей библиотеки, не что-то, что вводит эта
  фича — упомянуть в Assumptions, не пытаться обойти (не Simplicity
  First — переписывать odin-http ради снятия этого предела).
- Сколько panos-процессов вызывают `.принять_запрос()` параллельно —
  НЕ обязано совпадать с `thread_count`: если процессов больше, лишние
  просто дольше ждут своей очереди на `chan.recv(listener.incoming)`
  (несколько получателей на одном канале — штатное поведение
  `core:sync/chan`, ровно один получатель забирает каждый элемент).

## §5: Типы на стороне panos (data-model.md разворачивает подробнее)

**Decision**: `Слушатель` и `Запрос` — НАСТОЯЩИЕ opaque-типы (как
`Соединение`/`Файл`), не плоские кортежи — оба несут живой ресурс
(поток/канал), который должен быть закрыт/использован ровно один раз, в
отличие от `ос.выполнить`/`синтаксис.*` (мгновенные, без пережившего
вызов состояния, поэтому там — плоские данные).

## §6: Закрытие/остановка слушателя

**Decision**: `Слушатель.закрыть()` — синхронный builtin, вызывает
`http.server_shutdown(&srv)` (штатная функция библиотеки — graceful,
ждёт активные соединения) — НЕ ждёт завершения потока изнутри самого
вызова (не блокирует panos) — поток слушателя сам заканчивается и
освобождает ресурсы асинхронно после `server_shutdown`.

## §7: Разрыв соединения до `ответить()` (Edge Case из spec.md)

**Decision**: `.ответить(...)` использует НЕБЛОКИРУЮЩУЮ попытку отправки
в `response_chan` (`chan.try_send`, не `chan.send`) — если получателя уже
нет (например, `on_body`'s стек уже размотался из-за ошибки/разрыва
соединения на стороне odin-http, что теоретически МОЖЕТ произойти,
хотя основной путь §2 всегда синхронно ждёт) — `.ответить()` просто
возвращает `Результат.Неудача` с понятным сообщением, не паникует и не
блокируется навсегда.

## §8: native/wasm файловый пласт

**Decision**: новая пара `core/vm_http_server_native.odin` (`#+build
!js`) + `core/vm_http_server_wasm.odin` (`#+build js`, `fmt.panicf`
при вызове — тот же паттерн, что `vm_http_wasm.odin`) — не расширяем
существующие `vm_http_*`/`vm_io_*` файлы (Surgical Changes: серверная
часть — отдельная возможность, не вариант уже существующих). Новый
`Async_Result`-вариант — в платформонезависимом `vm_async.odin`, поле с
`rawptr` (response_chan) там же, где уже есть платформенно-специфичные
варианты для `Tcp_Connect_Result_Data` (см. существующий прецедент).

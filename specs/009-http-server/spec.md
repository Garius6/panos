# Feature Specification: HTTP-сервер как языковая возможность panos

**Feature Branch**: `009-http-server`
**Created**: 2026-07-23
**Status**: Draft
**Input**: User description: "Native HTTP server capability for panos, bridging vendored odin-http server to the single-threaded VM via async channels."

## Контекст

panos уже умеет быть HTTP-КЛИЕНТОМ (`сеть.http_запрос`) и работать с уже
открытыми TCP-соединениями (`сеть.подключиться` + `Socket_Value.получить*`/
`.отправить`) — оба через неблокирующий actor-model I/O (worker pool +
`Await_Async`, см. `docs/src/architecture/compiler-and-vm.md`). Не хватает
роли СЕРВЕРА: принимать входящие HTTP-запросы и отвечать на них.

В проекте уже вендорена (`external/odin-http/`) полноценная реализация
HTTP-сервера — свой event loop (`core:nbio`) и пул потоков, используется
ТОЛЬКО клиентской частью пакета сейчас. Серверная часть НЕ интегрирована
с panos-VM. Прямая интеграция невозможна "в лоб": воркер-потоки
odin-http вызывают Handler на СВОИХ потоках, а GC panos не имеет
блокировок и требует монопольного однопоточного доступа (та же
дисциплина, что уже действует для `Socket_Value`/`File_Value` — см.
"Неблокирующий I/O" в документации архитектуры) — значит нужен МОСТ, не
прямая обёртка.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Принять запрос и ответить (Priority: P1)

Пользователь пишет panos-скрипт, который слушает TCP-порт и на каждый
входящий HTTP-запрос отвечает текстом/JSON — без коллбэков, тем же
"цикл + приостановка на I/O" стилем, что уже применяется к TCP-клиенту.

**Why this priority**: Единственная причина фичи — без этого panos не
может быть сервером вообще, только клиентом.

**Independent Test**: Запустить panos-скрипт с `сеть.http_сервер_
слушать(порт)` + циклом `принять_запрос()`/`ответить(...)`, сделать
РЕАЛЬНЫЙ HTTP-запрос (curl/http-клиент) на этот порт с другого процесса,
получить ожидаемый ответ (код/тело/заголовки).

**Acceptance Scenarios**:

1. **Given** panos-скрипт слушает порт и в цикле принимает запросы,
   **When** внешний клиент делает `GET /` на этот порт, **Then**
   panos-код видит метод/путь/заголовки этого запроса и посылает ответ
   (код, тело, `Content-Type`), внешний клиент получает ИМЕННО его.
2. **Given** запрос содержит тело (например, `POST` с JSON), **When**
   panos-код читает тело запроса, **Then** оно доступно как строка,
   доступная сразу (не требует отдельного потокового чтения — тело
   уже полностью получено к моменту, когда `принять_запрос()` вернул
   запрос, см. Assumptions).
3. **Given** несколько внешних клиентов делают запросы ОДНОВРЕМЕННО,
   **When** panos-скрипт запускает НЕСКОЛЬКО процессов (акторов),
   каждый в своём цикле вызывающих `принять_запрос()` у ОДНОГО и того
   же слушателя, **Then** запросы обрабатываются параллельно (не
   блокируют друг друга) — конкурентность достигается штатной
   actor-моделью panos, без нового примитива.
4. **Given** сервер запущен, **When** panos-процесс, который его
   слушает, завершается (или вызывает явное закрытие), **Then** порт
   освобождается, новые подключения получают отказ в соединении (не
   зависание).

### Edge Cases

- Клиент обрывает соединение ДО того, как panos-код успел ответить —
  `ответить(...)` не должен паниковать (просто становится no-op/
  игнорируемой ошибкой — некому доставлять ответ).
- Некорректный HTTP на входе (не парсится как запрос) — соединение
  закрывается на уровне сервера, panos-код не видит "запрос-мусор".
- Порт уже занят другим процессом на машине — `http_сервер_слушать`
  возвращает `Результат.Неудача`, не паникует.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: panos MUST дать возможность начать слушать TCP-порт для
  HTTP-запросов и получить обратно объект-слушатель (`Результат`,
  ошибка — если порт занят/недоступен).
- **FR-002**: Слушатель MUST давать операцию "принять следующий запрос",
  приостанавливающую вызывающий процесс (тот же стиль, что подключение
  по TCP уже приостанавливает процесс) до прихода реального запроса —
  не блокирует ВЕСЬ scheduler, только вызывающий процесс.
- **FR-003**: Принятый запрос MUST давать доступ к методу, пути,
  заголовкам и ТЕЛУ (уже полностью прочитанному к моменту получения
  запроса panos-кодом — см. Assumptions про потоковые тела).
- **FR-004**: Принятый запрос MUST давать операцию "ответить" (код
  статуса, `Content-Type`, тело) — доставляет ответ РЕАЛЬНОМУ клиенту,
  подключение закрывается/переиспользуется по правилам HTTP
  (keep-alive — забота нижележащего сервера, не panos-кода).
- **FR-005**: Несколько panos-процессов MUST мочь одновременно вызывать
  "принять следующий запрос" у ОДНОГО и того же слушателя — сервер
  распределяет реальные запросы между ожидающими процессами, давая
  конкурентную обработку через штатную actor-модель.
- **FR-006**: Отсутствие ответа на запрос (клиент отвалился, panos-код
  упал) MUST не ронять сервер целиком и не блокировать другие запросы.
- **FR-007**: Эта возможность MUST быть недоступна в wasm/браузерной
  сборке (серверные сокеты бессмысленны в браузерной песочнице) — тот
  же паттерн, что уже у `сеть.http_запрос`/`сеть.подключиться`: ошибка
  времени выполнения при попытке вызвать, не ошибка сборки.

### Key Entities

- **Слушатель** — открытый серверный сокет на порту, источник входящих
  запросов для одного или нескольких panos-процессов.
- **Запрос** — метод, путь, заголовки, тело (уже полностью считанное),
  плюс скрытая ссылка на то, КУДА доставить ответ.
- **Ответ** — код статуса, `Content-Type`, тело — то, что panos-код
  передаёт операции "ответить".

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Реальный HTTP-клиент (curl или аналог) успешно получает
  корректный ответ от panos-скрипта, слушающего порт — без сторонних
  инструментов между ними.
- **SC-002**: Один запрос, зависший в обработке (panos-код "думает"
  долго), не задерживает ответы на ДРУГИЕ параллельные запросы,
  обрабатываемые другими процессами того же слушателя.
- **SC-003**: Сервер не падает и не течёт памятью при разрыве
  соединения клиентом до получения ответа (Edge Case).

## Assumptions

- **Тело запроса читается ЦЕЛИКОМ до передачи panos-коду** (не
  потоково) — упрощение первой версии: потоковое тело (по образцу
  `File_Value.прочитать*`) добавило бы координацию на каждый чанк через
  мост поток-VM ради случая (стриминг больших тел), не нужного для
  типового panos-скрипта веб-хука/API-эндпоинта. Ограничение размера
  тела — по умолчанию нижележащей библиотеки (`external/odin-http`),
  не переопределяется отдельно в этой фиче.
- **Не WebSocket/HTTP2/TLS-terminated-на-panos** — только обычный
  HTTP/1.1 запрос-ответ, TLS (если нужен) — забота внешнего
  reverse-proxy, не panos-кода (как и у большинства встраиваемых
  языковых HTTP-серверов).
- **Один процесс на "принять_запрос()" за раз, но много процессов на
  одного слушателя** — конкурентность через штатную actor-модель
  panos (FR-005), не через встроенный пул воркеров ЯЗЫКА (тот пул
  принадлежит нижележащему `external/odin-http`, невидим panos-коду).
- **`external/odin-http/server.odin` используется как есть** — не
  форкается и не патчится, все изменения — в новом native-builtin
  слое поверх него (Surgical Changes) — если чего-то не хватает в
  библиотеке, это отдельное решение на этапе `/speckit.plan`, не
  предрешённое здесь.
- Реальный сквозной прогон (panos-сервер + внешний HTTP-клиент,
  Acceptance Scenario 1) тестируем ПОЛНОСТЬЮ по-настоящему — в отличие
  от 1С-специфичных частей других фич этого проекта, здесь нет
  недоступного внешнего мира: `curl`/сокеты реальны в этой песочнице.
